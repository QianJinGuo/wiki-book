# Agent 生产级 Harness 工程实践

生产级 Agent Harness 的工程要求：高可用、可观测、成本控制、安全沙箱、渐进式披露。区别于 Demo 级 Harness，生产级需处理长时运行、状态恢复、并发隔离、失败回滚等真实工程挑战。

## 深度分析

### 生产级 Harness 的核心工程约束

生产级 Agent Harness 与 Demo 级 Harness 的根本区别在于对"边界条件"的处理。Demo 级 Harness 在理想环境下运行（网络通畅、单用户、短任务、无并发），而生产级必须面对网络分区、长时运行状态漂移、多租户资源争抢、部分失败等真实分布式系统的所有挑战。这些约束驱动了 Harness 架构的工程决策：状态持久化、心跳检测、优雅降级、断路器模式等稳定工程模式的系统引入。

### 可观测性：生产运行的"仪表盘"

在生产环境中，Agent 的行为不再是"黑盒一次调用"——它可能运行数分钟甚至数小时，涉及多次工具调用、多轮推理、跨系统交互。可观测性的三个支柱——日志（结构化事件记录）、指标（Token 消耗、步骤耗时、错误率）、链路追踪（跨 Agent 的调用链）——构成生产 Harness 的监控基础。没有可观测性，生产部署等同于盲飞。

### 安全沙箱与隔离机制

生产 Harness 必须为 Agent 的操作划定安全边界：文件系统访问范围、网络访问控制、API 调用频率限制、敏感信息脱敏。这些不能依赖 Agent 的"自觉"——必须在 Harness 层面强制实施。沙箱机制从简单的工作目录隔离到完整的容器级隔离，形成了多层次防御。同时还涉及审计日志的需求：所有 Agent 行为可以被事后溯源。

### 成本控制的多层策略

生产级运行意味着 Token 消耗不再是实验费用而是持续运营成本。成本控制需要从多个层面实施：模型路由（简单任务用轻量模型）、上下文窗管理（避免无意义历史累积）、预算旋钮（为不同场景设定 Token 上限）、子 Agent 隔离（长任务分解后独立计费）。成本控制与运行质量之间需要显式权衡，最优解取决于具体业务场景。

## 实践启示

1. **从可观测性开始建设**：不要让"先跑起来再说"成为借口——在部署第一天就打通日志、指标、链路追踪三条管线，事后补可观测性的成本远高于一开始就设计进去。
2. **渐进式安全策略**：安全沙箱的粒度应从环境变量白名单开始，逐步扩展到网络策略和文件系统隔离，避免过早的过度约束阻碍迭代速度。
3. **成本预算前置化**：在 Harness 层面设定硬性的 Token 预算上限（如单次任务不超过 N 万 Token），超出时触发降级而非直接失败，让成本异常在影响业务之前被捕获。
4. **设计状态恢复机制**：长时运行的 Agent 应支持 checkpoint——定期保存中间状态，失败时可从最近 checkpoint 恢复而非从头重跑。这对 10 分钟以上的任务尤为重要。
