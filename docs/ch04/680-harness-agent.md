# Harness 范式：Agent 的工程基座

Harness 范式将 Agent 从单次对话提升为可持续运行的工程系统：上下文管理、工具编排、循环控制、安全边界、人机协作构成五大支柱。是 Agent 工程化的核心方法论。

## 深度分析

### 五大支柱的层次关系

Harness 范式的五大支柱存在结构性层次：上下文管理是底层基础——决定 Agent 能感知的信息范围；工具编排是执行层——将感知转化为行动；循环控制是流程层——确保目标的收敛性；安全边界是约束层——划定行动的不可逾越范围；人机协作是交互层——决定人类介入的时机。下层为上层提供能力基础，上层反向提出约束要求。

### 从单次对话到持续运行系统的范式转换

核心在于"状态管理"的位置变化。在单次对话模式中，状态存在于 LLM 的上下文窗口中——对话结束状态即消失。在持续运行系统模式中，状态被外化到 Harness 层面——文件系统、数据库、队列——LLM 只是状态转换的"引擎"而非"仓库"。这带来了状态持久化、错误恢复、并发控制等连锁需求。

### 五大支柱的工程实现权衡

每个支柱都有多种实现路径。上下文管理：完整保留 vs 滑动窗口 vs 检索增强。工具编排：顺序执行 vs 并行扇出 vs 条件分支。循环控制：固定轮次 vs 自适应终止 vs 人类中断。安全边界：软约束（提示词）vs 硬约束（系统层强制）。人机协作：审批门禁 vs 异常上报 vs 全程监督。

## 实践启示

1. **从上下文管理开始**：五大支柱中上下文管理是底层基础，优先解决"Agent 能看到什么"再谈"Agent 能做什么"。
2. **状态外化优先于模型升级**：将状态从上下文窗口外化到持久化存储，比升级更强模型对系统可靠性的提升更大。
3. **每层都做显式权衡**：在正文管理、工具编排等每层记录显式的设计决策和权衡理由，避免隐形假设导致后期重构。
