# Thariq（Claude Code工程师）的Fable 5使用心法：地图≠领土，用未知消除法突破模型瓶颈

> Claude Code 团队核心工程师 Thariq 提出 Fable 5 级模型的工作质量受限于"澄清未知的能力"而非模型能力本身，并给出四类未知的应对框架。

## 核心洞察：地图≠领土

Thariq 的核心比喻——prompt、skill、上下文是"地图"（给 Claude 的说明书），而代码库、真实世界约束是"领土"。地图和领土之间的差距即为"未知"（unknown unknowns）。当模型能力足够强（如 Fable 5），瓶颈从"模型能不能做到"变成"你能不能说清楚你到底要什么"。

## 四类未知

1. **已知的已知**：团队清楚知道的需求，可以写进 prompt
2. **已知的未知**：知道可能存在但未明确的问题，需要提前规划
3. **未知的已知**：隐含在代码库或业务逻辑中的约束，需要在实施中发现
4. **未知的未知**：最坑的一类——埋在实施深处，挖到一半才冒头，甚至可能要求换个解法

## 指令困境

- 指令过细：Claude 死守指令，即使当下该转向也一条道走到黑
- 指令过粗：Claude 按"行业最佳实践"自行猜测，未必对路子
- 核心解法：未知消除法（unknown elimination）——在编码过程中持续识别和消除未知，而非一次性规划
