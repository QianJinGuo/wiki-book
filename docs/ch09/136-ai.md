# AI 原生开发工作流

AI 原生开发工作流区别于传统敏捷：以自然语言 Spec 为源、以 Agent Loop 为执行引擎、以 Human-in-the-loop 为质量门禁。Vibe Coding → Loop Engineering → Fleet Loop 的演进路径。

## 深度分析

### 从自然语言 Spec 到可执行代码的映射链

AI 原生开发工作流的核心是一条"自然语言 Spec → 结构化任务 → 代码生成 → 自动验证"的映射链。每个环节都涉及信息的翻译和精化：模糊的产品需求在 Spec 阶段被约束为明确的行为描述，Spec 在任务分解阶段被拆为可验证的原子单元，任务在代码生成阶段被实现，最后通过自动测试验证实现是否满足 Spec。

### Agent Loop 作为执行引擎

与传统的 CI/CD 流水线不同——传统流水线是被动的，代码提交触发构建测试，流水线本身不产生代码。Agent Loop 是主动的、持续运行的执行引擎：从目标出发→读取当前状态→决定下一步操作→执行→验证→反馈。Agent Loop 从"检验者"变为"执行者"，人类从"操作员"变为"监督者"。

### Vibe Coding → Loop Engineering → Fleet Loop 的演进逻辑

这一演进路径的内在逻辑是"逐步外化智能控制"：Vibe Coding 阶段，智能在 LLM 的单个对话中；Loop Engineering 阶段，智能被外化为循环的结构——提示词模板、验证步骤、回滚机制；Fleet Loop 阶段，智能进一步外化为 Agent 集群的协作协议。每一步都是将"隐式知识"编码为显式的系统结构。

## 实践启示

1. **从映射链的最弱环节开始**：分析你的工作流中哪个环节的信息丢失最严重（Spec 不明确、验证不充分），优先改进这个环节。
2. **让 Loop 设计可观察**：每个 Loop 循环都应输出状态日志和决策理由，否则调试循环行为就像调试黑盒。
3. **不要跳过 Vibe Coding 阶段**：直接从 Vibe Coding 跳到 Fleet Loop 是常见的过度工程——只有在同一类任务重复出现 5 次以上时才值得投资 Loop Engineering。
