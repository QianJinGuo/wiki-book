大家好，欢迎来到 code秘密花园 ，我是花园老师（ConardLi）。

用 Agent 操作浏览器已经不是新鲜事了。

查资料、看榜单、抓网页信息、整理竞品、跑一段重复流程...

这些事现在都可以交给 Agent 去做。

它能打开网页，能点按钮，能输入关键词，能把页面里的结果整理出来。

很多时候确实省事。

但用得越多，问题也越明显。

过去我自己也尝试过很多方案了

让 Agent 直接读网页、写浏览器脚本，或者临时用一些自动化工具把流程跑通。

每种方案都有能用的时候，但都并不完美

有些页面的数据不是一打开就有，而是等浏览器运行一会儿才加载出来；

有些任务需要保持登录状态；

有些验证码要人工接一下；

还有些流程今天好不容易跑通了，明天换个关键词又要重新摸一遍。

这也是我后来对 “Agent 自动操作浏览器” 这件事越来越谨慎的原因。

会点网页是一回事，能不能稳定复现、能不能留下可检查的数据、能不能把流程沉淀下来，是另一回事。

最近，我在刷 Product Hunt 的时候，发现这天的榜单第一名居然是一个专门给 Agent 操作浏览器的工具 - BrowserAct 。

这一下我就有点好奇了。

浏览器自动化不算新东西，Playwright、Selenium、RPA 这些方案都已经存在很久了。

为什么一个 “浏览器自动化工具” 会冲到 Product Hunt 第一？

它到底只是又一个自动化工具，还是解决了 Agent 使用浏览器时更深一层的问题？

于是我详细研究了一下 BrowserAct，也顺手做了一个真实案例。

在看实操之前，我们需要先捋清楚 Agent 操作浏览器的几个局限性。

## Agent 操作浏览器的几个局限

理想化 Demo 里的网页通常很简单：打开页面，输入关键词，复制前几条结果。

真实网站不是这样。

第一类问题来自页面本身。

很多网页不是打开就把所有内容摆出来，而是先加载一个框架，再由浏览器继
