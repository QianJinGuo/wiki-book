# 用 Spec 驱动 AI 编程半年，我总结了四个大多数人踩的坑

## 第一幕：有损管道 — SDD 的价值是让损耗可见

### 1. 所有开发范式都是同一条有损管道

人有意图，机器产生行为。中间是一条**有损管道**。结构性的必然：人脑意图含隐性成分（偏好/经验/语境/审美/风险直觉），代码是离散的，从连续到离散信息必然损失。[^1]

| 范式 | 控制点 | 损耗怎么处理 |
|------|--------|-------------|
| 古法编程 | 人脑 | 工程师隐式兜底 |
| Vibe Coding | 无 | 没人兜 — 损耗裸奔 |
| SDD | spec | 显式定位 + 人审 |

> SDD 不消除损耗——世上没有无损管道。它做的事是让损耗**可见、可审计、可归因**。[^1]
> Prompt 是一次性指令，Spec 是可审计的责任链。

### 2. Spec 的本质定义

**误区**：把 spec 理解成"写给 AI 看的更详细的文档"。

**正确定义**：`spec = 对"可接受实现空间"的最小、可验证、可演进的显式编码`。[^1]

| 关键词 | 反面坑 |
|--------|--------|
| 最小充分 | 把 PRD 全文复制进 spec.md |
| 可验证 | "界面要美观""性能要好" |
| 可演进 | 写完就不碰的 spec 快照 |
| 显式编码 | "大家都知道这里要做幂等" |

### 3. Spec 两层结构：L1 锁方向，L2 锁路径

- L1 需求意图 — Why + What
- L2 需求设计 — How

没对齐 L1 就先拍 L2——这是 AI Coding 最普遍的翻车模式。你以为 AI 写错了代码，其实是**你没说清什么叫"对"**。[^1]

## 第二幕：三大 SDD 工具比较

### Spec Kit（GitHub）：多阶段管道
四层管道（Intent → spec → plan → tasks → code），每层~15% 损耗，累积对齐度约 **52%**。用过程可审性交换信息保真度。[^1]

### OpenSpec（社区）：活基线
proposal → delta → tasks 一次生成、聚合评审。delta 回写保证基线永远活着。用卡点轻交换基线不腐烂。

### Kiro（AWS）：形式化需求
EARS 句式（WHEN…SHALL…）消除歧义。需 IDE 强耦合 + 形式化门槛。用工具自由交换需求无歧义。

> **选型是问：控制链断在哪？** 不是选"更好的工具"。[^1]

## 第三幕：三个认知陷阱

### 陷阱一：不能自动验证的 spec 注定会烂
判断标准：spec 能不能被自动化验证？能，值得维护；不能，就是换名字的技术文档，一样会烂。

### 陷阱二：Spec 是契约，不是蓝图
spec 若比代码短，必然省略某些实现决策。**唯一完整的程序规格，就是程序本身**。试图让 spec "完整到能推出所有代码"会导致维护成本爆炸。[^1]

### 陷阱三：不是所有需求都上 SDD
| 风险/复杂度 | 策略 |
|-------------|------|
| 低（脚本/原型） | vibe coding |
| 中（功能迭代/CRUD） | Plan Mode + 轻量 spec |
| 高（支付/权限/合规） | SDD 全流程 |

> SDD 不是银弹，是手术刀。不是所有伤口都值得开刀。[^1]

---

## 五句话带走

1. 核心矛盾变了：不是写不出代码，是没人能证明代码是对的。**意图持有者和代码编写者分离**，是所有 SDD 问题的第一因。[^1]
2. Prompt 用完即弃，Spec 可追溯、可审计、可回归。
3. 代码越来越便宜，**定义"什么叫对"越来越贵**。
4. Spec 是**契约**（系统该做什么），不是**蓝图**（系统具体怎么做）。
5. 判断一条 spec 有没有价值：**能不能在 CI 里自动判定 pass/fail？** 不能的，就是换名字的技术文档。
