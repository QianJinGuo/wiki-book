本文介绍了"高价率运营 AI 工作台"中基于"约定驱动 + AI 编排"架构的自动化评测优化实践。针对 LLM Agent 场景下传统测试失效、反馈周期长等痛点，文章详细阐述了将评测体系作为核心工程产物的设计方案：通过标准化目录结构固化规范，利用 Coding Agent 实现自然语言驱动的全流程自动化；构建包含真实业务数据驱动的评测集生成、14 个通用维度与专项指标结合的 Rubric 体系、以及采用二元评分机制的 LLM Judge 双引擎架构（auto-evaluation 与 pinchbench-eval）。同时，文章深入剖析了金标设计中的信息隔离、粒度匹配等关键陷阱，并探讨了从半自动向全自动演进过程中，在评测入口、指标自动迭代及记忆机制等方面面临的技术挑战与应对策略，最终实现了将 Skill 可用性从主观判断转化为可量化、可复跑的工程闭环。

一、引言与背景

1.1 业务背景与建设目标

高价率运营是电商行业小二的日运营场景，涉及高价播报、数据分发、同款校验、价格跟进、归因、日报等长链路工作。这是典型的"可标准化、可重复"工作，适合用 AI Agent 替代。

立项建设"高价率运营 AI 工作台"，围绕目标管理、数据分发、价格跟进、日报产出 4 类可标准化事项，把人工操作沉淀为一组业务 Skill。真正的难点不在于"做出 Skill"，而在于"持续保证 Skill 可用"——业务规则在迭代、上游数据接口在变、新场景层出不穷，任何一次 SKILL.md 的修改都可能让某个上线已久的 Case 突然崩掉。

1.2 为什么评测是工作台的核心问题

传统软件工程的"上线前测一下"在 LLM Agent 场景下基本失效：没有确定的输入输出、失败模式不可枚举、改一处影响一片。

工作台从一开始就把评测体系作为一等公民来建设——把"Skill 是否可用"从主观判断，变成可量化、可复跑、可对比的工程产物。

1.3 工作台的整体设计

工作台采用"约定驱动 + AI 编排"的架构：通过标准化目录结构（skills/ / skill-data/ / pinchbench-suite/）固化规范，通过 Claude Code 作为编排器实现自然语言驱动的全流程自动化。

三个层次：
- 业务层：16 个业务 Skill
- 规范层：14 个通用评测维度 + 各 Skill 专项 rubric + 216 条评测用例
- 执行层：auto-evaluation（评测大脑）+ pinchbench-eval（执行引擎）双引擎

二、核心架构：约定驱动 + AI 编排

2.1 约定驱动：通过目录结构固化开发规范

每个 Skill 遵循标准化目录结构。SKILL.md 用 YAML frontmatter 强声明触发词等关键信息。

skill-data/ 与 pinchbench-suite/ 分离：sync_to_suite.py 桥让"评测源管理"和"评测执行"彻底解耦。

2.2 AI 编排：让目录约定真正"跑起来"

把 Coding Agent 当作工作台的编排器，让自然语言成为唯一的入口，目录约定由编排器强制兑现。覆盖四个环节：需求管理、开发与评测、复盘与改进、发布管理。

三、自动化评测架构设计

3.1 核心创新：把评测从"一次性快照"变成"持续联动"

评测集、评测指标、Skill 定义三者不是独立产物，而是同一个飞轮的三个齿轮。SKILL.md 当作单一事实源。

3.2 评测集生成：基于真实业务数据

六大评测用例类型：典型场景、边界场景、追问场景、格式要求、触发词匹配、真实 Case。

3.3 评测指标体系

14 个通用评测维度，按 4 类分类：路由准确性、流程遵循、参数正确性、工具合规、名称编码区分、输出合规等。

severe_violation 机制：若任一严重违规维度得 0 分，整体评分取所有严重维度的最小值。

3.4 评测执行与评分

LLM Judge 评分，二元评分（0.0 或 1.0，禁止中间分数）。评估 prompt 采用 6 段式结构。

3.5 双引擎架构

auto-evaluation（评测大脑）：6 Phase 闭环；pinchbench-eval（执行引擎）：项目无关，可整包迁移。

3.6 金标（reference_data）设计

4 个关键陷阱：
1. 信息泄漏（类型层面隔离 runner/judge）
2. 粒度太细（粗化为语义大阶段）
3. 答案污染（人工过审 + 限制参考来源）
4. 多轮覆盖不足（增加 followup_expectation）

3.7 评测真实踩过的坑

LLM 不会按字面自律、同步时机错误、AI 自创非标准字段、触发词重叠导致路由误触。

四、业务落地效果

- 16 个业务 Skill，14 个通用评测维度
- 15/16 个 Skill 定制了专项 rubric
- 216 条评测用例（截至 2026.05.07），单次评测从 5 小时人肉到 40 分钟无人值守
- 评测飞轮反馈周期从周缩短到小时

五、自动优化飞轮的技术挑战

5.1 从全人工到半自动的演进

当前是"自动+人工兜底"：Case 入库前人工过审、评分后抽查 10%、核心 Skill 上线前跑 3-5 个真实 empId。

5.2 指标自动迭代的困难

本质是测量系统的自指。三种失败：放宽标准、收紧过严、过拟合。

5.3 飞轮难点（一）：线上问题怎么被"看见"

当前评测集是"我们想到的"，不是"用户问出来的"。理想：线上日志→异常检测→Case 抽取→进入飞轮。

5.4 飞轮难点（二）：修改建议如何"改对地方"

LLM 建议约 80% 不合格，根因是判断不了问题真正在哪一层。

5.5 飞轮难点（三）：没有"修改痕迹"的飞轮等于没有飞轮

四类 log 设计（annotation/rubric_change/skill_change/ai_suggestion），JSONL 格式，按月分文件，通过 ID 互引构成因果图。

六、未来演进方向

三条主线：入口（线上问题回流）> 记忆（log 体系）> 出口（精准归因）> 指标自动迭代（排最后）。

参考 autoresearch（Karpathy）的最小骨架，提出 SkillResearch 方案。
