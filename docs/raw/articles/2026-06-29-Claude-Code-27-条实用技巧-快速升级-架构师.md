这种清单适合收藏，也容易照着试。

放到工程里看，它背后其实是同一个问题：会几个命令是一回事，让 Claude Code 在真实项目里稳定推进，是另一回事。

我对“快速升级”的理解更朴素：少背技巧，多把 Claude Code 放进稳定、可验证、可接手的 工程工作流 。

Claude Code 进入真实项目以后，怎样少猜一点、少跑偏一点，最后还能留下能复盘的 证据 。

下面仍然按 27 条讲，只是换一个更实用的顺序：先处理上下文和边界，再把验证放进过程，最后才谈并行、自动化和长期运行。

这样看，27 条技巧就从工具清单，变成了一条从个人使用走向团队工作流的升级路径。

## 快速版

前 9 条，重点是让 Claude Code 少猜。项目 上下文 、任务边界、计划和自检要先清楚。

中间 9 条，重点是让过程可控。Skills、模型选择、规则维护、纠偏、截图和通知，都是为了减少反复解释和无效返工。

后 9 条，才是更高级的工作流。DevTools、resume、worktrees、API、Loop、数据库和高思考预算，都需要配套 隔离 、 验证 和 硬边界 。

这里先保守一点： 先跑稳一个小闭环，比一口气打开所有高级能力更有价值。

## 先换一个理解

很多人第一次用 Claude Code，会自然把它当成一个更强的自动补全。

输入需求，等它改代码，再看结果能不能跑。

这个用法当然能省时间。但一进真实项目，它会遇到很多人类工程师也会遇到的问题：不知道项目约定，误判完成标准，读了旧上下文，或者沿着一个看起来合理的假设继续做下去。

更贴近工程现场的看法是，Claude Code 已经是一个会进入仓库的执行体。它会读文件、跑命令、调用工具、修改代码，也会总结自己的工作。

既然它会执行，就不能只靠一句提示词。

我们需要给它上下文、边界、验证和停止条件。

这也能接上我们前几篇 Claude Code 文章里反复出现的线索：  CLAUDE.md  解决的是  进仓库前先看什么  ，  指令层级解决的是规则放在哪里  ，  Loop 解决的是任务跑起来以后怎么验证、怎么停  。

下面 27 条技巧的价值，主要在它们组合出来的协作方式。

## 第一组：先把上下文放正

### 1. 新项目先跑 /init

/init  的价值，不在于生成一份更长的项目说明，而在于给 Claude Code 一张 入口卡 。

这张入口卡里，优先放这些东西：项目做什么、不做什么；常用测试和构建命令；关键目录边界；哪些证据可以说明任务完成；哪些配置要谨慎。

第一版不用长。60 行左右，通常比 300 行规则更有用。

这和《  CLAUDE.md拆解：Agent进仓库前的上下文入口  》里的看法一致：  CLAUDE.md  可以当入口上下文，不必写成项目百科。

- API handlers validate input and shape responses only.  
- Services own business orchestration and transactions.  
- Separate code changes, local checks, deployment status, and production evidence in summaries.

这类规则有对象、有动作，也更容易被执行。

### 2. 用 /statusline 或 /usage 看状态

长会话里，体感经常不可靠。

上下文用了多少，成本是否异常，当前会话是不是已经变得很重，都适合用  /statusline  、  /usage  这类命令看一眼。

个人使用时，它提醒你该压缩还是换任务。团队使用时，它还能反过来暴露问题：某类任务是不是太贵，是否该拆成脚本、Skill 或更小的验证闭环。

### 3. 语音输入适合捕捉想法

语音输入适合补充想法、描述页面问题、快速反馈 UI。

但涉及权限、数据、不能改的范围和验收条件，最好还是写成短清单。口述很快，边界最好落成文字。

一个稳一点的用法是：先说问题，再让 Claude Code 复述约束和计划，确认后再开始改。

### 4. 用 /context 看上下文负担

Claude Code 跑久以后，输出变差不一定是模型能力问题。很多时候，是上下文里混进了旧文件、过期讨论、无关日志和太长的历史对话。

/context  适合用来做一次清理前检查。

把上下文当成运行资源看，很多问题会更容易理解。资源脏了，执行自然容易漂。

### 5. 同一任务用 /compact ，换题用 /clear

/compact  适合长任务继续推进，  /clear  适合换到无关任务。

压缩前，可以先让 Claude Code 输出四件事：当前目标、已验证证据、未完成项、不能碰的边界。

这一步很小，但能避免压缩后把关键限制丢掉。

### 6. 复杂任务先计划

计划模式不是写一份漂亮大纲。

它的价值在于提前暴露假设。

比如不必只说：

帮我修登录页。

可以换成：

用户使用包含特殊字符的密码时无法登录。  
这是报错。  
约束是：不能改数据库结构，不能改变现有登录 API。  
开始前先列出可能原因、改动范围和验证步骤。

多花几分钟，通常能少掉后面几轮返工。

### 7. 描述问题，不只下命令

“做一个登录页”是命令。

“为什么这个用户在这个场景登录失败”是问题。

Claude Code 拿到问题、约束和已有证据后，才更容易做判断。只给命令，它会自己补默认设定，而默认设定往往就是返工来源。

### 8. 开始前，让它先问清楚

复杂任务里，可以加一句：

开始之前，请先列出需要澄清的问题。  
能从代码库或文档中查到的事实，请先自己查。

这句话的重点，不是多问几个问题。它是在做分流：哪些事实可以自己查，哪些产品、权限、业务边界需要人来定。

这样协作会轻很多。

### 9. 把自检写进 Todo

很多任务失败，并不是写不出来。更常见的情况是，代码刚能跑，Claude Code 就已经开始总结“完成了”。

把 验证 写进 Todo，效果很直接。

Todo:  
- 添加支付表单。  
- 添加输入校验。  
- 验证：空输入、特殊字符、超过额度金额都有明确错误状态。  
- 验证：移动端错误提示可见，不遮挡提交按钮。

写完以后，它至少知道该拿什么检查自己。

### 10. Subagents 先看隔离，再看并行

Subagents 容易被理解成“多叫几个助手”。

更值得看的是隔离。

一个 subagent 只读探索，一个负责实现，一个做审查。它们不共享同一段推理过程，就能减少“自己证明自己正确”的问题。

并行是收益，隔离才是底层价值。

### 11. 把重复提示沉淀成 Skills

同一段提示写第二遍时，就可以考虑 Skill。

code review、发布检查、排障、事实核验、去 AI 味，这些都有固定步骤，不适合每次临时写一段 prompt。

Skill 里写清楚输入、步骤、只读边界、停止条件和输出格式，比把一大段提示塞进  CLAUDE.md  更稳。

入口文件管第一眼上下文，Skill 管可复用过程。

### 12. 模型要匹配任务

不是每个任务都需要最高规格模型。

重命名、格式化、信息抽取、样板代码，可以用更轻量的模型。架构设计、安全关键、多系统重构、复杂根因分析，再给更强模型和更多思考预算。

这不是单纯省钱。模型规格长期和任务不匹配，团队会看不清真实成本结构。

### 13. CLAUDE.md 要持续维护

/init  生成的  CLAUDE.md  不是一次性文件。

项目变了，它也要变。

但维护要克制。每犯一次错就加一句  IMPORTANT  ，很快会把入口文件写成噪声。

更稳的做法，是先判断这条经验该放哪。每次启动都要知道的，放入口文件；某类目录才需要的，放局部规则；某个流程才需要的，做成 Skill；漏掉会出事故的，交给 Hook、权限、CI 或脚本。

自然语言提醒不是安全边界。

《  我终于搞明白了：Claude Code为什么会忽略指令了  》也提到过这个问题：很多指令失效，不一定是模型完全没读；更常见的是规则放错了层，或者它被塞进了不该承担这类职责的位置。

### 14. 用引用代替堆文档

不必把所有文档都塞进  CLAUDE.md  。

入口文件可以只留路径：

## Architecture  
See: docs/architecture.md  

## Release  
Use: .claude/skills/release-check/SKILL.md

详细内容按需读取。对大仓库来说，这比所有任务加载所有规则更稳。

### 15. 跑偏时先停下来

Claude Code 有时会很勤奋地沿着错误方向继续工作。

发现方向不对，先停，不急着让它再改一版。

我常用这种方式重新对齐：

先停一下。现在不要继续修改。  
重新对齐：  
- 真实问题是……  
- 不能改的是……  
- 已验证的证据是……  
- 下一步只处理……

这比在错误分支上继续补丁更省时间。

### 16. 第一版输出要 review

第一版输出能用，不代表已经稳。

复杂任务后，可以追问三件事：最薄弱的假设是什么，有没有更小的改法，哪些地方还没真实验证。

这不是挑毛病，是把 review 提前放进流程。

### 17. 通知让人回到节点检查

长任务一直盯着终端，人会被拖进 Agent 的节奏里。

通知 Hook、命令完成提醒、任务结束摘要，可以让人从实时监工退回到节点检查。

通知也不宜太多。值得提醒的是阻塞、验证失败、状态变化和需要人判断的节点。

### 18. UI 任务要看真实页面

Claude Code 改 UI，不能只看代码。

截图、浏览器控制台、Network、移动端视口，才更接近真实用户。

前端任务至少要确认：页面是否跑起来，按钮是否可见，移动端是否溢出，控制台有没有错误，关键请求是否符合预期，空态和错误态有没有覆盖。

UI 的完成证据，不能只停在组件文件里。

这一点和《  Anthropic工程师：我们日常如何使用Claude Code  》里的经验很接近：需求先问清楚，规格先落下来，最后还是要把运行时证据看一遍。

## 第三组：再谈自动化和长期运行

### 19. 用 DevTools 看运行现场

接入 DevTools 后，Claude Code 可以看 DOM、控制台、网络请求和页面状态。

这会改变调试方式。以前是人把错误复制给它，现在它可以直接观察现场。

但 DevTools 看到的是当前页面事实，不等于业务闭环完成。登录、权限、数据写入、生产环境行为，仍然要有对应证据。

### 20. resume 之后先重述边界

resume  适合长任务恢复。

恢复后先别急着继续改。可以让 Claude Code 输出当前目标、已完成证据、未完成项和当前禁止事项。

长任务最怕上下文看似还在，边界已经变薄。

### 21. worktrees 解决文件碰撞

并行会话最好配合 worktrees。

多个会话在同一个 checkout 里改文件，冲突会很乱。worktree 至少能隔开文件系统层面的碰撞。

但它解决不了判断冲突。最后仍然要有人合并、review、跑验证。

### 22. 高频调用再考虑 API 直连

有些高频、稳定、性能敏感的内部流程，直接 API 可能比 MCP 更轻。

但这不是通用结论。

低频探索、权限复杂、需要审计的场景，MCP 的统一接口反而更稳。选择哪条路，要看调用频率、权限、日志、重试和成本。

工具接入本身就是架构取舍。

### 23. Loop 先做只读小闭环

Loop 适合重复、可验证、低风险的 只读小闭环 。

比如每天读一次 CI 失败并归类，每周检查文档失效链接，PR 前抽取事实性断言并核来源，发布后做只读 smoke 并生成证据摘要。

这些任务读多、写少、证据明确，失败后也容易停。

生产配置、账单、敏感数据，不适合一开始就交给 Loop 自动处理。

我们在《  Loop工程实战：从任务循环到可维护闭环  》里已经把这点展开过：第一版 Loop 不宜追求全自动，先把状态、动作、验证和 停止条件 摆出来。

### 24. 远程控制适合触发，不适合放权

手机远程启动会话很方便。

这类入口更适合低风险触发，比如启动构建、拉取只读报告、检查任务状态。

涉及写入、部署、删除、账单和生产数据，最好回到更完整的操作环境里确认。

入口越方便，权限边界越要清楚。

### 25. 自然语言查数据库，先限定只读

让 Claude Code 用自然语言生成 SQL 很方便，也更容易误操作。

第一版最好明确只读：

只允许 SELECT。  
先解释你要查哪些表、为什么查。  
不要执行 UPDATE、DELETE、INSERT。

真要改数据，再走备份、where 条件、影响行数预估、事务、回滚方案和执行后复核。

### 26. 难题再用高思考预算

跨系统故障、安全边界、多模块重构、性能瓶颈、架构取舍，适合慢一点。

先让 Claude Code 做事实梳理、假设列表、反例检查、最小验证路径和回滚方案，再考虑动手。

高思考预算适合疑难问题，不必变成日常默认档。

### 27. 高级模式留给高风险任务

高级编码模式、深度 review、workflow 视图，都会随着产品版本变化。

工具名会变，判断标准相对稳定。

多文件、多系统、安全或数据风险高、需要独立 review、需要长时间运行、结果需要可复盘，这类任务才值得上更高规格能力。

## 更实用的练法

这 27 条看完以后，不必今天全用上。

个人第一周，先练四个动作：  /init  生成入口上下文；复杂任务先计划；自检写进 Todo；长任务压缩前先输出目标、证据和边界。

团队第一个月，先沉淀三件东西：一份短而准的  CLAUDE.md  ，一个常用 Skill，一个只读验证闭环。

判断 Claude Code 有没有真的变得更有用，可以看这些信号：返工率有没有下降，任务中断后能不能接手，验证证据是否更完整，重复流程是否少解释，高风险动作是否被系统挡住。

这些指标不如“效率提升多少倍”好传播，但更接近真实收益。

## 写在最后

这几个月，从 Prompt、Context、Harness、Loop 到 Environment，我们一直在看同一件事：

模型能力越来越强，模型外面的工程结构也越来越重要。

Claude Code 的 27 条技巧，也可以放在这条线里理解。

它们表面上是一串命令，放到工程里看，其实是在把 Agent 放进真实现场：入口上下文减少错误假设，计划提前暴露边界，验证负责校正结果，隔离降低污染，Skills、Hooks 和 Loop 则把反复出现的流程沉淀下来。

提示词当然还重要，只是位置变了。

以前更常问的是：

我该怎么提示它？

现在更值得多问几句：

它进入项目后，先看到什么？  
它每一步怎么验证？  
它跑偏时谁能拦住？  
它做过什么，后面的人能不能接手？

这几个问题不花哨，但更接近真实工程里的 Claude Code。

27 条技巧不用逐条背。真正值得带走的，是把 Claude Code 从“会生成代码”，慢慢带到“能在工程流程里稳定交付”。

先从一个小闭环开始。能验证，能暂停，能复盘，能接手，这比一张全自动路线图更值得信。
