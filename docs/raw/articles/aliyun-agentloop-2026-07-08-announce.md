# 阿里云刚发布的 AgentLoop 是什么？

每个时代的软件工程进化，都始于对不确定性的收敛。

传统软件时代，工程师用日志、指标和链路追踪，观测分布式系统的运行状态；云原生时代，容器和微服务提升了系统的复杂度，要求观测指标更加丰富和细颗粒度。到了 Agent 时代，新的不确定性出现了：一个智能体不再只是执行一段确定代码，而是在模型推理&自主规划、工具调用、知识检索、记忆读写中运行，观测需求从原子化的指标演进到 Agent 的完整推理路径（Trajectory）。

## 01 Agent 优化，为何要观测先行？

我们无法优化看不见的东西。

传统应用的线上问题是相对确定的，例如一段慢查询、一个内存泄漏、一次超时的 RPC 调用。这种"发现问题、定位根因、修复、验证"的工作流，建立在系统行为可预测的前提之上。

Agent 应用打破了这个前提。同一段 Prompt、同一个模型，在不同时间输入时的行为路径都不完全一样。

因此，Agent 观测的往往不是某个指标是否报错，而是关注输出效果，包括答案的准确度和丰富性、推理和行动规划的合理性，甚至 token 消耗的合理性等。

## 02 Agent 应用的观测有什么不同？

引入了新名词：Trajectory（Agent 的推理路径）。

Agent 从接收任务到完成任务，中间经历了哪些思考步骤、调用了哪些工具、检索了哪些知识、做了哪些决策，形成了 Trajectory。

**观测的维度变了：**
- 推理步骤的合理性（Agent 的规划是否正确）
- 工具调用的准确性（是否调对了工具、参数是否正确）
- 检索结果的相关性（检索到的文档是否真正有用）
- 模型输出的质量（回答是否准确、是否产生了幻觉）
- token 消耗与成本效率等

**观测的数据粒度也变了：** 传统链路追踪的粒度是应用、服务、接口级别。Agent 观测的粒度则是一次对话 sessionId、一次端到端的调用 traceId、一次请求 requestId。

## 03 LLM 优化和 Agent 优化，是一套方法论么？

随着 Agent = Model + Harness 深入人心，Agent 场景下模型能力不再是决定系统表现的唯一变量。

Agent 的行为具有三个特点：任务通常是开放式的（没有唯一正确答案）、执行路径是动态的、表现高度依赖上下文。

## 04 Agent 优化的挑战

**数据从哪来？** 需要采集 Agent 的推理路径，数据量大、结构复杂、语义丰富。

**怎么评？** Agent-as-a-Judge — 用 Agent 来评估 Agent。

**怎么改？** Agent 系统的参数空间非常大：prompt、工具定义、知识库、编排逻辑、模型选择，且存在复杂耦合关系。

## 05 AgentLoop 如何去优化智能体？

AgentLoop 通过 **MVP 五环** 实现智能体的进化：

**第一环：Agent 观测与审计。** 支持 Dify、LangChain/LangGraph、AgentScope 等框架，以及 OpenClaw、Hermes、Qoder、Claude Code、Codex、Cursor 等客户端。无侵入数据采集，不需要修改 Agent 代码。

**第二环：Agent 轨迹分析。** 以 ATIF（Agent Trajectory Interchange Format）为标准格式，Pipeline 数据处理管线自动批量处理轨迹数据。

**第三环：Agent 效果评估。** Agent-as-a-Judge 机制，20+ 开箱即用的评估器 Agent。

**第四环：Agent 实验回测。** CI/CD 基线测试（每次 Agent 资产变更时自动回归）+ 场景化测试。

**第五环：Agent 持续优化。** Prompt、Skill、模型配置多维调优 + 经验库自动提取。

## 06 进化不是万能药

进化本质上和 Harness 一样，是模型能力尚未完全覆盖时的工程补位。当模型足够强，这一层外部脚手架终将被内化。

在离业务近的非通用场景（每家企业的审批流程、行业合规要求、领域经验），Agent 进化的工程体系有着最不可替代的价值。
