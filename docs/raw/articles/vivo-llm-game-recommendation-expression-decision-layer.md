# 当 AI 进入推荐系统：从"推什么"到"怎么选"

> 推荐系统擅长回答"推什么"，用户却卡在"怎么选"。本文探索：不改排序，补上排序之后的"表达与决策"——把游戏理解透，让同类多款可比较、可解释、可追溯。做法是让大模型放开探索、用工程约束收住，使生成稳定进生产。

## 核心问题

用户打开页面同时看到几款游戏（如元气骑士→Roguelike、元气骑士前传→RPG、我的勇者→弹幕射击），排序已把最可能被点的放最前面，但用户没点——他在犹豫：这几个看着差不多，到底有啥区别？

## 已有标签体系为何不够？

旧的内容标签擅长回答"这是什么"，却回答不了玩家**为什么玩、为什么付费**。以捕鱼游戏为例，旧标签能说"捕鱼类·PVE·模拟·轻度松弛"，但对"该不该推给这个用户"几乎没有帮助。

新体系必须补的增量是两层：
- **游戏动机**：玩家为什么玩、为什么付费（爽感刺激、解压逃避、随机刺激）
- **付费机制**：它靠什么结构承载付费（概率付费、抽卡/转盘）

## 探索→收敛的流程

不是"先定体系、再让 AI 执行"，而是：

```
模型放开探索 → 撞出边界 → 验证哪些维度真有用 → 收敛成 schema → 规模化生产
```

自由生成是大模型的能力上限，稳定输出得靠约束。分工：探索交给模型，收敛和立规矩由人来做。

## 多游对比：从"一次能跑"到"稳定生产"

最初做得轻（直接把三款丢给模型讲差异），结果够用。但顺着"做正经系统"往复杂走（拆多步、加 pipeline、分段 prompt），结果反而退化了。

**教训**：复杂化本身会吃掉价值。加 pipeline 不是天然的进步——只有当它产出的东西不比直出版本少、还能稳定复现，才算进步。

最终结构：**品类配置** = core_dimensions（比什么）+ expression_schema（怎么说）+ highlight_priority（谁上卡片）。目前已配出数十个品类草案，新品类原则上加一行配置、不动代码。

## 可复用的工程经验

1. **探索和生产，要用模型的两副面孔** — 先发散探索，再收敛执行，不可混用
2. **模型的原始输出不算接口** — normalize 层（闭集 schema + 缺字段回落 + 脏值过滤）是生产前提
3. **Prompt 的结构，决定理解的深度** — 喂给模型的结构就是它输出的上界
4. **把"比什么""怎么说""谁上卡片"解耦** — 分别约束而非一把出
5. **让能力不绑上下文，才能复用** — 无状态、不绑定调用场景的 LLM 能力插件

## 产品视角的变化

AI 没让产品设计变简单，它把"系统结构"本身变成了产品的一部分。core_dimensions、expression_schema、rejectList 这些东西，既是工程实现，又直接决定用户看到什么——它们是长在系统语义层里的产品设计。

需求路径变化：业务目标→功能设计→工程实现 → 业务问题→能力探索→结构设计→原型验证→系统协同

## 成果形态

一层挂在排序之上、只读不回写的能力层：事实来自数据，表达来自模型。排序负责"推什么"，AI 补"怎么说清楚"。其中部分前端场景（推荐解释、场景化引导）正在接入，待上线。
