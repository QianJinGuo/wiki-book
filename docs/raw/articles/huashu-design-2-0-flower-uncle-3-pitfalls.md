## 坑二 · 内容网站没有真图 = 空壳

**问题**: v1 最容易翻车 — 让它做鹦鹉科普网站, 排得漂漂亮亮, 但**没有一张鹦鹉的图, 全是灰色占位块**.

> **一句话总结**: 鹦鹉网站没有鹦鹉图, 等于失败.

**解法**: 2.0 加了 **图片前置** 步骤. 开工之前先问: "图片是不是这个内容的必需品?"

- **科普/旅游**: 图就是命根子 → **先把真图取齐, 再开始设计**
- 案例: 鹦鹉网站 → 先跑去 Wikimedia Commons, 把各种鹦鹉真实照片 + 公共领域古典博物版画抓一整批, 再动手设计

> 这步看着简单, 但它是"能看的 demo" 和 "能用的成品" 之间的**分水岭**.

## 坑三 · AI 一本正经地瞎编产品事实

**问题**: 让它给大疆 Pocket 4 做发布动画, 它张口就来: "Pocket 4 还没发布, 我们做个概念 demo 吧" — 然后自作主张做了一版概念剪影动画. **结果 Pocket 4 早就发布了**, 几天前官方 Launch Film / 产品渲染图全都有. 它基于一个错误事实做了一版废稿.

**根因**: AI 凭训练语料记忆对不确定的事做断言. "我记得好像还没发布""应该是 v3 版本" — 听着自信, 全是猜的. 

> **不是设计才有的毛病, 是所有 AI 应用的通病**. 但放在做设计物料上代价更直接: 事实错了, 画得再美都是废稿.

**解法**: 2.0 把 **「事实验证先于假设」** 定成 **第 0 原则**, 凌驾于所有流程之上.

- 涉及具体产品 / 版本号 / 规格 / 发布状态 → **第一步必须先搜一遍核实**
- 把事实写进文件, **绝不靠记忆**
- **搜不到就问你**, 而不是自己编一个

> 案例: 新加坡招商 PPT → 樟宜机场吞吐量 / QS 排名 / 税率, 每个都先核实出处再写.

> **搜一遍花 10 秒, 返工花两小时, 这账太好算了**.

## PPT 替代品: HTML deck 优于 pptx

**态度转变**: 做完之后, 作者反而劝你别再导成 pptx.

**HTML deck 优势**:
- 打开是一面**能鸟瞰全局的概览墙**
- 点任意一页 → 全屏滑进去一页页演示
- 跟真正的 Keynote 一个手感
- 观感和流畅度已经比传统 pptx 好一截

**仍然支持 pptx 导出** (如果公司要求):
- 导出后拖进 PowerPoint, 双击标题光标就进去了
- **文字是真文字, 图表是真形状** (不是把每页截成图片塞进去那种假的)

**PPT 案例**:
- **新加坡招商引资 deck**: 12 页, 数据全核实过
- **网球入门**: 60 页, 优雅杂志感, 封面一颗网球落在草地上

## 适用场景 vs 不适合

**适合** (高质量视觉产出, 不想打开设计软件抠):
- 网站原型 / App 原型 / PPT deck
- 动画 / 信息图 / 带解说的长视频
- 可点击的 App 原型 (交付前自己跑点击测试, 保证按钮点了不白屏)

**不适合**:
- 生产级 Web App
- 需要后端的动态系统
- SEO 网站

## 核心定位: 决策者 vs 动手者

> **设计软件**: 画布 / 图层 / 工具栏, 预设你是那个**动手的人**.
> **Skill**: 预设你是**做决策做判断的人**.

> 2.0 做的所有事, 本质都是在让**「拍板」**这个动作更值得做更容易做. 给你的三版都是真东西, 不是三个一模一样的安全极简. 剩下的, 你只负责**选哪一个**.

## 通用方法论 (跨域可复用)

3 个 2.0 引入的原则, **对任何 agent skill 设计都适用**:

1. **撞/借/请 — 三套并行反收敛**: 当 AI 容易收敛到"最安全选项"时, 强制 3 套独立逻辑并行跑 (随机 / 真实世界锚定 / 大师哲学). 解决"千篇一律"问题
2. **图片前置 (或任何核心资产前置)**: 内容型任务开工前, 先验证"核心资产是否齐备". 没有就先去抓 (Wikimedia Commons / 公共数据集 / 真实 API). 分水岭: 能看 demo vs 能用成品
3. **事实验证先于假设 (第 0 原则)**: 任何涉及具体事实 (产品 / 版本 / 数据) 的任务, **第 1 步必须搜索核实**, 搜不到就问用户. 凌驾于所有流程之上
