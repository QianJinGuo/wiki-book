# 怎么短平快地把RAG做好：厦门国际银行RAG初赛方案分享

前段时间笔者参加了厦门国际银行的第五届数创金融杯大模型应用挑战赛，其中初赛是金融监管制度智能问答，属于经典RAG问答，具体比赛任务如下：

> 基于大模型的文档问答，根据输入的问题（如"个人理财产品的销售需满足哪些监管要求？"），基于给定的金融文档库，生成准确、合规的答案。题型包含不定项选择题和问答题。

> 初赛阶段选手可以使用预训练、微调、RAG等方式完成任务，整体工程需在限定硬件资源下能进行推理（CPU8核、32G内存、24G显存），结果评估采用A/B榜，最终以B榜成绩作为初赛排名依据。

笔者差不多在比赛两周内去冲刺的这个比赛，差不多用时10天左右可以冲到top10内，首先说明哈哈我不是第一，也非常好奇前排方案，所以笔者在这里抛砖引玉分享下初赛思路，欢迎大家交流。

> 这里说的做好不是做的完美，做完美确实很难，对召回和问答要求更严格，这里其实想的是怎么短平快地把RAG任务做的差不多，或者说效果能够让用户接受。

## 整体技术架构

方案采用了经典的RAG架构，主要包括7个步骤：

1. **文档加载解析** — 把docx文件解析成结构化数据
2. **文本切块** — 把长文档切成适合检索的小块
3. **问题检索排序** — 根据问题找到最相关的文档片段
4. **指令数据集构造** — 构造训练数据
5. **模型微调** — 用LoRA/QLoRA微调Qwen3模型
6. **模型推理** — 生成答案
7. **结果投票** — 多模型融合提升效果

代码几乎都在TrustRAG项目可以看到：**https://github.com/gomate-community/TrustRAG**

## 第一步：文档加载解析

原始金融监管文档都是docx格式，需要先提取内容，同时统计基本信息为后续处理做准备。

具体做法：
- 递归扫描比赛数据集目录下的所有docx文件
- 用TrustRAG框架的DocxParser解析文档内容，提取段落文本
- 用Qwen3-8B的tokenizer计算每个文档的token数量
- 标记哪些文档包含表格（表格信息在金融监管中很重要）
- 按文件夹结构保存为JSON格式

> 使用什么模型问答最好用它的分词器来统计长度，这样可以比较精准地设置窗口大小，提高大模型输入token长度预算的利用效率。

最终，所有解析出的内容连同文件名、Token长度、是否含表格等元数据被统一保存为JSON格式的文件，并维持了原有的目录结构。整体来说大部分文档token在1600左右。

输出：
- `output/documents/`：按目录结构保存的JSON文档文件
- `output/docx_token_lengths.xlsx`：详细统计数据
- 各种可视化图表：token长度分布图、箱线图等

## 第二步：文本切块与数据组织

原始文档通常篇幅很长，直接用于检索不仅效率低下，而且会严重稀释问题的相关性。整篇word文档输入时针对问题答案信息密度稀疏，大模型捕获答案能力太弱。

采用TrustRAG中的`SentenceChunker`确保切分时不会割裂完整的句子。分别实验了256和512两种chunk_size，也分析了768和1024的召回率。

**设计优化**：将切块内容（`corpus_chunks`）与文件元信息（`file_metadata`）分离存储，减少数据冗余。

输出：
- `corpus_chunks_{chunk_size}_hf_optimized.parquet`：核心切块数据
- `file_metadata_{chunk_size}.parquet`：文件元信息
- `chunk_to_file_mapping_{chunk_size}.parquet`：映射关系表
- `corpus_chunks_full_view_{chunk_size}.parquet`：完整视图

**召回率实验结论**：较小的chunk大小（256）表现最佳，信息密度更高；所有配置回答率超过94%。

**困难案例**：
- 答案可能在一句话中且难找，或相似内容太多导致排序靠后（如"双线报告"）
- 答案分布在多个chunk中
- 部分问题无需检索，大模型自身知识即可回答

## 第三步：问题检索与重排序

采用**混合检索（Hybrid Retrieval）**策略：

| 组件 | 模型/方法 | 权重 |
|------|-----------|------|
| Keyword (BM25) | 关键词匹配 | 0.3 |
| Dense (Vector) | bge-m3 或 bge-large-zh-v1.5 | 0.7 |

流程：混合检索召回15个候选 → BGE-reranker-large二次精排 → 选择题将选项内容补充到查询中。

**配置组合**：使用两个embedding模型（bge-m3, bge-large-zh-v1.5）和两个chunk_size（256, 512），交叉组合构造不同输入样本为后续模型融合做铺垫。

## 第四步：指令数据集构造

针对选择题和问答题分别设计专门的prompt：

**选择题Prompt**：强调"精准分析"和"逐一验证"，要求与文档原文精确对比，输出格式严格如"A,C,D"。

**问答题Prompt**：五步推理（问题解构→信息检索→内容筛选→答案组织→答案优化），要求使用规范的金融监管术语，强调逻辑清晰。

**指令构造**：加载检索结果 → 取前6个检索结果（实验证明6个结果在信息量和准确性间最佳平衡） → 格式化内容 → 应用prompt模板 → 组合训练样本。

## 第五步：模型微调

微调补齐了与前排的差距，通用大模型在金融监管领域表现不够好，需用领域数据微调。

使用Qwen3-8B+QLoRA/LoRA和Qwen3-14B+QLoRA，实测8B和14B在排行榜分数差不多，甚至8B比14B效果更好。

## 第六步：模型推理

- temperature：0.0（确保输出稳定）
- max_new_tokens：512
- batch_size：1（内存限制）

## 第七步：结果投票融合

**选择题**：多数投票（Majority Voting）原则，选项超半数模型选中则认定为最终答案。

**问答题**：选择语义相似度最高的作为最终答案。

## 核心经验总结

1. **数据处理要细致** — 256和512两种块大小各有优势，最好都试试
2. **Prompt工程很关键** — 步骤推理提高选择题准确率，格式严格提升问答拟合效果
3. **混合检索效果好** — BM25+Dense比单一方法好，加上reranker进一步提升
4. **多模型融合有效** — 不同配置模型各有优势，通过投票融合显著提升
5. **硬件资源要合理利用** — QLoRA是好选择，能微调大模型又不爆显存

> RAG没有标准答案，更加考验面对不同任务能够灵活变通的能力，也不要从头开始，建议可以快速基于已有脚手架进行改造，发现问题解决问题。
