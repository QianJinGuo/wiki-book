### 实测一：全栈开发挑战，从0到1手搓微信小程序

对于很多开发者来说，日常最头疼的就是从零开始梳理复杂的业务逻辑并完成繁琐的前端组件开发。Hy3 这次重点升级了 Coding Agent 能力，特别针对前端需求理解、组件生成及交互逻辑编写做了深度优化。

为了探底它的极限，我们不搞玩具代码，直接让它以高级架构师的身份，手搓一个包含地图选址、表单验证和动态时间轴的商业级微信小程序「新能源汽车充电桩智能预约与管理」

    请担任高级全栈架构师和产品经理的角色，为我设计和生成一个“新能源汽车充电桩智能预约与管理”微信小程序的项目方案与核心代码。  
    一、项目核心目标  
    开发一个集“充电站地图查找、空闲桩实时查询、一键预约、充电状态动态监控”于一体的综合服务小程序。核心要求为流程标准化、状态实时透明。  
    二、详细需求说明  
    首页与地图选址（核心交互）：  
    集成微信小程序地图组件（Map），基于用户位置展示附近充电站坐标点。  
    点击坐标点可上拉半屏弹窗展示站点详情（站名、距离、快充/慢充空闲数量、当前电价）。  
    预约充电表单：  
    核心表单包含选择车型、输入车牌号（支持一键选择历史车辆）、选择预约时间段、预计充电度数。  
    具备根据当前时段电费（峰谷电价不同）进行预估费用的实时计算逻辑。  
    充电状态监控（核心功能）：  
    以动态时间轴或进度条形式展示充电状态（如预约成功、占位等待中、设备连接成功、充电中、已充满、支付完成）。  
    支持实时展示当前已充度数和预估剩余时间。  
    个人中心：  
    我的爱车档案管理（增删改查）。  
    充电历史账单列表。  
    三、技术栈与框架要求  
    前端要求使用微信小程序原生框架，UI组件库必须使用Vant Weapp，充分利用其丰富的表单和半屏弹窗组件。  
    后端（模拟端）要求使用Node.js (Express) 提供API。数据库结构请用JSON文件模拟。  
    四、交付要求  
    第一步：输出项目整体目录结构，要求符合原生小程序开发规范。  
    第二步：输出核心数据结构的JSON格式定义（重点包含User, Station, ChargeOrder, Vehicle）。  
    第三步：优先实现并交付以下关键页面的前端代码（WXML, WXSS, JS）和对应的后端API接口定义：  
    首页地图选址页：地图组件与Vant Weapp底部弹窗的联合交互逻辑。  
    预约充电表单页：带有峰谷电价预估计算的表单验证逻辑。  
    状态监控页：充电状态时间轴组件的实现与数据渲染。

### 实测二：单文件 3D 游戏开发

既然 Hy3 重点强调了在 复杂推理与数理能力上的显著跃升，我们就不测常规的跑酷或赛车游戏了。这次我给它出了一道比较罕见、考验空间几何与底层算法的题目：用纯代码写一个带真实光线反射物理规则的 3D 塔防游戏。

【直接复制以下Prompt去WorkBuddy测试】：
    
    
    请你扮演资深图形学算法工程师，为我创建一个单文件的 3D “光学射线防御”游戏。要求仅输出一个可直接在浏览器双击运行的 HTML 文件。  
    核心开发要求如下：  
    技术栈要求：必须使用 Three.js 构建 3D 场景，绝对禁止依赖任何外部的图片或 3D 模型文件。  
    核心物理与玩法：场景中央有一个固定的“激光发射器”，持续发射一条红色的射线。玩家可以用鼠标在场景中点击放置“玻璃镜面”，并通过键盘旋转镜面的角度。  
    硬核数学计算：当激光射线命中“玻璃镜面”时，必须严格根据法线向量和物理反射定律，实时计算并渲染出反射光线。光线可以经过多面镜子的连续反射。  
    动态敌人与碰撞检测：场景边缘会随机生成代表敌人的悬浮球体，它们会向中心移动。当激光的射线轨迹（包括所有反射轨迹）穿过敌人时，敌人即被销毁并增加玩家分数。  
    内存优化（硬性要求）：必须通过“对象池（Object Pooling）”技术自动回收并复用被击毁或移出边界的敌人球体，绝对禁止无限实例化新对象。  
    视觉效果：场景采用赛博朋克暗黑风格，激光需要有明亮的自发光材质效果，代码结构必须高度模块化且包含关键数学计算公式的注释。
