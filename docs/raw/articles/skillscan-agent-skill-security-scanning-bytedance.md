随着 AI Agent 技能（Skills）生态的迅速发展，社区开发者贡献的技能数量与日俱增。然而，这些技能来源多样、质量参差不齐，其安全性缺乏有效保障。攻击者可能借机发布恶意技能，对用户设备进行攻击或窃取数据。

## 五类技能安全风险

### 1. 包体文件合规风险
攻击向量：恶意文件植入（可执行二进制后门）、硬编码凭证（API Key/密码）、资源耗尽攻击（超大文件/高压缩比）、目录穿越（../ 路径穿越）
影响：本地文件包含、敏感信息泄露、系统资源耗尽、任意文件写入

### 2. 声明层安全风险
攻击向量：提示词注入（skill.md 描述中嵌入恶意指令）、敏感行为声明、高风险依赖声明
影响：模型被诱导执行恶意操作、系统提示词泄露、供应链攻击

### 3. 代码层安全风险
攻击向量：恶意代码植入（反向 Shell/数据窃取）、危险函数调用（eval/exec/subprocess）、不安全的第三方库、容器逃逸、敏感路径访问
影响：远程代码执行、数据泄露、容器逃逸、权限提升

### 4. 网络与资源安全风险
攻击向量：未加密 HTTP 通信、恶意域名/IP 连接、C2 通信特征（心跳/指令交互/加密隧道）、资源耗尽（无限循环/内存泄漏）
影响：数据泄露、远程控制、拒绝服务、平台资源耗尽

### 5. 开源合规与供应链风险
攻击向量：许可证不合规（GPL 传染性）、已知漏洞组件（CVE）、依赖链投毒、威胁情报匹配
影响：合规风险、已知漏洞利用、供应链攻击

## SkillScan 安全检测体系

### 检测能力分层
- **基础检测层**：默认开启的核心检测，覆盖包体合规、声明安全、代码恶意行为、网络连接风险、供应链威胁
- **增强检测层**：可选配置，包括开源许可证合规检测、SCA 软件成分分析、内网敏感信息检测、数据外发检测
- **运营保障层**：威胁情报持续更新、误报漏报处理、高危技能应急响应

### 安全准入要求（触发即需人工复核）
- 包体文件存在内容安全风险
- skill.md 中存在提示词注入风险
- skill.md 中存在敏感行为声明且未通过安全评估
- Python/JavaScript/Bash 代码存在已知恶意代码模式
- 存在容器内敏感信息收集或容器逃逸行为
- 代码尝试访问文件系统敏感路径（~/.ssh/、/etc/passwd 等）
- 尝试建立反向 Shell 等 C2 特征的网络连接
- 开源组件许可证存在严重合规风险

### 接入模式
支持云上技能市场（FindSkill、CN-ClawHub、Arkclaw）和内场技能共享两种场景

## 技能安全开发最佳实践

### 原则一：最小权限
- 仅申请真正需要的权限
- 避免 exec/subprocess，优先使用安全封装的 API
- 文件访问限制在指定目录
- 网络访问仅允许必要域名

### 原则二：输入验证与输出编码
- 对所有外部输入严格验证
- 输出时进行编码转义
- 避免将用户输入传入危险函数（eval/exec/system）
- 对 skill.md 等声明文件进行安全校验

### 原则三：依赖安全管理
- 使用官方源和可信渠道
- 定期更新依赖版本修复已知漏洞
- 新增依赖进行安全审查
- 尽量减少不必要的依赖
- 使用 lock 文件锁定依赖版本

### 原则四：敏感信息保护
- 禁止硬编码密钥/密码
- 使用环境变量或密钥管理服务
- 日志输出脱敏处理
- 不在公开文档中记录内部系统信息

### 原则五：网络通信安全
- 优先使用 HTTPS
- 避免与不明来源服务器建立连接
- 对外部 API 响应进行安全校验
- 限制网络请求频率
- 传输敏感数据时额外加密

### 原则六：安全测试与验证
- 发布前进行完整安全自测
- 使用 SkillScan 等工具自动化扫描
- 对敏感功能进行人工安全审查
- 测试异常输入和边界情况
- 定期安全复检

### 原则七：包体规范与完整性
- 打包时排除不必要的文件（.git、临时文件、测试代码）
- 避免包含可执行二进制文件
- 不使用符号链接（避免路径穿越）
- 确保声明文件内容真实准确
- 提供清晰版本号和变更记录
