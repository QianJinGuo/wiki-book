系列：Claude Code vs OpenAI Codex 架构拆解 · 第 02 篇

两套系统在"如何让模型看到它需要的信息"这个问题上，给出了方向完全相反的答案。Claude Code 构建了一套精密的五层压缩管道，把有限的上下文窗口用到极致；Codex 用容器文件系统绕开了这个限制，按需读取，理论上没有上限。

Claude Code：上下文窗口是一切的边界

Claude Code 的整个上下文管理系统都在回答同一个问题：当对话历史 + 工具输出 + 项目文件超过模型上下文窗口时，怎么办？

Claude 4.6 系列的上下文窗口是 100 万 token。这个数字看起来很大，但在实际的软件工程任务里——反复读取大文件、运行命令、解析输出——会比预想的更快填满。

CLAUDE.md：上下文注入的入口

在进入压缩管道之前，先看 queryLoop() 的 Step 3（Context assembly）如何把信息注入上下文。

CLAUDE.md 文件是 Claude Code 持久化"项目知识"的核心机制。它有四个层级，按优先级从低到高依次加载：

层级 1：系统级全局（~/.claude/CLAUDE.md）
    所有项目共享，用户写一次到处生效
    典型内容：个人代码风格偏好、常用命令别名
层级 2：用户项目全局（~/.claude/CLAUDE.md 里的 @import）
    通过 @import 指令引入其他文件
层级 3：项目级（./CLAUDE.md，git 追踪）
    提交到仓库，团队共享
    典型内容：架构约定、构建命令、测试流程
层级 4：本地私有（./CLAUDE.local.md，gitignore）
    不提交，个人本地配置
    典型内容：本地路径、临时调试开关

四层都会在 Step 3 被加载，直接拼接进 context。这是硬注入——无论 token 占用多少，CLAUDE.md 内容总是出现在上下文里。

Auto Memory：LLM 驱动的记忆选取

除了 CLAUDE.md，Claude Code 维护一个 memory/ 目录，存储跨会话的工作记忆。关键设计在于它不用向量相似度检索，而是用一个专门的 LLM 调用来做记忆选取：

每次 context assembly 时，系统先把 MEMORY.md（索引）注入上下文——它只是摘要，token 消耗低。然后在需要时用 LLM 扫描 frontmatter 的 description 字段，选出最相关的至多 5 个记忆文件，把它们的完整内容加入上下文。

这个设计牺牲了向量检索的效率，换来了人类可读、可版本控制的记忆系统——记忆文件是普通 Markdown，用户可以直接编辑。

五层压缩管道

CLAUDE.md 和 Auto Memory 是"往上下文里加东西"的机制。压缩管道是"从上下文里减东西"的机制。两者在 Step 4（Pre-model context shapers）里协同工作。

五层压缩按代价从低到高依次触发，每层只在更轻的层不足以解决问题时才启用：

第一层：Budget Reduction（预算削减）

触发条件：任何包含大型工具输出的 turn。

机制：每个工具都有一个 maxOutputTokens 预算。bash 工具默认上限约 50K token。如果工具输出超出预算，StreamingToolExecutor 在写入对话历史之前就截断，并追加一条 [output truncated: N bytes total, showing last M bytes] 说明。

这是最轻量的层，因为它在数据进入上下文之前就介入，不需要修改已有历史。

第二层：Snip（片段截断）

触发条件：queryLoop() Step 4 检测到 context 超过某个阈值（具体阈值依模型窗口而定）。

机制：从最早的非系统消息开始，识别并截断大型工具结果块（tool_result 消息）。只截断工具输出，不截断用户消息和 assistant 文本。截断后插入 [content snipped to save context] 占位符。

Snip 是纯文本操作，不调用模型，延迟极低。

第三层：Microcompact（微压缩）

触发条件：Snip 之后 context 仍然接近上限。

机制：用一个专门的压缩模型调用，把连续的工具调用/结果序列压缩成更短的摘要。例如，一系列读文件操作（read file A → content、read file B → content）可能被压缩成："读取了 A、B 两个配置文件，关键信息：[摘要]"。

Microcompact 需要一次额外的 LLM 调用，有延迟和成本，但能保留语义信息而不是直接截断。

第四层：Context Collapse（上下文折叠）

触发条件：context 使用量持续接近上限，Microcompact 已多次触发。

机制：这是一个关键的设计细节——Context Collapse 不修改 REPL 数组（对话历史的内存表示），而是在读取时执行投影。

具体来说，系统在 collapse store 里存储一条摘要消息（对之前全量历史的语义压缩），然后在 Step 3 的 context assembly 阶段，把"摘要 + 最近 N 轮历史"作为实际发给模型的 context，而不是完整的历史。

这个"读时投影"设计的好处：对话的完整历史仍然保存在磁盘（JSONL 文件），不会丢失。如果用户需要恢复，或者需要审计 Agent 的行为，完整记录还在。Context Collapse 只影响"当前给模型看什么"，不影响"历史是什么"。

第五层：Auto-compact（自动压实）

触发条件：context 使用量超过约 95%（接近模型上下文窗口上限）。

机制：触发一次完整的语义压缩——用一个压缩 prompt 把整个对话历史（包括所有工具调用记录）重写成一份结构化的工作状态摘要，然后用这个摘要替换历史，重置对话。这不是渐进式操作，而是一次性的全量重压。

Auto-compact 之后，从模型视角看，就像开始了一次新对话，但工作状态（已完成的工作、当前进度、关键发现）通过摘要保留。

这是代价最高的层：一次专门的 LLM 调用、明显的延迟、以及不可避免的信息损耗（压缩总是有损的）。

Codex Cloud Agent：文件系统即上下文

Codex 的上下文策略从根本上不同：不是管理上下文里有什么，而是按需从文件系统取。

仓库快照注入

任务启动时，Codex 把整个 GitHub 仓库克隆进容器。这是一次性操作，之后模型可以随时通过 shell 命令访问任何文件：




没有预先的"把所有相关文件塞进上下文"步骤。模型在推理过程中按需读取，上下文只包含它实际用到的文件内容。

理论上限 vs 实际约束

Codex 不受"一次性必须塞进上下文"的约束，理论上可以处理任意大小的代码库。但这不意味着没有上下文限制——codex-1 模型仍然有有限的上下文窗口，只是它通过选择性读取来避免触顶。

实际的约束变成了：模型能否正确判断需要读哪些文件？

对于一个 100 万行的大型代码库，模型不可能在一次任务里读完所有文件。它依赖搜索（grep、find）和模式识别来定位相关代码。这在大多数情况下工作良好，但在代码库组织不规范、或者问题需要跨越很多文件才能理解的场景里，模型可能遗漏关键上下文。

没有 CLAUDE.md 等效物

Codex 没有 CLAUDE.md 机制（至少在早期版本里没有）。项目特定的约定和上下文需要通过以下方式传递：

任务描述本身
：在提交任务时显式说明约束
README 和文档文件
：模型会主动读取项目说明文档
代码本身
：从已有代码模式里推断风格约定

这个设计偏向"从代码里读取约定"而不是"从配置文件里读取约定"，更符合 Codex 的云端异步工作流（用户不在旁边，无法在任务过程中补充上下文）。

两种策略的关键差异

Claude Code 选择把所有相关信息主动注入上下文（精确控制），代价是随着任务增长需要精细的压缩管理，且压缩有损。它的优势是完整性——只要压缩算法足够好，模型"看到的"信息是完整且有组织的。

Codex 选择让模型自主决定读什么（按需读取），代价是可能遗漏关键文件，且每次 shell 读取都是一次工具调用，有延迟。它的优势是理论上的无上限——无论代码库多大，不存在"塞不进去"的问题。

这个差异的深层原因，还是回到第 01 篇提到的执行模型差异：Claude Code 的同步交互需要每轮对话都有完整上下文（用户可能在任何时候问"你刚才为什么这么做"），而 Codex 的异步执行只需要完成任务，中间过程不需要对用户可解释。
