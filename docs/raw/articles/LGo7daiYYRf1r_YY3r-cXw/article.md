# 从 Vibe Coding 到 Harness—— 一套大仓 AI 工程化实战

> 腾讯技术工程 · 作者：fitchzheng、leoshli · 2026年7月7日 17:39 · 广东

> 一篇属于后端微服务 + 前端微应用大仓的 Harness 实战分享。不讲理念有多重要，只讲我们这一路是怎么搭起来、怎么撞墙、撞完怎么补的。

---

## 背景：TAB A/B 实验平台

TAB 是腾讯内部的 A/B 实验平台。30+ 微服务、10+ 前端微应用、各平台 SDK 库，集成在大仓统一管理。Go/Ts 为主，Docker Compose 沙箱做集成验证。

**四个真实痛点**（没有一个是模型问题，全是协作/流程/信任问题）：
1. PRD 不能被信任 — 一句话需求背后有无穷的隐含影响
2. 方案对不上代码 — 方案漂亮但落不了地
3. 改完没人验证 — "改完了"不等于"跑过了"
4. 交付环节碎成一地 — TAPD/iWiki/工蜂/MR 动作全靠人记

## Harness 六层资产

### 1. Rule（最少）
能判定的不留 Rule — 全部下沉为 Skill 或 lint 脚本。"数据访问层必须用 GORM" → lint 检查器。"错误用统一错误码" → lint 检查器。

### 2. Skill（11 个，4 组）
**注意力管理而非能力增强**。每组解决不同问题：

| 组 | 数量 | 用途 |
|----|------|------|
| 脚手架与代码层规范 | 4 | 协议/数据访问/服务层/HTTP 层 |
| 测试规范 | 2 | 单元测试/接口测试 |
| 需求理解 | 3 | PRD 自评/影响半径/模板 |
| 代码审查 | 2 | 范围溢出检查/前端视觉审查 |

### 3. Sub Agent（4 个）
1. **需求 Agent** — TAPD 链接 → 结构化的需求理解 + 影响半径分析
2. **方案 Agent** — 需求 → 可执行技术方案 + 任务清单
3. **开发 Agent** — 按设计写代码 + 接口测试产物（业务代码 + 单测 + 接口用例文档 + 增量接口测试代码）
4. **代码审查 Agent** — 4 维度审查（方案一致性/验收覆盖/质量基线/前端增量）

**核心纪律**：下游 Agent 不可直接修改上游产物。只能提阻塞项，由总控 Agent 打回上一棒。

**为什么只有 4 个**：能写成 bash 的就别让 Agent 跑。

### 4. Workflow（13 阶段接力赛）
初始化 → 需求分析 → 需求确认 → [Bug 复现] → 技术方案 → [方案确认] → 方案评审 → 分支准备 → 开发 → 集成测试 → 代码审查 → 验收 → 交付收尾

每个阶段独立因为：有独立失败模式 + 失败时回退位置不同。

**关键决策**：集成测试前置到代码审查之前。代价：开发 Agent 产出口径扩到"业务代码+单测+接口用例文档+接口测试代码"。收益：CR 打回率从 1.8 次 → 0.4 次。

### 5. Scripts（7 道门禁）

| 阶段 | 门禁 | 类型 |
|------|------|------|
| 开发完成 | 覆盖率检查（全量≥40% 增量≥85%） | 软 |
| 开发完成 | 制品一致性检查 | 软 |
| 开发完成 | 静态检查+安全扫描（golangci-lint 12检查器） | 硬 |
| 集成测试 | 沙箱部署 | 硬 |
| 集成测试 | 接口测试（真实 HTTP） | 硬 |
| 集成测试 | 前端冒烟（Playwright） | 软（条件触发） |
| 验收 | 日志验收 | 软（可选） |

**核心设计**：
- **基线对比**：开发前跑基线快照，开发后只对比增量失败，剥夺 AI "这是历史问题"的解释权
- **软门禁伤疤**：软门禁失败不阻断但留警告痕，代码审查必然读到

### 6. MCP（5 个外部接口）
TAPD MCP / iWiki MCP / 工蜂 MCP / Knot MCP / 企微 MCP

**原则**：读多写少 / 写操作有清晰触发点 / MCP 调用全部留痕

## 关键撞墙：Team Mode 卡死

**症状**：人工关卡必现卡死。用户输入后主 Agent 无响应。

**真相**：Team Mode 中主 Agent "下班"后，用户消息仍在收件箱无 Agent 消费。

**根因**："复杂度自给自足"陷阱 — Team Mode 的存在主要为了 fix 它自己引入的问题。

**决策**：删除 Team Mode，改用项目级子 Agent + 同步阻塞调用。删掉主 Agent 200+ 行代码，问题彻底消失。

---

## 人工关卡设计（5 个）

1. **需求确认** — 10~30 秒，看 5 行摘要
2. **方案确认** — 30~60 秒，40% 需求触发
3. **方案评审打回** — 1~2 分钟，<10% 触发
4. **前端冒烟选择** — 5 秒，三选一
5. **熔断暂停** — 异常情况

**硬约束**：不能让用户每分钟点一次以上。

**三级兜底**：UI 按钮 → 纯文本关键词 → 语义识别

## 项目级记忆（两层）

1. **仓库代码导航地图**（~26KB）：由开发 Agent 维护，"这座工程长什么样"
2. **任务看板**：跨需求的总控 Agent 维护的 CheckPoint

> 团队要对齐的知识必须落到仓库；Memory 只放纯个人偏好。

## 实测数据（50+ 实际需求）

| 指标 | 当前值 |
|------|--------|
| 端到端耗时（不含人工等待） | 30~75 min |
| 人工介入次数 | 2~3 次 |
| 集成测试一次通过率 | ~70% |
| CR 阻塞率 | ~25%（多数为"方案一致性"维度） |
| 开发阶段平均打回次数 | 0.4 次（接口测试前置前: 1.8 次） |
