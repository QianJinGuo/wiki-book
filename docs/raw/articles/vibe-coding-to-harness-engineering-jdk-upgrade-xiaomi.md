# 从 Vibe Coding 到 Harness Engineering：JDK 升级中可控演进的 AI 工程实践

小米技术 2026年7月10日

## 核心问题

维护性工程（框架迁移、版本升级、安全修复）占工程团队大部分时间，但"写出来"不是难点，"改得对"才是。AI 直接 Vibe Coding 方式做这类工作时，容易修改范围失控、依赖混乱、配置冲突。

## Harness Engineering 三部分框架

### 1. 构建执行约束——让 AI 在正确边界内工作

**项目上下文构建**：把散落在资深开发者经验中的隐性知识变成 AI 可消费的结构化规范。包括项目结构文档、技术栈文档、构建指南、第三方集成约束。

**工程工具编排**：把工程动作封装成输入输出明确、失败路径清晰的可调用工具。AI 只需要决定"调用哪个工具"，保证对固定问题表现稳定。

### 2. 沉淀工程经验——把一次升级变成长期能力

**Skill: jdk-upgrade**，明确 6 条核心约束：禁止修改业务逻辑、禁止优化或重构、禁止添加新功能、禁止删除原有功能、仅做必要兼容修改。

渐进式组织知识：SKILL.md 作为入口，references/ 目录下 Maven/Dependency/JVM/Code 四个参考文件通过渐进式披露加载。

5 步标准化执行流程：
1. Maven 插件升级（目标：构建工具链支持 JDK 21 + JUnit 5）
2. Pom 依赖处理（javax → Jakarta EE 迁移）
3. JVM 参数调整（移除 CMS 等废弃参数）
4. 代码兼容修改（包名替换、JUnit 4→5 迁移，含反馈闭环）
5. 验证（编译 + 测试，含反馈闭环）

**经验回流 Skill: jdk-upgrade-lessons**：升级完成后自动触发，按 5 个维度（阻断性问题/隐蔽陷阱/配置遗漏/依赖冲突/代码迁移）引导经验沉淀，并基于经验生成新版本 Skill（如 jdk-upgrade-21）。

### 3. 建立反馈闭环

测试前置驱动验证策略：
- **升级前**：AI 撰写集成测试（@Timeout、真实 Spring 上下文、结构化输出）
- **升级后**：重新执行测试集，AI 根据失败信息定位修复
- 验证通过才确认升级成功，防止 AI "谎报结果"

## 效果数据

| 指标 | 结果 |
|------|------|
| 文件修改量 | 30+ 个文件 / 50+ 个依赖变更 |
| 一轮完成升级比例 | **70%**（无人工介入单次完成） |
| CPU 使用率 | 平均降幅 **22.5%** |
| 堆内存 | 同等负载下降约 **23%** |
| GC 停顿时间 | 降至 1ms 以下 |
| P99 延迟 | 平均降幅 **18.1%**（最大 47%） |
| 峰值 QPS | 提升 **30%+** |

## 核心判断

- **规范正在成为比代码更重要的资产**：项目约束、升级经验一旦结构化沉淀，可持续被 AI 复用。代码反而变成了产物
- **文档的首要读者正在从人转变为 AI**：工程知识是为了约束 AI 而非约束人
- **开发者的角色从代码编写者转变为约束设计者**：当 AI 承担执行层后，人的核心价值是定义"什么能做、什么不能做、做到什么程度"
- **真正提升工程效率的，不是让 AI 更聪明，而是让工程体系能够持续约束、复用和进化 AI 的能力**
