# 智能UI用例生成与执行的四阶进化实践

> 作者：苗星 & AiDD
> 来源：快手技术（微信公众号）
> 原文链接：https://mp.weixin.qq.com/s/N7sZlK4G4q36PV0m3Rk6LQ

快手在过去两年AI+测试探索中，从Prompt工程一路走到Agentic自进化的四阶进化之路。整理自快手研发Agent负责人苗星在 AiDD 大会的分享。

## 一、痛点诊断

测试效率的真实瓶颈不在执行自动化，而在"用例从何而来、质量能否保证、怎样能规模复用"。

- 用例编写占比仅13%，执行却消耗38%资源
- 资深QA和新手写出的用例质量天差地别（知识孤岛化）
- 历史缺陷知识难以系统复用
- PRD变更频繁，维护成本高

**假设：** "用例不应该是'一次生成就完成'的产物，而应该像代码一样，经历设计、评审、迭代、沉淀的全生命周期。"

## 二、V1.0：Prompt工程探索

**做法：** Few-shot学习（3-5个样本） + 场景化Prompt模板 + 输出强约束

**流程：** PRD + Few-shot + 场景化Prompt → LLM推理 → 结构化用例

**实际效果：** 生成率仅8%。用户反馈"生成得太泛了，没法直接用。"

**三个根本问题：**
1. 长文档理解衰减：PRD超过30K tokens时模型遗忘前文
2. 业务知识缺失：隐含规则不会被模型自动推导
3. 生成过程黑盒：用户无法干预中间步骤

## 三、V2.0：Multi-Agent协作

把QA的工作流复制出来：PRD解析Agent（拆解测试点）→ 用例生成Agent（生成结构化用例）→ 用例评审Agent（质量把关）

**分工：** AI负责快速生成和结构化框架，人负责Review决策和纠偏

**效果：** 生成率8%→15%（+87%），采纳率40%→55%（+37%），用户渗透20%→30%（+50%）

**瓶颈：** 生成的用例"正确但浅显"，缺乏业务知识。

## 四、V3.0：知识工程

**体系化知识注入的四种资产：**
1. 文本测试物料（PRD、技术方案、代码DIFF）
2. 多模态测试物料（UI原型图、流程图、架构图）
3. 历史用例知识（RAG召回历史缺陷记录）
4. 业务私域知识（业务规则图谱、团队模板）

**RAG流程：** 查询→检索→增强→生成。如生成"直播送礼"用例时，自动检索历史缺陷"并发送礼导致重复扣款"。

**效果：** 生成率15%→35%（+133%），采纳率55%→65%（+18%），用户渗透30%→50%（+67%）

**关键：** 历史缺陷覆盖率从12%→76%（+533%）

**代价：** Review成本依然高，维护170+个业务私域模板需要专人，知识更新有滞后。

## 五、V4.0：Agentic自进化

**三条主线：**
1. **自主决策 & ReAct Loop：** 三层资产（Skills 170+种定制、Memory 用户偏好、Knowledge 知识图谱）
2. **自主评审 & Review-Critique：** 模块级评审（覆盖完整性/结构合理性/层级清晰度/命名规范性）+ 用例级评审（PRD覆盖度/场景完整性/步骤清晰度/预期明确性）
3. **BadCase规则库 & 自进化：** 低质量用例→提取模式→规则库→自动召回纠正→知识沉淀

**关键数字：** 场景规则模板更新从天级降到5分钟，维护成本下降99%。全公司推广，累计支持120W+用例生成。生成率从35%提升到70%。

## 六、智能用例执行：从"能生成"到"能可靠执行"

**第三条路（非纯脚本也非纯自由探索）：** 测试意图驱动。

**前提1：** 用例必须是AI-Friendly（精简全面、原子化步骤、可观测预期、环境依赖显性化）
**前提2：** 执行引擎"四觉"能力（感知→决策→执行→反馈）

## 七、四层架构方法论

1. **场景分层：** 优先突破覆盖类+质量类问题，再重点攻关垂类场景
2. **用户运营：** BadCase驱动闭环
3. **知识运营：** 自动沉淀知识资产
4. **Agentic架构：** System Prompt动态组装、Skills扩展、Memory学习、Knowledge自动检索

## 八、总结

**规律：**
1. 任何工作流的自动化，都应先复制其"分阶段审查"结构
2. 生成质量上限由知识丰富度决定，非Prompt精妙程度
3. BadCase系统化收集是自进化的唯一可靠燃料
4. AI最终取代高频工作，但永远取代不了最关键决策

**建议：** 从高价值低复杂度场景开始；历史缺陷库 > 业务规则模板 > Prompt优化；从V1就要设计反馈闭环；明确人机分工边界。
