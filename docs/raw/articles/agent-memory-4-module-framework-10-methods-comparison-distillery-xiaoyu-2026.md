# Agent 记忆怎么设计才靠谱：Memory in the LLM Era 论文 4 模块框架 + 10 方案对比 + 新方法 F1 38.79 + 4 条工程原则

**作者**：蒸馏小余 | **发布时间**：2026-05-06 08:00

**参考论文**：Yanchen Wu, Tenghui Lin, Yingli Zhou, Fangyuan Zhang, Qintian Guo, Xun Zhou, Sibo Wang, Xilin Liu, Yuchi Ma, Yixiang Fang. *Memory in the LLM Era: Modular Architectures and Strategies in a Unified Framework*. **arXiv:2604.01707**, 2026-04-02 (ICLR 2026 投稿).

**PDF**: https://arxiv.org/pdf/2604.01707

## 序：Agent 记忆不是普通聊天记录

Agent 记忆系统不能只靠"把历史对话塞进向量库"解决。长期运行的 Agent 需要处理的是一套数据系统问题：哪些信息值得留下，旧事实怎么更新，冲突版本怎么处理，查询时如何找回正确证据。

- 个人助手需要记住用户偏好、项目背景和最近变更
- 代码 Agent 需要记住仓库结构、失败测试和用户长期约定
- 科研 Agent 需要在多轮探索里积累证据，而不是每轮从零开始

只把历史拼进上下文，会遇到三个瓶颈：
1. 历史越来越长，token 成本持续上涨
2. 旧事实和新事实会冲突，需要版本判断
3. 相关证据可能分散在多个 session，不能只靠相似度检索

所以 Agent Memory 更接近一套长期运行的数据基础设施，而不是一个 prompt 技巧。

### 论文工程判断

- 层次结构通常更稳
- 原始证据不能丢
- 关系连接很重要
- 复杂的 LLM 自主管理不一定比清晰的规则式架构更划算

## 四模块框架

作者把记忆系统拆成四个模块：

| 模块 | 职责 | 常见做法 |
|------|------|---------|
| 信息提取 | 从当前消息里抽什么 | 原文归档 / 摘要提取 / 图谱三元组抽取 |
| 记忆管理 | 新旧记忆怎么合并和更新 | 连接 / 整合 / 层级迁移 / 更新 / 过滤 |
| 记忆存储 | 记忆以什么结构保存 | 扁平 / 层次 / 向量 / 图 |
| 信息检索 | 查询时怎么找回记忆 | 关键词 / 向量 / 结构 / LLM 辅助 |

不同方案名字很多，底层差异基本都落在这四个模块里：
- **Mem0**：更强调从对话里提取和更新记忆
- **Mem0^g**：加入图谱式表达
- **MemoryOS**：把短期/中期/长期记忆分层
- **MemTree / MemOS**：用树结构组织多粒度信息
- **Zep**：更依赖图结构和社区组织

## 信息提取：结构化之前，先保住原始证据

三类信息提取：

1. **直接归档**：保存原始消息和时间戳，不聪明但不容易丢细节
2. **摘要式提取**：LLM 提炼关键词/主题/标签/简短摘要，能减少冗余但压缩过程可能抹掉上下文
3. **图谱式提取**：抽出实体和关系转成"主体-关系-客体"三元组，方便关系推理但**抽取错误会沿后续存储和检索继续放大**

**关键观察**：结构化表示不一定带来更完整的记忆。Mem0 的图谱版本 Mem0^g 在一些任务上并不总是优于普通 Mem0。三元组保留关系却可能压掉语气、上下文和细粒度事实。

**更稳的设计是双轨保存**：结构化信息负责组织和检索，原始对话片段负责校验和补细节。

## 记忆管理：难点是更新，不是写入

五类管理操作：

1. **连接相关经历**：把语义相近、时间相近或上下文相关的记忆连起来
2. **整合碎片信息**：把重复或零散内容压缩成摘要、主题或用户画像
3. **跨层级迁移**：把短期记忆逐步转成中期、长期记忆
4. **更新已有记忆**：处理新旧事实冲突，保持记忆一致
5. **过滤过时内容**：删除、降权或标记不再有效的信息

### LLM-as-OS 方案的扩展性陷阱

论文在上下文扩展实验里观察到：**MemOS、MemGPT 这类更接近"LLM-as-OS"的方案，在上下文规模扩到 200% 时，更容易受到候选空间变大、工具调用复杂和索引冲突的影响**。MemoryOS 使用更明确的规则式分层管理，扩展稳定性反而更好。

### 工程结论

底层、高频、影响一致性的记忆操作，**需要确定性机制托底**；LLM 更适合参与需要语义判断的环节。

## 存储结构：单层向量库不够用

两个维度：

1. **组织方式**：扁平 vs 层次
   - 扁平存储：实现简单，但历史变多后检索空间会变脏
   - 层次存储：短期/中期/长期分生命周期
2. **表示方式**：向量 vs 图
   - 向量适合语义相似度检索
   - 图适合关系、层级和多跳推理
   - **两者不是竞争关系**

### Benchmark 数据

| 评测 | 7B F1 | 72B F1 | 最佳方法 |
|------|-------|--------|---------|
| LONGMEMEVAL | 36.92 | 46.04 | MemTree (7B) / MemoryOS (72B) |
| LOCOMO | 37.05 | 42.79 | MemOS (both) |

高分方法共同点：**都没有把记忆压成单层向量空间**。树结构和分层结构能同时保留上层概括和下层细节。检索时先定位范围，再下钻到证据。

## 检索机制：相似度不等于推理

四种检索机制：

1. **关键词检索**：适合名字、术语和精确短语
2. **向量检索**：适合同义表达和语义接近内容
3. **结构检索**：适合沿图或树做扩展
4. **LLM 辅助检索**：适合改写查询、识别实体、判断隐含依赖

### 经典反例

用户问"我上次提到的那个项目后来换负责人了吗？" — 系统可能需要跨多个会话串起项目名称、负责人、更新时间和最新状态。**单纯 top-k 相似度可能召回几个相关片段，但不保证能把版本关系和多跳关系组织起来**。

论文观察：缺少关联操作的方法（MemoryBank / MemGPT / MemoChat）在 LONGMEMEVAL 的 Multi-Session 任务和 LOCOMO 的 Multi-Hop 任务上表现偏弱。

> 长期记忆不是更长的搜索框，而是一个能保存关系、版本和证据链的系统。

## 实验设置

### 10 个代表性方法
A-MEM, MemoryBank, MemGPT, Mem0, Mem0^g, MemoChat, Zep, MemTree, MemoryOS, MemOS

### 两个 benchmark

- **LOCOMO**：长期人类对话问答任务，每段对话平均 27.2 session / 588.2 轮对话。问题类型：单跳检索 / 多跳检索 / 时间推理 / 开放域知识
- **LONGMEMEVAL**：长期用户-AI 交互记忆评测，500 个高质量问题，每个问题对应平均 50.2 session / 115,000 tokens 历史上下文。评估：信息提取 / 多会话推理 / 知识更新 / 时间推理

### 主干模型
Qwen2.5-7B-Instruct (默认) / Qwen2.5-72B-Instruct / LLaMA3.1-8B / GPT-4o-mini。Embedding: all-MiniLM-L6-v2 (统一)。

## 强模型仍然重要，但系统不能只靠模型硬推

时间推理任务很依赖主干模型规模。比如在 LOCOMO 上，**MemoryOS 和 MemoChat 从 Qwen2.5-7B 换到 Qwen2.5-72B 后，时间推理表现出现超过 2 倍的提升**。

更稳的方向：把时间能力变成系统结构的一部分（显式时间戳、版本记录、失效标记、时间检索算子）。**模型负责解释和生成，系统负责提供可靠的时间线**。

## 高分通常更贵，架构决定性价比

整体趋势：性能更高的方法，通常消耗更多 token。

| 方法 | 准确率 | Token 成本 |
|------|--------|----------|
| MemTree / MemOS | 高 | 高 |
| MemoryOS | 中高 | 中（均衡） |
| MemoChat / MemoryBank | 低 | 低 |

### 信息处理粒度优化

如果每一轮对话都单独抽取/更新/插入，记忆系统成本会随着会话增长快速上升。**MemoryOS 把对话组织成 segment 再进入中期记忆**，MemoryBank 有按天汇总的思路。

记忆粒度不是越细越好。太细会让索引膨胀、更新昂贵、检索噪声变大；**适当粗粒度反而能提高成本效率**。

## 上下文越大，检索噪声越难压住

LONGMEMEVAL 构造了 50% / 100% / 150% / 200% 四种上下文规模。结果：**上下文规模从 50% 扩到 200% 后，几乎所有记忆架构的 F1 都会下降**。

- 性能下降主要原因不是模型突然变弱，而是检索空间变大后**无关信息密度上升，信噪比下降**
- **知识更新任务尤其敏感**（用户从上海→深圳→杭州，系统必须回答杭州）
- 时间推理相对稳定（时间关系有明确顺序线索）

> 工程评测不能只看小上下文。一个记忆系统在 10 条历史上表现稳定，不代表它在 1000 条历史上还能可靠。

## 早期证据更容易被遗忘

位置敏感性实验：关键证据放到 Early / Middle / Late 不同位置。

多数方法都有**近因偏差**：后期证据更容易被召回，早期证据更容易被后续对话冲淡。

| 方法 | Early F1 | Late F1 | 差距 |
|------|---------|---------|------|
| MemTree | 34.13 | 37.37 | +3.24 |
| MemOS | 29.10 | 34.40 | +5.30 |
| MemoryOS | - | - | **+1.29**（差距最小） |

近因偏差不一定总是坏事——用户刚刚修改地址时系统应该优先使用最新信息。

**问题在于系统不能把"最新"误当成"永远正确"**。有些任务需要回忆旧事实（例如"我第一次提到这个项目时的目标是什么"）。如果记忆系统没有版本管理和证据保留机制，更新任务和历史回忆任务会互相干扰。

## 新方法（lme-sota）：树结构加三层记忆

基于实验分析，作者设计的新方法不是从零发明概念，而是组合已有方案里的有效模块：

- 借鉴 MemTree、MemOS 的树状组织
- 借鉴 MemoryOS 的短期、中期、长期分层
- 用更粗粒度的 segment 降低 token 成本
- 从多个层级独立检索，再合并上下文

### 工作流

```
新消息 → 短期记忆
            ↓
       短期记忆满了
            ↓
       最旧部分按语义切成片段
            ↓
       片段进入中期记忆树（叶子节点保存片段摘要，父节点保存聚合摘要）
            ↓
       经常被访问/最近仍然重要的片段 → heat score 晋升到长期记忆
            ↓
       查询时同时检索短期/中期/长期记忆 → 交给 LLM 生成答案
```

### 核心数据

| Benchmark | 主干 | Overall F1 | Token/对话 |
|-----------|------|-----------|-----------|
| LONGMEMEVAL | Qwen2.5-7B | **38.79**（vs MemTree 36.92, +5.17%） | < 450 |
| LOCOMO | Qwen2.5-7B | 38.03 | - |
| LOCOMO | Qwen2.5-72B | 43.87 | - |

**新方法并不是每个子任务都第一**——LOCOMO 里 MemOS 在多跳检索和时间推理等复杂任务上仍然很强；GPT-4o-mini 设置下 MemOS 的 Overall F1 也略高于作者方法。

**作者方法的优势更接近工程上的综合性价比**：整体分数高，token 成本低，换模型后表现也比较稳定。

## 4 条工程原则（蒸馏小余提炼）

1. **先搭层次结构，再引入复杂智能管理**
   - 短期/中期/长期分层 + 每层明确迁移规则
   - 不要一开始就追求全自动记忆管理

2. **保留原始证据**
   - 摘要/标签/三元组/embedding 都是索引不是事实本身
   - 结构化信息负责提速，原始片段负责校验

3. **给记忆加版本意识**
   - 保留历史 + 标注有效期/更新时间/置信度/失效状态
   - 直接覆盖旧记忆会丢掉解释链路；只追加新记忆会让旧版本污染检索

4. **评测要同时看准确率、成本、规模和位置**
   - 还要测 token 成本、上下文扩大后的衰减、早期证据能不能找回、换主干模型后是否稳定
   - Demo 里的好表现不能自动推导到生产环境

## 落地 4 个问题

把这些边界设计清楚，Agent 记忆才会从"看起来聪明"变成"长期可靠"：

1. 哪些信息应该进入记忆？
2. 新旧事实冲突时怎么更新？
3. 查询时先检索哪一层？
4. 旧证据什么时候失效，什么时候只降权？
