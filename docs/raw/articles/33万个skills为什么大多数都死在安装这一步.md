# 3.3万个Skills，为什么大多数都死在“安装”这一步？

作者： Rambo，sherry ，laven

谁先成为入口，谁才配留在桌面上。

我们拆了2026年3月24日的一份Clawhub公开数据快照，样本共33,760个skills。

这个市场的问题，不是供给不足，而是 大部分供给没有穿过安装这道门 。

下一阶段真正会赢的，不是功能最多的，也不是概念最大的，而是那些能让用户迅速完成一个动作、迅速看到结果、迅速形成依赖的 skill。

这意味着Clawhub已经不缺供给，缺的是能真正形成安装的供给。
很多skill完成了曝光，没有完成安装，而安装，已经明显向头部集中。

这组分布说明，Clawhub先长出来的，不是娱乐型需求，也不是人格型需求。而是工作流型需求。 用户优先安装的，仍然是能直接接进现有任务链的工具。

代码、搜索、文档、浏览器、数据库、摘要、转写、导出、同步 ， 这些都属于同一类需求： 不是重新发明流程，而是缩短原有流程。

这组数据给出的结论很直接：开发/编程仍然是主航道。不是“供给多所以卷死了”，而是“供给大，安装也强”。办公/文档、搜索/研究、图像/音视频已经形成稳定装机层。

电商/营销、金融/分析的问题最明显，不是没有需求，而是安装承接弱。

四、真正成立的 skill，都更接近现有流程

从类别表现看，当前更容易成立的skill，基本都有四个共同特征：

1.输入明确  用户知道该给什么
2.输出明确  用户知道会得到什么
3.结果可验证  用户能判断对不对
4.离现有流程近  不要求学新范式

所以GitHub、SQL、数据库、浏览器、CLI、PDF、文档、邮件、日历、知识库、摘要、 转写、抓取、导出、同步这些方向更容易跑出来。

原因并不复杂。它们不要求用户先理解一个新范式。只要求用户把一个旧动作更快做完。这也是当前Skill市场最重要的产品逻辑：谁更接近流程入口，谁更容易获得安装。

五、头部 skill 的成功公式

只看头部样本，代表性的 Top skill 包括 Summarize、Github、AgentBrowser、Weather、Gog 等。它们跨赛道，但共性高度一致。

🔑 零配置门槛  不需要 API Key、不需要填参数、不需要前置设置
💬 一句话讲清楚  用户不需要读文档就知道它能做什么

🔥解决高频问题  不是偶尔用一次，而是反复依赖
🔗接入已有生态  GitHub、浏览器、文档……用户已经有这些工具

这五条，比“属于哪个赛道”更重要。因为用户当前做的不是深度比较。而是快速判断：值不值得装，装完能不能立刻用。

所以头部skill真正的共性，不是功能最多。而是判断成本最低。

六、高转化黑马：垂直和明确

高转化黑马更能说明问题。这批样本里，代表包括 Canvas（32.2% 转化）、Ordercli/Blucli/Eightctl（12%–13%）、FoodOrder（12.3%）。

它们的共同点不是最泛化。而是： 场景窄，动作具体，不需要学习，价值立即可感 知。

❌多功能大而全  像一个小型操作系统   用户不知道从哪开始用
✓低门槛 + 高确定性  像一个明确动作   越像动作越容易被安装

这说明 当前Clawhub奖励的不是“多功能”。而是：低门槛+高确定性，skill不是越像一个小型操作系统越容易成功。反而是越像一个明确动作，越容易被安装。

七、电商/营销和金融/分析为什么更难

这两类并不是没有需求。问题在于，它们更容易停留在“泛能力描述”上，而不是“短链路动作”上。

用户需要的
当前常见问题

✕ "会写文案、会做分析"

✓ 一次调用可感知价值
✕ "会提建议、会给策略"

✓ 不需要理解新范式
✕ 切口不够锋利

先看电商/营销，它的典型问题是

会写 文案、会做分析、会提建议、会给策略，但不一定形成一个必须安装的入口。从数据上看，它表现为：安装效率偏低。零安装比例高。供给饱和度也高。这说明问题不在需求。而在切口不够锋利。

再看金融/分析

它的问题主要有两个：结果验证周期更长。skill形态还不够短链和直接，所以用户会下载、会看看。但不太愿意把它装成默认入口。这两类赛道的问题 ， 不是价值不存在。而是价值到达得太晚。

## 八、配置成本，正在直接杀死转化

文档里还有一组对比很说明问题：

无需 API Key  11.4  平均安装数
需要 API Key  3.5  平均安装数

这不是小波动。这是结构性差异。当前用户面对skill时，真正的决策不是“这东西厉不厉害”。

而是：能不能立刻装，能不能立刻用，要不要先去找key、配账号、填参数。配置成本本身，就是转化门槛。

结语

Clawhub(Agent)现在不缺skill，缺的是值得安装的skill。再说得更直一点：

大多数skill失败，不是因为赛道不对，不是因为用户不懂，也不是因为平台不给机会。而是因为 它们既没有缩短流程，也没有缩短判断，更没有缩短价值到达时间 。

下一阶段真正会赢的，不是功能最多的，不是概念最大的，而是那些能让用户迅速完成一个动作、迅速看到结果、迅速形成依赖的skill。
