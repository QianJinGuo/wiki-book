**文章主旨**：从历史演进和结构性优势两个维度，分析 CLI 在 Agent 时代复兴的根本原因，提出 CLI/MCP/SKILL 三层分层模型（食材/预制菜/菜谱）和 AI 友好产品设计四原则。

**1. CLI 的进化史**

- 第一次：CLI（1970s-1990s）—— 人类用文本跟计算机说话
- 第二次：GUI（1990s-2025s）—— 人类用鼠标和窗口跟计算机说话
- 第三次：Agentic CLI（2025s-）—— 人类跟 AI Agent 说话，AI Agent 跟计算机说话

Agent 需要读文件、写代码、跑测试、管 Git、调 API——它需要的不是好看的按钮和语法高亮，而是文本输入、文本输出、完整的系统访问权。

**2. Agent + CLI 结构性优势（五个维度）**

**天然同构**：LLM 是 text-in、text-out 的机器。终端是 text-in、text-out 的界面。让 AI 操作 GUI 得截图→视觉模型识图→模拟鼠标点击，一行命令的事拆成四步，每步都可能出错。

**自描述性**（CLI 相对于 API 最被低估的优势）：AI 敲一下 --help 就知道 CLI 有哪些能力。API 需要文档、端点、认证方式。工具本身就是文档。

**Unix 管道与组合**：antwork calendar agenda --next-week | grep "张三" | wc -l 一行查出下周和张三有几个会。MCP 适合标准操作，CLI 可即兴编排。

**并行原生**：CLI 命令本质无状态、可序列化。一个字符串就是一个完整指令。Agent 可批量生成命令、并行分发、独立重试。

**上下文干净**：MCP 把工具清单常驻上下文窗口，就算不用也占空间。CLI 无此开销。

**3. CLI / MCP / SKILL 三层分层模型**

- **CLI = 食材**：最灵活的原子操作。灵活性最高，每次从零组装。
- **MCP = 预制菜**：常用操作封装成结构化工具。省事，但只有菜单上的品种。
- **SKILL = 菜谱**：完整编排逻辑，组合多个 CLI 命令和 MCP 工具。告诉你先放什么后放什么、火候多大。

这三个不是二选一的关系，是能力栈的不同层级。Agent 需要的是根据场景在三层之间自由切换。

CLI 的繁荣不会让 MCP 衰落——相反，CLI 层积累的操作经验会沉淀为 MCP 工具和 SKILL 工作流。CLI 是创新的试验场，MCP 是标准化的沉淀池，SKILL 是最佳实践的固化层。

**4. AI 友好产品设计四原则**

1. **可调用**：能力不能锁在 GUI 里。每个有意义的操作应有 CLI、API 或 MCP 接口。GUI 是给人的前门，Agent 需要一个同样通畅的后门。
2. **可理解**：Agent 拿到接口后不需要翻 200 页手册。--help 就是最好的例子——工具自己会说话。
3. **可组合**：每个操作是原子的，多个操作可以自由串联。如果每个操作输入输出格式不同，Agent 就没法组装新流程。
4. **可恢复**：操作最好是幂等的，状态变更是可追溯的，失败不会导致不可逆损害。

**5. 人机协作的可观测性**

- **计划可观测性**：Agent 先出方案让用户审（plan 模式），从逐行审批变成审批方案
- **过程可观测性**：Agent 执行时展示正在调什么工具、读哪个文件、改哪几行，人可随时叫停

核心命题：人需要在"完全不放手"和"完全放手"之间找到合适位置。好的协作界面让人花 10% 注意力获得 90% 掌控感。

**6. Agent First = 新设计范式**

类比 Mobile First 时代（2010 年），Agent First 要求重新思考三个根本问题：
- 能力怎么暴露：每个产品能力应有"人能操作"和"Agent 能调用"两个版本
- 数据怎么流动：从"展示给人看"重构为"既展示给人看，也喂给 Agent 用"
- 协作边界画在哪：动态可调，非代码里写死
