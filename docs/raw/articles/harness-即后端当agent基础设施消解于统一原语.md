当前 AI 基础设施领域的核心问题已不再是模型选型，而是构建可用系统所需的基础设施规模与复杂度。围绕模型外层，业界逐渐形成了一套被称作 Agent Harness（编排基础设施）的结构，涵盖编排循环、工具调用（如 MCP、A2A 协议）、记忆管理、上下文调度与错误处理等模块。各方共识在于模型本身并非最终产品，承载其运行的基础设施才是差异化的关键。

不同方案对 Harness 的设计呈现出明显的谱系特征：从精简型编排到重度结构化编排。精简方案仅保留提示组装、模型调用与工具执行的闭环，将决策权最大程度交由模型；结构化方案则引入指令栈、编排模式与显式交接机制，对流程进行额外约束；更复杂的方案采用确定性流控制与自主 Agent 混合编排，甚至将每个决策点编码为显式节点、每条状态转移定义为有向边，整个工作流由 Harness 完整承载。这一谱系本质上反映了对模型信任程度与逻辑编码强度之间的取舍。

然而在这场讨论中存在一个未被审视的预设：Harness 被默认视为独立于传统 Backend 的额外层。Agent 的循环、工具与记忆被归入编排层，而队列、状态管理、HTTP 路由、服务端渲染、可观测性等传统后端组件则被归入另一侧。这种分层暗示 Harness 是外在于后端架构的附加物，而非后端本身的一部分。

这一分离状态很可能只是技术演进过程中的过渡形态。随着 Agent 架构被更广泛地采纳，编排逻辑与后端基础设施之间的边界将逐步消解，最终融合为统一的后端抽象模型。

## 架构断层：随机性与确定性的范式冲突

当前主流的 Agent 架构中，Harness 与 Backend 作为两个独立的系统运行。Harness 以独立进程的形式包裹模型，当 Agent 决定执行动作时，将 Tool Call 翻译为 HTTP 请求，进而触发后端系统中的队列发布、数据库写入等操作。后端是一个与 Agent 完全隔离的世界。

这两个世界各自运行着独立的控制逻辑。Harness 按照自身的调度策略进行重试，队列根据其内部条件执行重试，HTTP 层管理着独立的超时配置。这些分散的系统之间不存在直接关联的 Trace 机制。当故障发生时，调试意味着跨系统关联日志，从碎片化的观测数据中重建行为路径。

跨系统日志关联在传统后端工程中是常规操作，因为此前的系统大多是确定性的——相同的输入产生相同的输出，问题路径可被精确复现。但 Agent 的行为本质上是随机的。相同的输入可能产生完全不同的决策序列，相同的 Tool Call 可能触发不同的下游调用链。

调试难度随系统中 Agent 数量的增加呈组合级数增长。基本公式为 Agent 数量的平方乘以后端服务数量。一个 Agent 与五个后端系统之间存在五条随机路径需要追踪，四个 Agent 与五个后端系统之间则膨胀为八十条随机路径。每新增一个 Agent，需要排查的随机路径组合都在非线性扩展。

这种随机性并非实现的缺陷，而是设计的本意。Agent 的核心价值恰恰在于对相似甚至相同的输入产生差异化的响应，使计算机能够以全新的方式处理非结构化问题。试图通过约束使 Agent 变得更加确定，与其基本功能目标相矛盾。

当前 Harness 设计的根本前提，是尝试在旧的确定性范式（传统后端）内部运行新的随机性范式（LLM 驱动的 Agent）。Agent Harness 的构建方向本身并非错误，但有效的解决方案必须从解构"后端是什么"这一基本问题开始。

## 后端的三要素抽象

对后端基本构成的探寻，最初容易陷入自顶向下的复杂性陷阱。当试图用服务分类、集成方案、架构图和编排代码去描述后端时，构成要素的清单只会不断增长。这种自上而下的拆解方式，本质上是在罗列现象而非追溯本质。转换到自底向上的视角后，后端的真实结构反而清晰起来。

后端由三种基本原语构成： Worker 作为连接引擎并注册能力的工作进程， Trigger 作为声明式触发器定义函数在何种事件下被执行， Function 作为具有稳定标识符的工作单元，接收输入并可选地返回输出。三者分别承担了后端的连接、调度与执行职责，其余所有复杂性均可还原为这三种原语的组合。

Function 是独立于语言和运行环境的纯粹工作单元，以稳定标识符（如  orders::validate  ）进行寻址。Trigger 是声明式的绑定关系——HTTP 端点、Cron 调度、队列订阅、状态变更、流事件等任何外部信号均可映射到同一个函数上。当新增触发条件时，函数本身无需修改，Trigger 直接在声明层面完成组合。这种解耦使得同一工作单元可以被多种事件源驱动，而无需为每种调用方式编写适配逻辑。

Worker 是任何能够连接引擎、注册函数与 Trigger 的进程。无论是传统的 API 服务、数据处理管线还是微服务，其本质都是向同一个执行引擎注册自身能力的 Worker。当所有服务以统一方式接入时，服务间的通信不再是不同系统之间的桥接，而是同一引擎内部的编排调度。

这一抽象对 Agent 架构的启示在于：Agent 完全可以直接作为 Worker 参与后端系统，而非通过额外的集成层与后端交互。Agent 连接至执行引擎，注册自身的函数与 Trigger，使用与其他服务完全相同的状态持久化机制、队列机制和 Pub/Sub 广播机制。Agent 的工具即函数，记忆即状态，编排即 Trigger 的组合与级联。

由此产生的结构变化是根本性的。Agent 通过队列驱动的 Trigger 将任务交接给其他 Worker，通过状态变更触发下游函数的自动执行，通过同一套原语完成协作与通信。系统中不再需要为 Agent 设计专用的基础设施层，因为 Agent 与其他后端组件共享同一套执行上下文。 Harness 即后端本身 ——当 Agent 与后端服务使用相同的抽象原语时，Harness 不再仅仅是调度 Agent 的控制面，而是构成了后端的运行时基础。这种统一消除了"Agent 调用后端"的层级划分，使 Agent 成为后端执行平面的原生参与者。

## 品类折叠：当答案永远是"添加一个 Worker"

大多数平台中，每项新能力都对应一个独立品类。需要队列，就去评估队列产品；需要流处理，换一套产品；需要沙箱，又是另一套。每个品类有各自的内部机制、生命周期和集成方式，平台本质上是一个产品目录，使用者的工作是在其中挑选并拼装。

而统一原语模式下，几乎所有问题的答案都指向同一件事：添加一个 Worker，由 Worker 注册它所提供的 Trigger 和 Function。需要沙箱，添加一个 Worker；需要一个研究主题的 Agent，添加一个 Worker；需要实时流处理，添加一个 Worker；需要线索评分、邮件序列、CRM 同步等业务能力，同样是添加一个 Worker。Cron 调度是 Worker，可观测性是 Worker，答案不因能力类型而改变，不因编程语言而改变，不因属于基础设施还是业务逻辑而改变，也不因创建者是人类还是 Agent 而改变。

这种统一不止于架构层面的整齐。它带来的是品类折叠（collapse of categories）。传统系统中，每种能力栖居于各自的本体论之下：队列承载 Broker 语义，HTTP 承载路由语义，Cron 承载调度语义，Agent 承载编排语义。而在统一原语中，它们全部归结为同一件事：一个注册了 Function 和 Trigger 的进程。语义存在于 Function 之中，而非基础设施之中。

Worker 连接后注册自身能力，系统随即吸收它：即刻上线、可被发现、可被观测。这套机制对所有能力一视同仁，不区分来源，不区分用途。系统的边界不再是预先定义的产品清单，而是运行时可无限扩展的注册表。

软件领域的范式转变从不靠堆叠功能实现，而是通过折叠品类完成。"一切皆文件"使 Unix 获得了可组合性；将组件抽象为函数使框架的心智模型得以成立。当所有问题的答案收敛为"添加一个 Worker"时，品类边界随之消融，系统从拼装工具变为生长平台——能力不再是静态分类下的固定条目，而是运行时动态涌现的注册行为。

## 三大实时特性与递归能力

当系统统一于单一架构原语时，三种传统架构无法产生的属性自然涌现。它们不是通过叠加组件实现，而是架构模式本身的内生产物。

实时发现 。Worker 接入引擎时，即刻获得全量函数目录——所有已注册 Worker 上声明的每一个 Function 对 newcomer 可见。新函数出现，所有 Worker 收到通知；Worker 断开连接，下线事件广播至全网。引擎作为唯一的真相源，维持着系统能力的实时拓扑。对于 Agent 而言，这意味着认知基础设施的确定性：Agent 始终能够看到系统当前完整的能力边界，不存在接收过期上下文的风险。能力的供给与消费在同一时刻对齐。

实时扩展 。在运行中的系统里添加新的 Worker 和能力，不需要重新部署，不需要重启服务，不需要修改配置。新 Worker 连接、注册 Function 与 Trigger、能力分发到全网，整个过程在运行时完成。任何 Agent 都可以按需调用这些新能力，甚至自主创建 Worker 来扩展系统。这正是 Agentic 系统的运行方式——向生产环境添加新能力无需中断，架构在运行中生长而非在变更中重建。

实时可观测 。可观测性内建于调用链路本身。每一次 Function 调用携带 Trace ID，每一次 Trigger 调用将该 Trace ID 跨 Worker、跨语言、跨队列传递自动传播。所有通过系统日志接口发出的日志，自动关联到活跃的 Trace 与 Span，以结构化格式输出并路由至配置的可观测性后端。这不是需要额外安装集成的独立组件，而是由引擎自身产生的遥测数据——Trace、Metric 和结构化日志从架构底层自然流出，而非应用层中间件的后缀装配。当 Agent 调用一个 Tool，该调用入队触发下游 Function，下游 Function 写入 State，整条链路是一条完整的 Trace。不是三个独立系统通过时间戳关联或手动追踪 Trace ID 勉强拼接，而是跨越语言、跨越 Worker、跨越 Agent 与后端边界的统一追踪。从瀑布状的 Span 直接到达关联日志，定位问题的路径被折叠为一步。

递归能力将这一模式推向更深层。系统支持硬件隔离的微虚拟机 Worker，沙箱本身就是一个 Worker——拥有独立的文件系统、网络栈和进程树，通过单一命令创建并接入引擎。沙箱注册自己的 Function 和 Trigger，以与其他 Worker 完全相同的方式参与系统运行。

当 Agent 具备这种能力时，模式发生了质变。Agent Worker 可以在运行时启动一个新的沙箱 Worker，该沙箱获得隔离环境，注册自身的 Function 和 Trigger，这些 Function 立即出现在全系统的实时目录中，其他 Agent 和服务可以调用它们。沙箱不再需要时，断开连接即自动注销。沙箱不是独立的沙箱产品，它是使用相同原语的 Worker，恰好提供了硬件隔离能力。一个 Worker 创建另一个 Worker，这是同一套机制的自然延伸。

需要隔离执行不受信任的代码，这是沙箱 Worker。需要临时专用 Agent，创建 Worker、注册 Function、任务完成后销毁。需要并行任务执行集群，让 Worker 创建其他 Worker。原语不变，模式变化。基础设施从产品品类转变为设计模式，统一架构的递归性质使得系统能力的组合方式不再受预先设计的限制。

## 边界消解：Harness 即后端

回到开篇的 Harness 设计之争——架构应当精简还是厚重，认知结构应当交由模型自行判断还是显式编码。当所有 Agent 都被视为 Worker 时，这场争论的性质发生了变化。精简 Harness 与重度 Harness 不再是两种对立的架构范式，而只是同一组原语之下的不同组合方式：前者注册少量函数，由模型自主决定触发路径；后者注册更多函数，引入审批门控与条件逻辑来控制步骤编排。底层原语相同，差异仅在于设计模式的选择。

脚手架的演进逻辑也随之改变。行业普遍认为 Harness 脚手架是临时性的，随着模型能力提升应当逐步移除。在统一 Worker 架构下，移除脚手架不再意味着重构集成层或重建两个系统之间的接口。它仅仅意味着注册更少的函数，或者以不同的方式组合已有的函数。架构简化回归为函数层面的调整，而非系统级别的重新设计。

Worker 的边界是无限的。任何能够建立连接、注册函数、遵循原语协议的事物都是 Worker。协议本身是开放的，语言、设备、运行时都不是系统的边界。云端服务、Agent、浏览器标签页、边缘传感器、临时启动的隔离沙箱，它们在系统中享有同等地位。引擎不区分这些参与者的身份，它看到的只有函数、触发器和连接。新语言、新设备、新运行时只要实现原语接口，便自动获得完整的实时发现、实时扩展、实时观测和跨一切调用能力。这不是为每种参与者构建专属集成，而是原语本身允许这种组合。

当 Agent 基础设施不再与后端基础设施被当作两个独立世界来对待，当每一类参与者不再被赋予架构层面的特殊身份，系统复杂性会以添加功能永远无法达到的方式被消解。Harness 与 Backend 的边界、云端与边缘的边界、基础设施与应用的边界、人工编写服务与 Agent 动态创建 Worker 的边界，全部折叠进三个原语之中。

Harness 不在后端之上，Harness 就是后端的一部分。而后端是什么，取决于什么接入了系统。当原语设计得当，品类折叠，边界消解，复杂性被根本性地简化。
