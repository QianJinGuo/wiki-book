# 让 Agent 越用越准、成本越来越低：AgentLoop 的 Agent 经验自进化闭环

> 阿里妹导读

## 问题：AI Agent 的天然不确定性

传统软件追求确定性。AI Agent 则天然具有不确定性——模型采样、上下文变化、任务规划、工具返回和长链路执行都可能影响结果。同一个问题连续运行多次，Agent 可能选择不同的工具、走不同的路径，甚至给出不同的答案。

当 Agent 进入运维、研发、数据分析和企业业务流程，企业关心两个结果：
1. **准确率**：任务成功率、首次完成率、多次执行稳定性
2. **成本**：每完成一个成功任务需要消耗多少 Token、时间、工具调用和人工介入

## 核心洞察：每一次运行都在产生待挖掘的经验

Agent 的每次运行不只产生最终答案，也会留下执行轨迹。成功轨迹中包含有效路径，失败轨迹中则记录了反复出现的错误和恢复线索。原始 Trace 并不天然等于经验——只有经过清洗和组装形成 Trajectory，再结合结果评估进行比较，才能提炼出可验证、可召回、可复用的行动经验。

**数据飞轮的价值不是积累更多日志，而是让每一次运行都为下一次运行提供经过验证的优化依据。**

## AgentLoop 经验自进化架构

AgentLoop 通过经验自进化，在模型之外构建了一层可持续更新的经验系统：

### 流程

1. **Trace 采集**：接收 Agent 的真实 Trace
2. **Trajectory 构建**：将高噪音执行数据清洗并组装为标准化 Trajectory
3. **经验挖掘**：从多个轨迹中自动挖掘有效路径、失败模式、工具约束、参数规则和恢复策略
4. **经验召回**：Agent 面对相似任务时，Recall Skill 和 CLI 根据当前目标、工具、进度和错误状态，召回少量适用经验并注入运行时上下文
5. **闭环**：新运行结果再次形成 Trace，进入下一轮挖掘

### "自进化"的含义

"自进化"不是修改模型权重，而是让 Agent 在模型通用能力之外，持续获得来自真实业务的行动经验。模型负责推理，工具负责执行，知识库提供事实，经验库则帮助 Agent 判断：当前情境下应该优先做什么、哪些路径容易失败、遇到问题后如何恢复。

## 经验注入降低不确定性

很多失败不是因为模型不具备能力，而是 Agent 在关键节点做出了错误选择。经验注入的价值在于：

- **任务开始时**：提供经过验证的入口选择和行动顺序
- **工具调用前**：补充参数约束、数据范围和前置条件
- **错误出现时**：优先提供有效的恢复方法
- **交付准备时**：提醒验证结果是否真正满足业务目标

企业衡量经验库的指标：
- 平均任务成功率
- 首次完成率
- 同类任务多次执行的成功率波动
- 失败模式的集中度
- 人工接管率和返工率

## Benchmark 结果

| Bench | 注入前 | 注入经验后 | Token 变化 |
|-------|--------|-----------|-----------|
| StarOps 指标查询 | 正确率 7.1% | 正确率 36.1% | -6.8% |
| OpenClaw / PawBench | 通过率 24.53% | 通过率 30.67% | -58.16% |
| PinchBench | 0.2928 | 0.3464，提升 18.3% | +2.9% |
| ClawProBench | 74.51% | 78.43%，提升 3.92pp | -11% |
| SWE-bench Verified | 67.2% | 74.4%，提升 7.2pp | 362.4M → 536M |

StarOps 实验中，平均工具调用次数下降 25.1%，有害事件下降 27.8%。

## 核心优势

### 1. 广泛接入真实 Agent 运行数据

AgentLoop 提供多种探针、OpenTelemetry、LoongSuite Pilot、eBPF，以及面向不同 Agent 和 AI Coding 工具的接入能力。

### 2. Trajectory 提炼高价值数据

原始 Trace 清洗后，高价值轨迹可降到原始 Trace 约 **4%-6%** 的数据量级。更小、更结构化的 Trajectory 降低存储和挖掘成本。

### 3. 面向真实轨迹的挖掘与召回算法

AgentLoop 在多个轨迹之间比较，识别反复出现的有效动作和高风险路径，生成结构化经验类型：
- 工具选择和调用顺序
- 参数规则与前置条件
- 容易导致失败的反模式
- 错误恢复和问题绕行策略
- 结果验证与完成判断规则

召回阶段结合文本相似度、任务、工具、进度、错误状态和经验适用范围进行排序和过滤。

### 4. Skill + CLI 快速接入

不需要重新训练模型或重建 Agent。客户端安装 Recall Skill，通过 CLI 配置经验库。Agent 可在任务开始、调用关键工具、遇到错误或准备交付时主动检索经验。

### 5. 多 Agent 共享经验

不同客户端和 Agent 使用同一个经验库。一个 Agent 验证过的有效路径可在权限范围内被其他 Agent 召回；已遇到的失败可成为其他团队的反模式。

## 与 Memory、RAG、微调、RL 的差异

| 能力 | 解决的问题 | 如何生效 |
|------|-----------|---------|
| Memory | 用户/会话过去发生了什么 | 检索用户事实和偏好，保持跨会话连续性 |
| RAG / 知识库 | 规则、文档和业务事实在哪里 | 从外部知识源检索相关内容 |
| Workflow / SOP | 标准流程应该怎样执行 | 人工定义的步骤和规则 |
| Prompt Engineering | 如何约束模型的通用行为 | 修改系统提示词或任务模板 |
| Skill | Agent 具备什么可复用能力 | 封装操作方法、脚本、工具和资源 |
| Fine-tuning / SFT | 如何改变模型的整体行为倾向 | 使用训练数据更新模型权重 |
| RL | 如何通过奖励优化模型策略 | 基于轨迹和奖励训练或更新策略 |
| **经验自进化** | **当前情境下，过去哪些行动有效** | **从 Trajectory 挖掘经验，运行时检索注入** |

## 开启方式

1. 创建经验库（控制台 → 上下文工程 → 经验库）
2. 创建访问凭证（API Key 或 AK/SK）
3. 安装 Recall Skill 并配置环境变量
4. 验证召回链路
