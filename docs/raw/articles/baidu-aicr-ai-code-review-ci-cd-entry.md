**核心问题**：AI 写代码越来越快（网盘主端 FE 团队每月~2000 次 CR，AI 生成代码占 55.87%），传统人工 Code Review 压力放大。本文分享百度网盘主端 FE 团队如何在 CI/CD 流水线中嵌入 AICR 强制检测链路。

**1. 为什么需要 AICR**

- 每月 ~2000 次 CR，AI 生成代码覆盖率 55.87%
- 大量 CR + 大规模修改 → 人工 CR 精度与专注度下降
- 目标：强制检测链路 + AI 多维度多角色审查 → 减轻 Reviewer 负担

阶段性数据（0401-0621）：累计检测 4388 次 CR，平均检测耗时 7.08 分钟（含排队），阻断~300+ 问题

**2. 检测链路**

iCode 提交 CR → iPipe 动态资源 → 开发机 → 检测脚本 → AI 审查 → 回写结果

步骤：下载代码 → AI 分析修改维度 → AI 代码缺陷审查 → AI 缺陷核实 → AI 缺陷复核 → AI 绝对安全检查 → AI 生成建议与评论

**3. CI/CD vs Pre-commit**

CI/CD 方案优点：规则热更新、深度接入工作流形成闭环、数据统一沉淀、统一标准 Agent 工具与动态资源环境。缺点：需额外下载代码。

Pre-commit 方案优点：不需要额外下载代码、适合个人自检。缺点：脚本规则难统一更新、无法深度接入 CR 审查流程、结果难以统一沉淀、运行环境受本地机器影响。

结论：CI/CD + Pre-commit 互补，CI/CD 负责准入与闭环，Pre-commit 负责前置自检。

**4. 多维度多角色审查架构**

- 审查前：逻辑序列化分析，识别修改维度、影响范围、调用链路
- 三路并行检测：不同方向独立审查（运行时风险/逻辑一致性/边界条件/调用链影响/代码质量）
- 聚合整理：去重、归并、聚合
- 核实 Agent：反向验证问题真伪
- 复核 Agent：对被核实删除的缺陷二次复核，防漏报

**5. 误报治理**

- 核实 Agent 反向验证：关注"是否真实存在"和"是否值得输出"
- 静态评论过滤：控制重复评论和低价值提醒
- 纠错本沉淀：高频误报、固定业务逻辑、历史沉淀规则
- 常态化纠错：用户反馈入口 + 每周两次复盘 + 模型优化 + 知识本补充

**6. 检测耗时控制**

甜点时间：平均 5 分钟（含下载代码到生成报告）
- 各阶段设置时间预算、超时策略和输出边界
- 拆分为 7 个明确阶段精细控时

**7. 模型选型与优化**

GLM5.0 → GPT5.5 迁移：

| 维度 | GLM5.0 | GPT5.5 |
|------|--------|--------|
| 缺陷检出率 | ~5% | ~21.8%（~25%） |
| 相对检出量 | 基线 | 3.2x |
| 规则策略 | 强规则、细约束 | 规则软化，给模型自主空间 |

关键发现：越高阶的模型，规则可以越轻。强规则反而限制高阶模型能力，增加推理负担。

基于 552 个 CR 对照实验验证：GPT5.5 检出 121 条（21.8%） vs GLM5 检出 38 条（6.9%）。

**8. 大模型预设立场问题**

- 审查 Agent 倾向"为发现而发现" → 过度推理、猜测缺陷
- 核实 Agent 倾向"为核实而删除" → 真实缺陷被误删
- 解决方案：多角色协作不依赖单模型判断，让 Agent 互相补位制衡

**9. Spec 模式探索**

尝试基于 Spec 文档增强 AICR 检测，经过半个月灰度验证发现：
- 文档与代码不同步 → 干扰判断
- 多仓知识冲突
- 后置检测收益偏低（Spec 阶段已做约束）
- 结论：应将检测规则前置赋能到 Spec SOP，而非后置检测

**10. 全流程 CR 建设展望**

| 阶段 | 状态 |
|------|------|
| Spec/需求阶段：前置需求特殊规则 | 规划中 |
| 开发中：Agent 集成审查 Skills | 100% |
| Pre-commit：轻量低噪音自检 | 20% |
| CI/CD 流水线 AICR：正式准入 | 95% |
| 静态代码审查：确定性工具兜底 | 完善中 |
| 合入后质量复盘：全库/分支整体 Review | 40% |
