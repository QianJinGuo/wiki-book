## Agent 操作浏览器的几个局限

理想化 Demo 里的网页通常很简单：打开页面，输入关键词，复制前几条结果。

真实网站不是这样。

第一类问题来自页面本身。

很多网页不是打开就把所有内容摆出来，而是先加载一个框架，再由浏览器继续请求数据、渲染内容。

Agent 如果只会“读取 URL 内容”，看到的往往不是用户眼前那一页。

就算能打开页面，还要处理下拉菜单更新、分页、详情页跳转、返回列表这些状态变化。

第二类问题来自环境。

很多网站会判断 “你是不是真人在用浏览器”，比如检查浏览器特征、插件、图形能力、网络指纹等自动化痕迹。

还有一些流程需要验证码、扫码登录、企业 SSO、短信确认或敏感操作确认。

完全无人值守当然好，但真实业务里经常需要人接一下手，再让 Agent 继续。

第三类问题是复用。

一次任务跑通，不代表流程被沉淀了。

如果每次都让 Agent 临场分析页面、临场写脚本、临场决定怎么翻页，浏览器自动化就会一直停留在 “这次可以，下次再说” 的状态。

## BrowserAct 的定位：给 Agent 一个真实浏览器执行层

BrowserAct 是一个专门面向 AI Agent 的浏览器自动化 CLI。

刚开始看它的介绍，很容易把它理解成“又一个让 Agent 点网页的工具”。

但我研究下来，觉得它真正有意思的地方不在这里。

过去很多浏览器自动化方案，默认把浏览器当成一个 “页面操作界面”：打开网页、找元素、点击、输入、截图、提取文本。

这个抽象在测试场景里很自然，因为测试脚本通常知道自己要点哪里，也知道页面应该长什么样。

但 Agent 做网页任务时，问题不只发生在 “点哪里”。

很多时候，它连稳定进入网页都不容易：

页面要等浏览器继续加载，登录状态要保住，网站会判断这是不是一个真实浏览器，多个任务还可能抢同一个页面状态...

BrowserAct 的第一个变化，是把 “浏览器环境” 本身变成了 Agent 工作流的一部分。

隐身浏览器、代理、浏览器身份、Cookie、Session 这些词单独看都像功能点，但放在一起看，其实是在回答同一个问题：

Agent 进入网页时，能不能像一个有稳定身份和上下文的浏览器，而不是一个临时脚本。

第二个变化，是它没有假装所有流程都能自动完成。

验证码、扫码、企业登录这些步骤，本来就经常需要人参与。

BrowserAct 比较务实的地方在于：人可以在卡住的地方接一下手，处理完之后，Agent 继续沿用原来的浏览器状态往下走。

这样“人工介入”不是任务失败，而是工作流里的一个中断点。

第三个变化，是它把“身份”和“任务”分开了。

浏览器更像账号身份的容器，Session 更像一次具体任务的工作区。

同一个账号下可以并行处理多个任务；不同账号之间，登录状态、Cookie、代理和浏览器配置也不会混在一起。

这个设计对 Agent 很重要，因为 Agent 往往不是只做一个页面动作，而是同时处理很多长期任务。

## BrowserAct 是怎么被 Agent 用起来的

看 BrowserAct 时，我觉得有必要把它的使用方式拆开看。

这产品其实是由最底层的一个  CLI  ，和上层的两个  Skills  构成。

CLI  负责真正执行浏览器动作：打开网页、点击输入、提取页面内容、抓取网络数据 ...

Agent 直接面对 CLI 并不方便。

它需要知道什么时候该打开浏览器，什么时候该等页面稳定，什么时候该提取内容，遇到验证或登录又该怎么停下来让人接手。

所以 BrowserAct 还提供了配套的 Skills。

browser-act  这个 Skill 做的就是这件事：把 CLI 的能力整理成 Agent 能遵循的操作流程。

https://github.com/browser-act/skills/tree/main/browser-act

在一次具体任务里，  browser-act  更像是执行向导。

它让 Agent 能按正确步骤调用 BrowserAct CLI，进入真实浏览器，把当前这次网页任务跑通。

但跑通一次还不够。

如果这个流程以后还会反复出现，就需要再往前走一步：把这次探索出来的网站能力沉淀下来。

BrowserAct 还提供了一个  browser-act-skill-forge  。

https://github.com/browser-act/skills/tree/main/browser-act-skill-forge

它会利用前面跑通的结果，把一个网站里的稳定流程整理成新的 Skill。

BrowserAct 官方的描述是：

你可以把任何网站转化为可复用的技能。

下面，我们来实际体验一下它们的表现究竟如何。

## 实战：用 BrowserAct 研究模型的真实口碑

研究 BrowserAct 时，我刚好在关注  GLM-5.2  。

一个新模型发布后，官方介绍和媒体报道通常会很快出现，但我更想用户对他们的真实反馈：

我一般会习惯去看 Hacker News，是因为它的讨论有一种特殊的“含金量”。

它当然不代表全网。

HN 的用户偏欧美工程师、创业者、研究者和工具型用户，和普通消费者社区不是一回事。

https://news.ycombinator.com

但也正因为这样，它很适合观察 AI 模型发布后的真实技术口碑。

这里的人不太会只停留在 “发布会讲得怎么样”，他们更关心能不能写代码、能不能接进 Agent 、本地部署成本高不高、BenchMark 有没有水分、和其他模型比到底差在哪里。

安装很简单，大家直接把下面的话术发给你的 Agent，然后根据引导完成一些简单的配置就可以了：

Install browser-act.  
Skill URL: https://github.com/browser-act/skills/tree/main/browser-act  
After installation, verify that the CLI is available.

下面我们直接在 Codex 中测试下结果：

帮我用 BrowserAct 做一份 GLM-5.2 模型舆情报告。  
请打开 hn.algolia.com ，探索 GLM 5.2 相关讨论，我要知道：  
1. 大家整体是看好、质疑，还是分化？  
2. 正向评价主要是什么？  
3. 负向评价主要是什么？  
4. 它经常被拿来和哪些模型比，比如 Claude、OpenAI、Gemini、DeepSeek、Qwen、Kimi？  
5. 对比结论是什么？  
请保存原始数据，并生成一份 Markdown 报告。

任务下发后，Skill 第一步不是让 Agent 复制搜索结果页上的几行标题，而是先打开 HN，观察页面怎么加载数据。

Agent 通过 BrowserAct 进入真实浏览器后，沿着页面自己的加载方式取数据：

先拿核心查询词  GLM-5.2  的总量，再分别看 stories 和 comments，接着扩展到主题词和对比模型。

跑了大概十几分钟，任务完成了。

这次执行最后落了四类文件：

- raw-data.json
  ：保留 stories/comments 原始样本

- raw-data.csv
  ：方便人工复核的样本表

- thematic-counts.json
  ：主题和模型共现矩阵

- report.md
  ：最终分析报告

这样做的好处是，报告里的判断不是 Agent 随口总结，而是可以回到原始样本和统计矩阵里检查。

第一层结果显示，GLM-5.2 在 HN 不是一个孤立话题。

它总命中约 436 条，其中 stories 约 63 条，comments 约 373 条。

整体覆盖了发布、榜单、本地运行、幻觉率、Opus 对比和 agent 工作流。

也就是说，HN 对 GLM-5.2 的关注不是单点式的，而是多条技术线同时展开。

第二层是模型对比。拿 GLM-5.2 和 Opus/Claude 直接比较的高达 31 条。

说明用户真正关心的是：它能不能用更低成本接近 Claude/Opus？

开放模型足够好之后，哪些任务还值得继续为闭源模型付溢价？

第三层是情绪判断，偏中性正向和帖子占比达到了 80%，主要来自开放权重、编码能力、Agent 潜力、成本压力和本地部署想象力。

最后，报告把结论压成一句话：

HN 把 GLM-5.2 看成开放模型阵营的一次显著跃迁，而不是闭源 SOTA 的终结者。

这个判断比简单说 “正向” 或 “负向” 更有信息量：

技术社区认可它的进步，也愿意试用，但并不会因为开放权重和榜单成绩就自动买单。

## BrowserAct 在这一步做了什么

随后我仔细研究了下它的执行过程，我发现 BrowserAct 还真是有点东西的。

第一，它让 Agent 进入的是一个真实浏览器现场，而不是一份网页文本。

HN Algolia 这种页面，真正有价值的数据不是一开始就摆在 HTML 里，而是页面加载后再请求回来。

普通网页读取很容易停在 “看见什么复制什么” 的阶段。

而 BrowserAct CLI 运行在真实浏览器里，Agent 可以等页面稳定、观察状态变化，再决定下一步怎么取数据。

第二，它让 Agent 不必在浏览器外面硬猜接口。

HN Algolia 页面自己会请求 Algolia 的搜索接口。

BrowserAct 让 Agent 在浏览器上下文里观察和复用这条路径：

核心查询怎么发，stories 和 comments 怎么分，分页和排序怎么处理，都可以从真实页面行为里确认。

这样拿到的数据更接近用户实际看到的页面，而不是靠外部脚本拍脑袋拼参数。

第三，它保留了任务过程里的证据链。

在任务完成后，它会把标题、分数、评论数、发布时间、链接、正文片段、主题命中、模型共现都保存下来。

最后的  raw-data.json  、  raw-data.csv  和  thematic-counts.json  ，让报告里的判断可以被人工复核。

第四，它把一次网页操作变成了可沉淀的路径。

普通浏览器自动化到这里可能就结束了：抓到数据，生成报告。

但 BrowserAct 的价值在于，Agent 跑通之后，已经知道了 HN Algolia 的查询路径、字段结构、主题扩展方式、模型对比方式和输出文件结构。

这整套流程可以继续交给  browser-act-skill-forge  ，封装成一个新的业务 Skill。

## browser-act-skill-forge：把网站变成可复用能力

我们直接跟 Agent 说：

/browser-act-skill-forge 帮我把刚才这套流程封装为一个 Skill ，支持输入任意的模型名称，产出详细的调研报告。

Forge 会把一次探索拆成几件事：

- 先确认目标网站和业务意图：这里是 HN Algolia 上的模型舆情调研

- 再探索稳定路径：优先看页面背后的接口，必要时再退回 DOM 操作

- 然后抽出可复用参数：模型名、对比模型、主题维度、样本量、翻页范围

- 最后生成
  SKILL.md + scripts
  ，让后续 Agent 不需要重新理解这个网站

最后得到的当然是一个  hn-model-sentiment-report  ，但重点不在名字上。

更重要的是，Forge 把 “我知道怎么操作这个网站” 变成了 “任何 Agent 都能复用这个网站能力”。

Forge 生成的 Skill 更像是一个小型网站能力模块。

它保留的不只是某次结果，而是这类任务的做法：怎么查、怎么分页、怎么分 stories/comments、怎么做主题矩阵、怎么做模型共现、怎么保存证据、怎么写报告边界。

这里容易误解的一点是：复用 Skill，不只是复用几段抓取代码。

真正应该复用的是方法。

在 HN 模型舆情这个场景里，方法包括几层：

- 数据范围固定：只使用
  hn.algolia.com

- 数据类型固定：stories 和 comments 分开处理

- 证据留存固定：保留原始样本、CSV 表、主题矩阵和报告

- 分析结构固定：总体热度、主题分布、模型对比、情绪判断、方法局限

- 边界表达固定：全量命中信号和样本情绪判断分开写

很快我们得到了包装好的 Skill，下面我们直接来试一下。

## 和传统方案相比，BrowserAct 改变了什么

静态抓取适合结构稳定、无需 JS、无需登录、无需交互的页面。

Playwright/Selenium 适合开发者写固定脚本，但暴露给 Agent 使用时，经常会遇到自动化检测、人机验证、状态隔离和流程复用问题。

RPA 更偏 UI 操作编排，适合固定流程，不太适合让大模型动态观察、推理和修正路径。

BrowserAct 更适合放在 Agent 工作流里。

它让 Agent 进入真实浏览器，用 Session 管理并发任务，用独立浏览器身份隔离多账号，在需要人工确认时保留接力点，并且能把跑通的流程封装成 Skill。

这不是说 BrowserAct 能替代所有方案。

更准确地说，它补上了 Agent 操作真实网页时最容易缺的那一段：稳定执行、状态管理、结果复核和流程复用。

browser-act  负责让 Agent 正确调用这套执行能力。

如果看完整流程，它更像一个小型工作流样板：

先让 Agent 进入真实网页完成研究，再把跑通的方法保存下来，下一次换模型继续用。

这也是我觉得 BrowserAct 值得关注的原因。

它是在真的回答一个实际的问题：Agent 在浏览器里做过的一件事，能不能变成以后反复可用的能力。
