> 上一篇 10 · 云端 Codex Cloud 把「让 Codex 在远端跑、回头收 PR」这条线讲透了。这一篇拉回本地，聊一份每个项目都该有、但九成新手写废了的文件——AGENTS.md，Codex 每次开工前都要先读一遍的项目说明书。

说个我刚上手 Codex 时干的蠢事。那是去年我把一个 Node 项目交给 Codex，开宗明义在根目录写了份 AGENTS.md，第一行赫然写着「本项目用 pnpm，禁用 npm」。结果它转头给我来了一句 npm install。我当场以为它没读到，气得把那条规矩复制了三遍、加粗、还打了感叹号。还是不听。折腾半天才发现——它读到了，只是那条规矩被我埋在了第 140 行。前面我塞了公司简介、产品路线图、技术选型的来龙去脉，洋洋洒洒一百多行。等 Codex 翻到「禁用 npm」那句，注意力早被前面那堆废话稀释没了。问题从来不是它不听话，是我把唯一有用的那条焊进了一坨没用的里。

AGENTS.md 之于 Codex，就是 CLAUDE.md 之于 Claude Code，同一个概念换了个文件名。但 Codex 这边的发现机制、覆写规则、字节上限自成一套，有几处容易踩的坑。

### 01 先搞懂：AGENTS.md 是 Codex 开工前的「交接清单」

AGENTS.md 是你写给 Codex 的一份持久指令，它每次启动、动手干活之前都先读一遍，当成这个项目的背景装进脑子。没有 AGENTS.md，你就得每次重新解释一遍。

类比：交接班的工作清单。工厂三班倒，上一班下班前会在墙上那块板写清楚——这台机器有点毛病别硬开、这批料要先质检、紧急联系人是谁。下一班接手，照着板上干就行，AGENTS.md 就是 Codex 的那块交接板——而且是它每次接班前都会先扫一眼的那块。

什么时候该往里加内容？几个信号：
- Codex 第二次犯同一个错
- 你这轮又敲了一遍上轮敲过的那句更正
- 代码审查时发现，它本该早就知道这个代码库的某个约定
- 新队友需要同样的背景才能快速上手

最爱的用法：把它当反馈回路。Codex 对你代码库做了错误假设，别光在对话里纠正（那是一次性的，下轮它又忘），直接让它把这条修正写进 AGENTS.md。

### 02 发现链条：Codex 是怎么找到这些文件的

按官方文档，Codex 构建指令链分两步走：

**第一步，全局层（Global scope）**。在你的 Codex 主目录（默认 ~/.codex）里，Codex 先看有没有 AGENTS.override.md，有就用它；没有才读 AGENTS.md。这一层只取第一个非空文件，不会两个都读。

**第二步，项目层（Project scope）**。从项目根目录（通常是 Git 根）开始，一路往下走到你当前所在的目录。沿途每一个目录里，Codex 按这个顺序挑：先看 AGENTS.override.md，再看 AGENTS.md，最后看 project_doc_fallback_filenames 里你自定义的备选文件名。每个目录最多只收一个文件。

**第三步，合并（Merge order）**。Codex 把找到的这些文件从根到叶依次拼接，中间用空行隔开。越靠近你当前目录的文件，因为排在拼接结果的越后面，优先级越高、能盖过前面的。

两个要点：
1. 是「拼接」不是「覆盖」。全局那份和项目那份同时生效，不存在「写了项目级，全局级就失效」。
2. 更近的赢。比如全局 AGENTS.md 说「字符串用单引号」、项目根 AGENTS.md 说「用双引号」——项目根离当前目录更近、排得更靠后，所以双引号胜出。

### 03 AGENTS.override.md：Claude Code 没有的「临时盖章」

AGENTS.override.md 这玩意儿 Claude Code 那边没有对应物，是 Codex 独有的设计。

类比：在文件上盖一张「以此为准」的便签。原合同还在抽屉里，你没撕；只是临时贴了张便签条「本月按这个新条款执行」。事儿办完，便签一揭，原合同自动生效。

官方给的典型场景：
- **全局临时覆写**：在 ~/.codex/AGENTS.override.md 写一套临时全局指导，原 AGENTS.md 原封不动。活干完删掉 override，共享的那份就恢复了。
- **子目录特殊规则**：某个子目录的团队需要一套跟外面完全不同的规矩。

关键理解：override 是「这一层用我，别用旁边的 AGENTS.md」，不是「全链路只听我一个」。整条链的拼接、就近优先规则照旧。

### 04 该写什么 vs 不该写什么

一句口诀：**写「Codex 在每一轮里都该保持的事实」**。

该写的五类：
| 类别 | 具体写什么 |
|------|-----------|
| 项目概述 | 一句话说清这是个啥 |
| 技术栈 | 语言、框架、数据库、关键工具 |
| 常用命令 | 测试、构建、检查怎么跑 |
| 代码约定 | 风格、命名、必须遵守的写法 |
| 明确的「不要做」 | 雷区、禁改文件、必须先确认的操作 |

不该写的：
- ❌ 长篇大论的背景：公司介绍、产品愿景——Codex 写代码用不上，纯占上下文
- ❌ 过时信息：换了包管理器却没更新 AGENTS.md
- ❌ 看代码就知道的东西：别去复述目录结构、别把 ESLint/Prettier 定义好的风格再抄一遍

**大小红线**：Codex 按字节算，合并后默认 32 KiB（project_doc_max_bytes）封顶，超了就截断。撞上限了，要么调高上限，要么把指令拆到多级子目录分散放。

### 05 两个旋钮：改文件名、调字节上限

旋钮一：`project_doc_fallback_filenames`——让 Codex 认你已有的文件名（如 TEAM_GUIDE.md）。不在这个列表里的文件名，Codex 一律忽略。

旋钮二：`project_doc_max_bytes`——抬高合并上限（默认 32 KiB）。但撞上限优先拆/删而非硬抬。

> ⚠️ 改完 config.toml 记得重启 Codex。

### 06 动手验证流程

三步验证法：
1. 建个玩具项目 `mkdir agents-md-demo && cd agents-md-demo && git init`
2. 在根目录写一份精简的 AGENTS.md
3. 跑 `codex --ask-for-approval never "Summarize the current instructions."`——Codex 会复述它读到的指令

子目录覆写验证：在子目录放 AGENTS.override.md，用 `codex --cd services/payments` 启动，Codex 会优先用 override 的规则。

### 小结

| 维度 | 关键结论 |
|------|---------|
| 是什么 | 每轮开工必读的交接清单 |
| 发现链 | 全局层 → 项目层逐级 → 从根到叶拼接 |
| 谁说了算 | 拼接不覆盖，越近优先级越高 |
| override | Codex 独有的「临时盖章」 |
| 写什么 | 概述/技术栈/命令/约定/禁区 |
| 大小红线 | 按字节算，合并默认 32 KiB |
| 两个旋钮 | fallback_filenames 改文件名、max_bytes 调上限 |
