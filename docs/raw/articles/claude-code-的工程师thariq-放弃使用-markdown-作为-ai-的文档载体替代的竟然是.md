# Claude code 的工程师@Thariq 放弃使用 markdown 作为 AI 的文档载体，替代的竟然是 …

我非常认同他这篇分享，恰巧我最近也在实践用 HTML 来展示内容，体感很接近，所以直接把他的内容翻译成中文表述，分享给大家，下面是文章内容，欢迎一起讨论。

Markdown 已经成了智能代理与我们沟通时最常用的文件格式。它简单、便携，具备一定的富文本能力，也方便你直接编辑。Claude 甚至已经很擅长在 Markdown 文件里用 ASCII 画图了。

但随着智能代理越来越强，我越来越觉得 Markdown 反而成了一种束缚。超过一百行的 Markdown 文件，我读起来就开始吃力。我想要更丰富的可视化、更清晰的颜色和图表，也希望这些内容能更方便地分享出去。

而且，我也越来越少亲手编辑这些文件。它们更多时候是规格说明、参考资料、头脑风暴结果等等。真需要修改时，我通常也是让 Claude 来改，这就削弱了 Markdown 最大的优势之一。

所以，我开始更偏向让 Claude 输出 HTML，而不是 Markdown。我也越来越常看到 Claude Code 团队里的其他人这么做。下面就是原因。

如果你想先看一些例子，可以去这里：https://thariqs.github.io/html-effectiveness/ ，不过记得回来继续看为什么。

# 为什么是 HTML？

## 信息密度

和 Markdown 相比，HTML 能承载的信息丰富得多。它当然可以做标题、段落、格式这些基础文档结构，但它还能表达很多 Markdown 很难自然呈现的东西，比如：

  * • 用表格展示结构化数据
  * • 用 CSS 表达设计信息
  * • 用 SVG 绘制插图
  * • 用 script 标签放代码片段
  * • 用 HTML 元素、JavaScript 和 CSS 做交互
  * • 用 SVG 和 HTML 展示工作流
  * • 用绝对定位和 canvas 表达空间信息
  * • 用 image 标签嵌入图片

我甚至会说，几乎所有 Claude 能读懂的信息，都能相当高效地用 HTML 表达出来。也正因为如此，HTML 成了一种非常高效的媒介，模型可以用它把复杂信息讲透，而你也更容易阅读和审阅。

如果做不到这一点，模型在 Markdown 里往往就会采取一些低效的办法，比如画 ASCII 图，或者我最喜欢的，用 Unicode 字符来“估算”颜色。下面这张 Claude Code 截图就是例子。

Claude Code 试图在 Markdown 里展示颜色

## 视觉清晰度与阅读体验

随着 Claude 能处理的任务越来越复杂，它写出来的规格说明和计划也越来越长。实际使用中我发现，超过 100 行的 Markdown 文件，我基本不会认真读完，更别说让组织里的其他人也去读了。

但 HTML 文档读起来轻松得多。Claude 可以用视觉方式组织结构，比如标签页、插图、链接等，让文档更适合浏览和理解。它甚至可以做成响应式，在手机、平板、电脑上根据不同屏幕呈现不同阅读方式。

## 更容易分享

Markdown 文件其实不太好分享，因为大多数浏览器并不能原生很好地渲染它们。你常常不得不把它们作为附件塞进邮件或消息里。

HTML 就简单得多。只要把文件上传到某个地方，比如 S3，你就可以直接分享链接。同事可以在任何地方打开，也很容易反复引用。

如果你的规格说明、报告或 PR 说明是 HTML，别人真正读它的概率会高得多。

## 双向交互

HTML 还可以让你和文档互动。比如你可以让它加上滑块或旋钮，用来调整设计；也可以让它暴露算法里的不同参数，让你边调边看效果。你甚至可以让它把你在界面里做出的修改复制成一段 prompt，再粘回 Claude Code。想看这种双向交互的例子，可以读我关于 playgrounds 的帖子：https://x.com/trq212/status/2017024445244924382

**数据摄取**

为什么用 Claude Code 来生成 HTML 文件，而不是 ClaudeAI 或 Claude Design？一个很重要的原因是，Claude Code 能吃进大量上下文。

比如写这篇文章时，我让 Claude Code 遍历我的代码文件夹，找出我生成过的所有 HTML 文件，给它们分组、归类，然后再生成一个 HTML 文件，用图表展示每种类型。你在这篇文章里看到的这些图，就是这一步的直接产物。

除了文件系统，Claude Code 还可以通过 MCP 获取更多上下文，比如 Slack、Linear 等，也可以读取你的浏览器上下文、git 历史等等。

## 它更有乐趣

和 Claude 一起制作 HTML 文档，本身就更好玩。它让我更有参与感，也更愿意投入到创作过程里。光这一点，就已经足够了。

## 如何开始

我有点担心，有人读完这篇文章之后会把它做成一个 `/html` skill 之类的东西。那也许有点价值，但我更想强调的是，你其实不需要做太多准备。你只要直接说“生成一个 HTML 文件”或者“做一个 HTML artifact”就可以了。

关键在于，你要知道这个 artifact 需要做什么，以及你接下来准备怎么使用它。以后你当然可以慢慢沉淀出一个 skill，但一开始，我建议你直接从零开始写 prompt，先熟悉 HTML 在不同场景里的用法。

# 使用场景

为了更具体一点，我已经为很多不同场景生成过 HTML 文件。你可以在这里查看完整示例：https://thariqs.github.io/html-effectiveness/ ，下面是一个概览。

## 规格说明、规划与探索

HTML 是一块很适合 Claude 深入思考问题的富画布。现在当我开始处理一个问题时，我期待得到的已经不是一份简单的 Markdown 计划，而是一组互相关联的 HTML 文件。

比如，我可能会先让 Claude Code 做头脑风暴，探索几种不同方案；接着让它对其中一个方向继续展开，可能做一些 mockup 或代码片段；最后，当我感觉方向对了，再让它写一份实现计划。等我认可这份计划后，我会新开一个会话，把这些文件全部交给它，让它开始实现。

在验证阶段，我也会让验证代理读入这些文件，这样它能掌握更完整的上下文，知道真正需要检查什么。

**示例 Prompt：**

  * • 我不确定 onboarding 页面该往哪个方向做。生成 6 个明显不同的方案，在布局、语气和信息密度上拉开差异，把它们放在一个 HTML 文件的网格里，方便我并排比较。每个方案都标注它做出的取舍。
  * • 创建一份完整的 HTML 实现计划，包含一些 mockup，展示数据流，并加入我可能需要审阅的重要代码片段。让它足够清晰、易读、好消化。

**适用场景：**

  * • 探索某个功能在代码层面的不同实现方式
  * • 探索多个视觉设计方向

## 代码审查与理解

代码放在 Markdown 文件里往往不好读。但在 HTML 里，我们可以渲染 diff、标注、流程图、模块结构等等。你可以用它来理解代理写出的代码，也可以用它做代码审查，或者向 PR 审阅者解释你的改动。我发现，这种方式经常比 GitHub 默认的 diff 视图更好用。现在我每次提 PR，都会附上一份 HTML 代码讲解文档。

**示例 Prompt：**

帮我审查这个 PR，创建一个 HTML artifact 来说明它。我对 streaming/backpressure 逻辑不是很熟，所以请重点讲这部分。把实际 diff 渲染出来，在旁边加行内标注，并按严重程度给发现的问题做颜色区分。只要有助于讲清楚概念的内容，都可以加进去。

**适用场景：**

  * • 创建 PR
  * • 审查 PR
  * • 理解代码里的某个主题

## 设计与原型

Claude Design 之所以基于 HTML，是因为 HTML 在表达设计方面非常强，即便你的最终交付形态并不是 HTML。Claude 可以先用 HTML 草拟设计，再把它改写成你需要的语言，比如 React、Swift 等。

你也可以用它来做交互原型，比如动画、动作效果等等。不妨让 Claude 加一些滑块、旋钮之类的控件，帮助你精确调出想要的效果。

**示例 Prompt：**

我想做一个新的结账按钮原型。点击后，它会播放一个动画，然后快速变成紫色。请创建一个 HTML 文件，里面提供几个滑块和选项，让我尝试不同动画参数，并加一个复制按钮，把我调好的参数复制出来。

**适用场景：**

  * • 创建设计系统 artifact
  * • 调整组件
  * • 可视化组件库
  * • 原型化有趣的动效

## 报告、研究与学习

Claude Code 非常擅长把来自多个数据源的信息综合起来，再转成易读的报告。你可以让 Claude 搜索 Slack、代码库、git 历史、互联网等等，然后为你自己、领导或团队生成非常易读的报告。

它可以是一份很长的 HTML 文档，也可以是一个交互式解释器，甚至可以是一组幻灯片。你还可以要求 Claude 使用 SVG 画图，帮助可视化说明。比如我写 prompt caching 相关帖子时，就让 Claude 先读 git 历史，然后生成一份深入的 HTML 研究文件，帮我理解我们对 prompt caching 做过的所有改动。

**示例 Prompt：** 我不理解我们的 rate limiter 到底是怎么工作的。请阅读相关代码，生成一个单页 HTML 讲解文档：包括 token-bucket 流程图、3 到 4 段关键代码片段及其注释，并在底部加一个“容易踩坑的地方”板块。优化目标是让人只读一遍就能理解。

**适用场景：**

  * • 总结某个功能如何工作
  * • 向我解释一个概念
  * • 写给老板的周报
  * • 写给管理层的事故报告
  * • SVG 插图、流程图、技术图解等

# 自定义编辑界面

有时候，光靠文本框很难说清楚你想要什么。遇到这种情况，我会让 Claude 为眼前这个具体任务做一个一次性的编辑器。它不是产品，也不是可复用工具，而是一个单独的 HTML 文件，专门为这一份数据服务。

诀窍是，最后一定要有导出能力，比如“复制为 JSON”或“复制为 prompt”按钮，把你在 UI 里做过的操作重新变成一段可以粘回 Claude Code 的内容。

**示例 Prompt：**

  * • 我需要重新排列这 30 个 Linear 工单的优先级。帮我做一个 HTML 文件，把每个工单做成可拖拽卡片，分成 Now / Next / Later / Cut 四列。先按你的判断预排一下。加一个“复制为 markdown”按钮，导出最终排序，并为每个分组附上一句理由。
  * • 这是我们的 feature flag 配置。帮我做一个基于表单的编辑器，按领域分组展示这些开关，显示它们之间的依赖关系。如果我启用了某个前置条件没打开的开关，就提醒我。加一个“复制 diff”按钮，只输出发生变化的 key。
  * • 我正在调这个 system prompt。做一个左右分栏编辑器：左边是可编辑 prompt，并高亮变量占位符；右边放三个示例输入，实时渲染填充后的模板。加上字符/token 计数器和复制按钮。

**适用场景：**

  * • 重新排序、分诊或分桶，比如工单、测试用例、反馈
  * • 编辑结构化配置，比如 feature flag、环境变量、有约束的 JSON/YAML
  * • 调整 prompt、模板或文案，并实时预览
  * • 整理数据集，批准/拒绝行、给样本打标签、导出选择结果
  * • 给文档、转录稿或 diff 做标注，并导出标注结果
  * • 选择那些很难用文字表达的值，比如颜色、缓动曲线、裁剪区域、cron 时间表、正则表达式

## 常见问题

我已经跟很多人聊过自己转向 HTML 这件事，也反复听到一些问题。

**这不是更浪费 token 吗？** Markdown 通常确实更省 token，但我发现，HTML 带来的表达力提升，以及我更有可能认真阅读它，最终会让我得到更好的输出。再加上 Opus 4.7 有 1MM 上下文窗口，HTML 多出来的 token 在上下文窗口里其实不太明显。

**那你现在什么时候还用 Markdown？** 说实话，我几乎已经不再用 Markdown 了。当然，我可能已经站在 HTML 最大化主义那一边了。

**HTML 文件怎么看？** 我一般直接在本地浏览器里打开，也可以让 Claude 帮你打开。如果你想分享链接，就上传到 S3 之类的地方。

**生成 HTML 会不会比 Markdown 更慢？** 会，确实更慢。HTML 可能要多花 2 到 4 倍时间，但在我看来，结果值得。

**版本控制怎么办？** 这确实是 HTML 最大的缺点之一。和 Markdown 相比，HTML 的 diff 很吵，也更难审阅。

**怎么让 Claude 符合我的审美，不要做得很丑？** frontend design plugin 能帮助 Claude 生成更好的 HTML 文件。但如果你想贴合自己公司的风格，可以让 Claude 读取你的代码库，先生成一个单独的设计系统 HTML 文件。之后做其他 HTML 文件时，就把这个设计系统文件作为参考。

## 保持参与感

说到底，我之所以使用 HTML，真正的原因是它让我在和 Claude 协作时更有参与感。之前我已经开始担心，自己因为不再深入阅读计划，最终只能把选择权完全交给 Claude。

但现在我很高兴地发现，使用 HTML 之后，我比以往任何时候都更能跟上 Claude 的思路。也希望你有同样的感觉。
