# Agent记忆系统的主矛盾：从存储问题到上下文调度

副题：用"主次矛盾"重看 Agent Memory 的方法谱系

MoonTidef 数海志 2026年6月13日 北京

## 摘要

Agent 记忆系统的核心，不是"要不要向量库"，也不是"该不该上知识图谱"。它的主矛盾是：**持续生长的历史状态，和有限、脆弱的临场上下文之间的矛盾**。

记忆系统不是存储系统的别名，它更接近一套**上下文调度机制**：记忆不是把历史全收起来，而是按任务、模型和预算挑一份可用材料。

## 一、主矛盾：历史增长 vs 临场上下文

设 Agent 的历史为 H，当前任务为 q，模型为 M，上下文预算为 B。记忆系统要做的，不是把 H 原样塞进 M，而是从 H 中构造一份可用上下文 C。

R 不是简单检索，至少包含抽取、管理、存储、检索、注入几步。

目标也不是"召回越多越好"，系统要让任务收益尽量高，同时守住三个约束：别超窗口，别带太多噪声，别歪曲历史。

## 二、形式化：记忆系统到底优化什么

记忆系统写成一个四元组 **M_sys = (E, G, S, I)**：
- **E** (extraction)：从轨迹中抽出候选记忆
- **G** (governance)：更新、合并、遗忘、冲突处理
- **S** (storage)：文本、向量、图、树、参数或混合形态
- **I** (injection)：把检索结果组织成模型可用的上下文

成熟的 Agent 记忆系统优化的链：**History → E → G → S → Retrieve → I → Context → Action**

工程上常犯的错是把 S 当成 M_sys。向量库只是 S 的一种。

## 三、六个关口：每一环都有主次

| 关口 | 主矛盾 | 典型取舍 |
|------|--------|----------|
| **来源** | 未来效用 vs 输入洪水 | 什么该进记忆，什么只留日志 |
| **抽取** | 保真 vs 可操作 | 原文/摘要/事实/事件/规则怎样取舍 |
| **管理** | 更新 vs 证据保留 | 新事实覆盖旧事实还是版本并存 |
| **形态** | 可解释 vs 可扩展 | 文本/向量/图/树/参数各担哪段 |
| **检索** | 相似 vs 有用 | 语义近不等于对任务有用 |
| **注入** | 足量 vs 干扰 | 给少了缺证据，给多了乱推理 |

## 四、三类功能：事实、经验、工作

- **事实记忆**：世界和用户现在是什么样。主矛盾：稳定与更新
- **经验记忆**：过去怎么做成/做坏。主矛盾：泛化与误导
- **工作记忆**：眼下走到哪一步。主矛盾：短暂与连贯

三者不能混放。事实要审计，经验要抽象，工作要快进快出。

## 五、方法地图：论文各抓一头

| 方法族 | 关口 | 代表方法 |
|--------|------|----------|
| 事实与偏好 | 来源、抽取、更新 | MemoryBank, MemoChat, Mem0 |
| 管理与调度 | 管理、注入 | MemGPT, MemoryOS, MemOS |
| 关系与演化 | 形态、检索、更新 | Zep, Graph Memory, A-MEM |
| 经验与反思 | 抽取、泛化 | ExpeL, ReasoningBank |
| 压缩与注入 | 检索、注入 | ACON, MemAgent, Memory-R1 |

没有通吃方法，只有场景里的主矛盾。

## 六、设计原则：以场景作总裁判

设计记忆系统不宜从组件清单开始。应先问四个问题：
1. 任务失败主要败在哪里？
2. 哪类历史有未来效用？
3. 记忆的证据链要不要保留？
4. 模型怎样感知 prompt？

关键不在"选最先进方法"，而在"找准瓶颈"。

## 结语

Agent 记忆系统的主要矛盾，不是存储容量不够。它是历史无限生长，临场上下文有限；世界持续变化，模型一次只能择要推理。

分类只能给坐标。公式只能给约束。真正的裁判，是任务结果。
