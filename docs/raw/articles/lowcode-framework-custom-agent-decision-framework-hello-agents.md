# 低代码 Agent、框架 Agent、自研 Agent，分别适合谁
## 核心框架
三条路对应三种不同诉求，不是高低级关系，更像三种不同的工程站位：
| 类型 | 解决什么问题 | 核心关键词 |
|------|------------|-----------|
| 低代码 Agent | 先把东西跑起来 | 速度、协作、可视化 |
| 框架 Agent | 别重造轮子但保留工程控制力 | 平衡、可复用骨架 |
| 自研 Agent | 这已经是长期系统能力 | 控制权、核心资产 |
## 一、为什么很多人一开始就选错路
选错的原因不是工具选型错误，而是没先想清楚：
• 这个任务到底是不是要长期做
• 它只是个原型，还是核心业务能力
• 谁来维护
• 它需要多深的定制
• 它要不要接公司内部系统
• 它能不能接受平台黑盒
• 出问题以后，谁来排查、谁来兜底
先问自己：现在最缺的是速度、控制力，还是长期可演化性。
## 二、低代码 Agent：适合"先把价值跑出来"的人
**最适合三类人**：产品经理/运营/业务专家（不关心架构，只关心今天能不能跑）、快速做原型的开发者、需和业务方共创的人（流程可视化让非技术人员也能参与）。
**真正强的四个地方**：
1. 原型速度快——上午想清楚，下午能演示
2. 可视化好——数据流、失败点、Prompt 稳定性都比纯代码直观
3. 平台集成快——模型、知识库、插件、发布渠道和基础运维平台已做好
4. 协作门槛低——产品、运营、业务同学都能一起改
**不适合的场景**：状态管理越来越复杂、人工确认点越来越多、工具调用和权限控制越来越细、需要深度接入内部系统、需要稳定测试/回归验证/版本治理。
## 三、框架 Agent：适合"已经确认方向，要开始认真做工程"的人
**最适合两类团队**：已有工程能力的开发团队（希望任务逻辑进代码库、工具定义版本化、状态管理可测试、关键流程可 review），已验证需求准备从 demo 走向产品的团队。
**框架真正解决的核心共性问题**：
- Agent loop 怎么组织
- 多 Agent 怎么通信
- 工具怎么注册和调用
- 状态怎么在节点间流转
- 终止条件怎么定义
- 回调、日志、观测点怎么挂
**四个代价别忽略**：学习抽象（学的是框架怎么理解 Agent）、版本漂移（领域变化快，框架接口可能很快变）、排障难度（同时要理解任务逻辑和框架抽象）、迁移成本（绑在框架独有写法上后切换不轻松）。
## 四、自研 Agent：适合"这已经是你的长期系统能力"的人
**最适合三类场景**：把 Agent 当成核心产品能力的团队、对权限/审计/稳定性/成本/内部系统接入要求很高的团队（金融、医疗、政企）、已走过一轮低代码或框架实践知道自己真正需要什么的团队。
**自研换来的是控制权**：
- 任务契约长什么样
- 状态如何外置
- 哪些工具暴露给模型
- 哪些节点必须人工确认
- 日志和审计怎么记录
- 模型供应商如何切换
- 回退机制、缓存策略、评估体系怎么接
**真正成本**：框架设计、API 抽象、测试策略、观测与告警、版本兼容、成本治理、团队内部文档和 onboarding。低代码和框架帮你省掉的"基础设施型痛苦"，到了自研这里都会重新回来。
## 五、三条路是阶段递进，不是对立
```
0 -> 1：先低代码，验证价值
1 -> 10：上框架，建设可维护工程骨架
10 -> 100：再自研，把核心能力沉淀成自己的系统资产
```
## 六、选型 5 问
1. **这是验证价值，还是建设长期能力** — 验证价值优先低代码，确定长期迭代优先框架或自研
2. **谁来维护这套系统** — 非工程师为主优先低代码，开发团队为主选框架或自研
3. **任务复杂度到底在哪** — 流程编排和资料接入复杂度低代码够，状态管理/回退/动态决策/多角色协作选框架，深入到权限/审计/成本治理/底层抽象一致性才自研
4. **你需要多深的控制力** — 需要自己定义状态结构/工具协议/人工检查点/评估日志体系很深，平台就会越来越不顺手
5. **这套能力会不会成为你的核心资产** — 辅助工具不必过度建设，决定交付效率/产品差异化/内部协作方式的就该选更长期的路线
## 七、总结
低代码回答"怎么最快把价值跑出来"，框架回答"怎么在不重造轮子的前提下把 Agent 正式纳入工程体系"，自研回答"当 Agent 已经成了长期系统能力，怎么把控制权真正拿回来"。
真正成熟的 Agent 选型，不是从工具崇拜开始，而是从任务判断和工程判断开始。
