# NL2SQL 在超大规模数仓场景的架构突破与工程实践

> 高德（Amap）基于 QoderWork Agent Skill 构建 NL2SQL 智能取数系统，覆盖 9 个业务域、352 张 ODPS 表、30,000 行知识文档。本文记录 V1 → V2 架构演进与知识工程方法论。

## V1：分域独立 Skill

V1 按数仓业务域拆分，每个域部署独立 QoderWork Skill。8 个取数 Skill（交易、业财、广告、流量、内容、供给、商品、服务）+ 1 个 odpscmd 工具 Skill。

结果：快速覆盖约 40 张核心 ADS 表，验证了 LLM 在企业数仓场景生成准确 SQL 的可行性。

### 五个结构性问题

1. **用户安装与选择成本高**：用户需从 9 个 Skill 自选安装和 @指定，产运同学不理解数仓域划分
2. **Skill 间路由天花板低**：语义匹配自动路由在术语歧义场景（"收入""供给"多域含义）下上限低且不可调试
3. **表规模扩充加剧域间冲突**：40 张→352 张后，跨域查询在多 Skill 架构下无法支持
4. **维护成本指数膨胀**：公共逻辑（日期计算、执行配置、权限处理）在 9 个 SKILL.md 各维护一份
5. **扩展迭代不便**：新增域需重组 Skill + 用户重新安装

### 认知转折

路由决策发生在控制范围之外，必须将路由逻辑收回到 Skill 内部。

## V2：统一 Skill + 确定性两级路由

### 四层架构

- **L1 域路由层**（SKILL.md 内置）：从用户问题提取意图，查确定性域路由表判定业务域，歧义时主动澄清
- **L2 知识加载层**（references/ 目录）：路由后只读取该域 reference 文件，9 个域 30,000 行知识，单次仅 200-400 行
- **L3 标准工作流层**（SKILL.md 内置）：6 步标准流程统一所有域
- **L4 公共服务层**（references/common/）：日期计算、SQL 安全约束等公共逻辑只维护一份

### 为什么选确定性路由而非 RAG？

1. **语义重叠**："收入"在广告域和业财域的 embedding 几乎一样但口径完全不同
2. **352 张表按域分组后可枚举**：9 个域，域内表 3-96 张，有限分类问题可用规则穷举
3. **可调试可追溯**：每次路由决策能回答"为什么选了这个域/表"

### 两级路由

- **一级路由**（SKILL.md 域路由表 ~200 行）：基于"用户想看什么指标、从什么视角看"的意图判断
- **二级路由**（DOMAIN.md ~100 行/域）：域内选表，按 ADS > DWS > DWD 优先级链回退

### 渐进式知识加载

四级加载（由粗到细）：域路由表 → DOMAIN.md → 表文档 → 字段细节。单次查询 200-400 行。

### 统一 6 步标准工作流

前置检查 → 域路由 → 知识加载 → 口径确认 → 选表+SQL 生成（5 点自检）→ 执行+结果解读

## 知识工程方法论

### 6 节知识卡片（每张表一个 Markdown 文件）

| 节 | 内容 | 作用 |
|----|------|------|
| S1 表元数据 | 全表名、数仓层级、粒度、分区、更新频率 | 决定"这张表能不能用" |
| S2 字段列表 | KEY/DIM/IDX 前缀标注 + 中文释义 | 区分 GROUP BY 和聚合字段 |
| S3 场景映射 | 适用/不适用什么问题 | 选表阶段避免"用错表" |
| S4 关联方式 | 需要 JOIN 的维度表和关联键 | 集中管理维度索引 |
| S5 指标口径定义 | 计算公式、口径说明、注意事项 | **最关键的**，口径来自知识库不由模型编造 |
| S6 SQL 模板 | 高频查询参考 SQL | 模板基础上参数化调整 |

### 元数据产出 Pipeline

1. 技术元数据拉取（Data Map MCP API 自动）
2. 初版文档生成（LLM 半自动，完成约 60%）
3. 业务知识注入（人工，BI 数仓同学补充口径定义）
4. 标准化校验（自动检查字段覆盖率、6 节结构）

### AI-Friendly 标准分

知识质量评分维度：字段覆盖率、口径完整度、SQL 模板覆盖率、场景标注率、路由规则完备度。

## 质量保障

- **900+ 测试题**覆盖全部业务域，错误归因分类（选表/语法/口径/语义）
- **DDL 一致性监控**（cici_monitoring）：定期对比线上表结构与知识库文档
- **用户反馈闭环**：纠正信号 + 重复提问 + 放弃对话三个沉默信号自动检测

## 设计原则

1. **确定性优于模糊性**：路由/选表/关键约束走规则而非概率输出
2. **按需加载优于全量灌入**：200 行路由表换 200-400 行按需加载
3. **知识分层对齐变更频率**：静态层/规则层/公共层
4. **标准化优于自由发挥**：统一 6 节文档、6 步工作流
5. **评测驱动优于经验判断**：标准分 + 回归测试

## 落地成效

- NL2SQL 准确率 >95%
- 智能问数场景覆盖率 >90%
- 对比传统手动取数（~8h）缩短 30 倍以上
- 同一架构可低成本复制到不同业务方向

## 未解决问题

- 取数 Skill 与分析 Skill 之间的跨 Skill 知识共享
- 知识库从树形目录到图状知识的平台化管理
- 端到端 Agent 评测体系（不只看 SQL 准确率）
