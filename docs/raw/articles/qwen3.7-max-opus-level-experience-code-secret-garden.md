# 国产模型，这回终于也有了 Opus 级体验！

大家好，欢迎来到  ** code秘密花园  ** ，我是花园老师（ConardLi）。

就在前几天，X 上刷到了阿里发布最新旗舰模型  ` Qwen3.7-Max  ` 。

说实话刚刷到的时候没特别在意，因为这一年国产模型刷榜太勤了。

GLM、DeepSeek、Kimi、MiniMax 轮流上，看着眼花缭乱的榜单，属实有点麻了。

不过，刷着刷着我发现有点不太对劲。

好多人都在拿  ` Qwen3.7-Max  ` 和海外的旗舰模型（GPT 5.5、Opus 4.7）进行实测对比。

似乎表现都还很不错？

这就勾起了我这该死的好奇心了。

恰逢周末，决定仔细了解下。

打开官方的推文，我发现这次真有点不一样。

官方给出了三个实测任务，全部围绕的都是，实测  ` Qwen3.7-Max  ` “到底能不能持续稳定的完成复杂任务”。

##  三个任务

Qwen 团队设计了三个任务，每一个都是需要模型连续工作几十个小时、做上千次决策的长线任务。

###  任务一：35小时，自己写出10倍加速的GPU代码

这个任务的目标很直接 — 给模型一段现成的 GPU 计算代码（用于大模型推理中的注意力计算），让它想办法跑得更快。

难点在于：跑代码的硬件是阿里自己的平头哥 PPU 芯片，模型训练时  ** 从没见过  ** 这个硬件。

没有文档，没有示例代码，没有性能分析工具的历史数据。

` Qwen3.7-Max  ` 拿到的只有一段基准代码和一个评测脚本。

接下来 35 个小时，模型完全自主地干了这些事：

写代码、编译、跑测试、看性能数据、找瓶颈、改架构、再跑、再改。

总共跑了 432 轮评估，调用了 1158 次工具。

最终成绩：比原始代码快了  ** 10 倍  ** 。

关键是优化不是一蹴而就的。

前几个小时提速最明显，但模型在第 30 个小时之后依然在找到新的优化点。

同一个任务，其他模型的成绩：

* GLM 5.1 做到 7.3 倍
* Kimi K2.6 是 5 倍
* DeepSeek V4 Pro 是 3.3 倍

###  任务二：在训练过程中抓"作弊"

这个任务理解起来需要一点背景。

大模型训练时，有一种方法叫强化学习（RL）：给模型出题，做对了给奖励，做错了扣分，靠奖惩信号让模型变强。

但模型有时候会 "钻空子" — 它会尝试去找到绕过评测的捷径。

比如偷偷从网上扒到标准答案，或者利用评测脚本的漏洞，让系统以为它答对了。

这叫  ** 奖励作弊（Reward Hacking）  ** 。

Qwen 团队让  ` Qwen3.7-Max  ` 去干一件事：监控另一轮 RL 训练的全过程，自动发现哪些训练样本存在作弊行为。

模型连续工作了 80 多个小时，自动拉取训练日志、回放模型的解题过程，总共调了上万次工具。

它自己总结出了多种作弊模式（比如各种方式偷看 GitHub 上的标准答案），然后把这些模式写成检测规则，再找反例来验证规则是否靠谱，不靠谱就改。

最终新增了 13 条检测规则，精准揪出了  ** 1618 个作弊样本  ** 。

###  任务三：经营一家创业公司，活一整年

最后一个任务是 YC-Bench — 一个模拟经营游戏。

模型扮演一家创业公司的 CEO，要在模拟的一整年里做几百轮决策：招人、签合同、接项目、识别骗子客户、控制人力成本、保证公司不亏钱。

每一步的选择都会影响后面的局面，而且环境是动态变化的 — 人力成本会涨，会出现故意坑你的恶意客户。

` Qwen3.7-Max  ` 最终营收达到  ** 208 万美元  ** ，完成了 237 个任务。

作为对比，上一代  ` Qwen3.6-Plus  ` 只赚了 105 万，再上一代  ` Qwen3.5-Plus  ` 只有 35.2 万。

更有意思的是模型的策略演化过程：

它一开始会主动探索不同类型的客户，踩了几次坑之后学会识别恶意订单，把骗子客户拉进黑名单，然后把精力集中在稳定的收入来源上。

中期遇到过一次 "经营危机"，模型自己调整了策略恢复过来，最终收敛到一个高效稳定的运营节奏。

这个任务考验的不是某一步的聪明，而是  ** 几百步连续决策的一致性  ** — 不能前面定的策略，后面就忘了。

##  关键榜单

再来看看官方公布的几个 Benchmark ，有几个我觉得还是挺亮眼的。

> [ 世界最顶级的大模型，都在 PK 些啥？ （大模型评估完全指南） ](<https://mp.weixin.qq.com/s?__biz=Mzk0MDMwMzQyOA==&mid=2247505276&idx=1&sn=d57b06af2f091ae00bc9ef1d723cb9d4&scene=21#wechat_redirect>) ，我之前的这篇文章详细介绍了大模型评测的主要方法。如果你觉得看各种大模型的榜单很吃力，可以先看看我这篇文章。

###  编程能力

` SWE-Verified  ` 这个基准专门测 "给你一个真实开源项目的 bug，你能不能修好"。

Qwen3.7-Max 拿了 80.4，Claude Opus-4.6 是 80.8，DeepSeek V4 Pro 是 80.6。

三家基本贴着打。

但换成更难的 SWE-Pro — 跨文件、跨模块的复杂工程问题 — Qwen3.7-Max 拿了 60.6，这个分数目前排第一名。

###  终端操作

` Terminal Bench 2.0  ` 测的是丢给模型一个终端，它自己敲命令、读日志、定位问题。

Qwen3.7-Max 69.7，DeepSeek V4 Pro 67.9。

别小看这两分的差距，终端操作是个链式任务，前一步错了后面全崩，能稳定拿高分说明它犯低级错误的概率已经很低了。

###  工具调用

` MCP-Mark  ` 60.8（GLM 5.1 是 57.5），  ` MCP-Atlas  ` 76.4（Opus-4.6 是 75.8）。

这两个基准测的是模型能不能正确理解一个工具怎么用、什么时候该用哪个、出错了怎么修。

前面那个 35 小时 Kernel 优化任务里连续调用 1158 次工具不崩盘，就是工具调用这项基本功够扎实。

###  推理能力

` GPQA Diamond  ` 是博士级科学问答，Qwen3.7-Max 92.4，Opus-4.6 91.3。

` HMMT 2026  ` （哈佛-MIT 数学竞赛题）97.1 对 96.2。

` HLE  ` （号称"人类最后的考试"）41.4 对 40.0。

这几个基准有个共同点：蒙对的概率几乎是零。在这种难度下，每多拿一分都得靠真本事。

总的来说，Qwen3.7-Max 在编程工程、工具调用、高难度推理这几个维度上，都站到了跟 Opus 4.6 同一档的位置。

对于一个要拿来当智能体底座的模型，这组成绩相当够用了。

##  实战测试

好，数据聊完了，已经迫不及待想试试了。

关注我的粉丝应该还记得，之前我发过一篇「让 Agent 自动制作知识讲解视频」的教程，把我自己开源的  ` web-video-presentation  ` Skill 完整拆了一遍。

这个 Skill 的核心思路是把一篇文章变成一个精美的网页演示，逐步揭示、有动画、有节奏感，录屏就能直接出片。

这次，我们直接让  ` Qwen3.7-Max  ` 给自己做个宣传视频。

想要使用这个模型，大家需要先注册一下阿里云百炼平台，然后生成一个 API Key，复制保存下来。

https://bailian.console.aliyun.com/

###  接入 Claude Code

接入过程非常简单，还是用 CC Switch。

这个工具之前教程里讲过，它可以让 Claude Code 切换为任意自定义模型。

打开 CC Switch，点右上角 "+"，选预设的阿里云百炼供应商：

填上我们刚刚复制的 API Key，然后把模型名称全部改成  ` qwen3.7-max  ` ，保存，回到首页点「启用」。

回到终端输入  ` claude  ` ，完事了。

Claude Code 的界面、工具链、Skill 系统全部保留，只是底层模型换成了  ` qwen3.7-max  ` 。

##  写文章

首先，我们让它写一篇关于自己的文章。

因为它刚发布没几天，网上的信息散落在官方博客、第三方评测、X 上的讨论、各种媒体报道里。

模型得自己去搜、自己去整理、自己判断哪些信息可靠，最后组织成一篇有结构的长文。

写作阶段大概花了 5 分钟。

出来的文章结构清晰，从定位到数据到技术路径到行业意义，层层递进。

数据引用准确，没有瞎编的分数。关键信息都标了来源。

这个任务本身不算特别难，但它验证了一件事：Qwen3.7-Max 在信息检索和内容组织上是靠谱的。搜索策略合理，信息整合有条理，写出来的东西能用。

###  口播稿 + 开发大纲

然后我把写好的 Qwen3.7-Max 文章丢给它，让它按照 Skill 流程来：

> 读取 @article.md，按照 /web-video-presentation 要求，完成第一步。注意编写完成后使用 Agent Teams 创建独立 Agent 进行质检。

开干。

它拿到文章之后，它开始同时产出两样东西 —— 口播稿和开发大纲。

口播稿把原始文章改成了适合视频的表达。原文里那些书面化的句子全部打散重组，变成短句、口语、像人在聊天一样的节奏。

开发大纲则把口播稿拆成了章节和步骤。7 个章节，每章 4-6 个步骤，总共 43 步。每一步标注了画面上应该呈现什么信息，用什么视觉形式。

这两份文件完成后，它按照 Skill 规范自动创建了两个 Reviewer Agent 做质检。一个审口播稿的口语化程度和信息保留率，一个审大纲的结构合理性和节奏规划。

质检完各自给出修改建议，主 Agent 整合建议修订了一轮，输出最终版本。

整个第一阶段大概 8 分钟。全自主，没有需要我介入的地方。

模型对 Skill 规范的理解非常准确。它知道什么时候该停下来做质检，知道质检应该交给独立的 Agent 做，知道质检结果应该怎么整合回主流程。

###  第一章开发 + 验收

Skill 的设计是先做第一章让人验收，确认视觉气质和节奏都对了，后续章节才以它为基准开始并行。

主题我选了  ` blueprint  ` ，深蓝底白色线条，技术绘图的质感。

第一章做出来之后，它又开了一个独立 Review Agent 质检，逐项对照清单检查：有没有动画效果、字体大小是否合适、信息密度够不够、CSS 有没有用章节前缀而不是全局类名。

质检通过后停下来等我验收，对味了，就是这个感觉：

###  多 Agent 并行开发

三个子 Agent 同时启动，分别负责不同章节，多 Agent 调度跑的还是非常稳健的。

并行开发的前提是  ** 隔离  ** 。

每章独立文件夹（  ` src/chapters/01-intro/  ` 、  ` src/chapters/02-benchmarks/  ` ），每章独立 CSS 前缀（  ` .intro-hero  ` 、  ` .benchmarks-chart  ` ），全局主题变量兜底视觉统一。

这些隔离规则全部写在 Skill 规范里。三个并行 Agent 只要有一个没遵守 — 比如 CSS 类名没加章节前缀用了个通用名  ` .card  ` ，合并的时候样式就冲突。

三章同时完成，各自走质检，全部通过。后续章节继续接上，全部完成。

我在浏览器里完整跑了一遍，43 个步骤全部正常。没有空白页，没有动画卡顿，没有内容错位，没有样式冲突。

###  几个体感

跑完这套流程，回头看那些 benchmark 分数，感受确实不一样。

** 长链路稳定性  ** ：从口播稿到最后一章完成，十几分钟的执行链路里，模型没有在任何一步丢失前面定下的规则。口播稿的节奏决定了大纲的步骤数，大纲决定了每章的信息密度，第一章的视觉基调决定了后续章节的方向 — 这些跨步骤的依赖关系，它全程保持住了。

** 指令遵循  ** ：Skill 里有几十条规则，从文件命名格式到 CSS 前缀到质检流程，一条都没丢。这是 IFBench 79.1 在实际场景里的直接体现。

** 多 Agent 调度  ** ：并行的三个子 Agent 互不干扰，各自独立完成任务。主 Agent 分配任务、传递上下文、回收结果的过程很干净。

** 跨框架适配  ** ：整件事是在 Claude Code 里完成的。Qwen3.7-Max 作为一个非 Anthropic 的模型接进来，tool calling 格式、文件操作、子进程调度全部正常工作，没有出现任何兼容性问题。

我之前用这套流程跑过很多次了，知道哪些环节容易出问题 — 多 Agent 并行时的约束遗忘、长链路后半段的上下文散失、质检环节被模型敷衍跳过。

这些坑 Qwen3.7-Max 一个都没踩。

##  最后

` Qwen3.7-Max  ` 的实测效果给我的感觉还是很惊艳的，体感上非常接近 Claude Opus 的感觉了。

看了眼价格，6 元/100 万 Token（当前 5 折），大概相当于 Opus 1/6 的价格。

在这个时间窗口上，性价比还是非常高的，推荐大家一用。
