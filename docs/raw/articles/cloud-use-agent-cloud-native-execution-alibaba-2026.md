# Cloud Use：当 Agent 开始真正使用云

> 泮圣伟 · 阿里技术（微信公众号） · 2026-07-09

Qoder Cloud Agents 工程实践系列。Cloud Use 的边界是云原生执行——让 Agent 以受治理的身份、凭证、环境、生命周期和审计轨迹，真正进入云的生产体系。

核心论点：仅具有 Tool Use 的 Agent 解决"模型如何调用工具"；Cloud Use 解决"云如何接纳 Agent 成为受治理的使用主体"。

## 01 云迎来了第二类使用者

过去的两个使用者：人类操作者、确定性自动化程序（CI/CD/Terraform/K8s Controller）。
Agent 是第三类：有人的目标理解能力 + 程序的执行速度和调用规模。不需要控制台和鼠标，却需要身份、权限、工具、运行时、状态管理、成本记录、失败恢复和审计链路。

## 02 为什么 Agent 很难成为云上工作负载

仅具 Tool Use 的 Agent 站在人的影子里：借人的账号、AK/SK、在线状态维持。无身份、凭证、运行时和审计体系，难成稳定工作负载。

## 03 Cloud Use 三类场景

1. **周期性任务**：从"有人每天打开系统"到"任务自己醒来"。样例：一次 BI 分析 111 次工具调用，21.5 分钟，0 人工干预。一次 ETL 116 个事件，13 分钟，cron 触发。
2. **诊断任务**：从"人喂材料"到"Agent 进入现场取证"。CI 失败诊断 3-8 分钟写回 MR，慢查询诊断 2-5 分钟给结论。
3. **执行任务**：从"脚本自动化"到"受治理执行"。低风险自动，高风险审批，全过程可追溯。

## 04 咖啡品牌 BI 案例——六道工程门槛

1. **凭证**：先进 Vault，不进 Prompt。vault_credential_id 引用，运行时由平台兑换注入。
2. **路径**：Skill 沉淀踩坑史。避免约 13 分钟错误路径探索。
3. **角色边界**：任务合约（结论带数字 → 数字回到查询 → 异常不能越权）。
4. **运行时**：云上 Session。21 分 32 秒，111 次工具调用，312 个事件，0 人工干预。
5. **结果回传**：监听正确事件（session.thread_idled），Webhook 签名验证。
6. **验收**：可执行的 Rubric。每个结论有数字，数字回到 SQL 查询结果。

## 05 Cloud Use 四层能力（依赖链）

1. **Identity Use**：谁在操作。Agent 不能冒用人的账号，每次动作可回答任务发起者、执行身份、权限来源、过期时间。
2. **Credential Use**：凭证怎么用。Vault、短期令牌、受控注入、服务端代理——能用但看不到。
3. **Tool/API Use**：工具怎么被治理。MCP + 权限/参数约束/调用审计/速率限制/风险确认。
4. **Runtime Use**：任务怎么活下去。云端 Session、状态管理、事件流、取消机制、结果回传。

## 06 从"人操作资源"到"Agent 承担任务"

Agent 让"任务"本身成为新的云抽象。过去云核心抽象是资源；Agent 时代还需要面向任务和机器执行体的新抽象。

## 07 Cloud Use 的难点——让 Agent 失败得可控

1. **错误路径**：Agent 会走向看似合理但不可用的 API。解法：正确+错误路径都沉淀进 Skill。
2. **基础设施异常**：资源组冻结 → 受控自救（检查→预授权创建→记录），超出边界停止。
3. **结果回传异常**：监听错事件拿半成品 → 线程 idle 后再拉取 + 签名验证 + 幂等 + 重试。
4. **验收过松**：Rubric 太宽易过，太细反复修正。写可执行可检查的标准。

## 08 边界：从低风险高频可验证开始

成熟度三阶段：巡检/诊断/生成（只读）→ 带确认的执行（IaC review/流水线确认）→ 高风险变更（仍需人审批）。
