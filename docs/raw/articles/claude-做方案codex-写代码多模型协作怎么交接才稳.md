# Claude 做方案，Codex 写代码：多模型协作怎么交接才稳

架构师（JiaGouX）我们都是架构师！  
架构未来，你来不来？

  


  


模型答错当然麻烦，更烦人的是每换一个agent，就得重新当一次搬运工。

设想一个很常见的任务：Claude 先查资料、做系统设计，Codex 进入仓库实现，Gemini 最后做核查。

Claude 已经研究了两个小时，列了来源，也推翻过一版方案。Codex 接手时，除了目标，还要确认设计版本、当前分支、未提交修改和测试状态。到了 Gemini 审查，它还得先弄清楚：眼前的代码，对应的是 Claude 哪一版设计？

工具换了，人还在搬上下文。

Rahul 在《How To Build a Multi-Model AI Team in 2026》里写的是 Claude、ChatGPT 和 Gemini，但他对这种状态的形容很准确：**人最后成了几个 AI 之间的 API。**

他推荐的 Unibase Memory，正是来减轻这部分工作的。这个 Chrome 扩展可以保存 ChatGPT、Claude 和 Gemini 的对话，统一搜索，再把选中的内容送到另一个模型。

坦率说，我最初把它看成一个更顺手的复制粘贴工具。翻完官方文档、Chrome 商店页面和发布代码后，这个看法有点低估它了。Unibase 至少把记忆从某一家模型的聊天历史里拿了出来，重新交还给用户。

不过，把它直接叫作“多模型 AI 团队”，又有些着急。

**可携带记忆解决了“过去放在哪里”。下一棒能不能接稳，还要看当前状态、验证证据和交接边界有没有说清楚。**

* * *

##  先把话说短一点

  * • **它能做：**  跨 ChatGPT、Claude、Gemini 保存、搜索和复用对话，少搬几次文字。
  * • **它带不过去：**  仓库现场、任务进度、测试结果和权限状态。
  * • **研发场景：**  Codex 更适合进入仓库实现，但不在 Unibase 当前的官方支持列表里。
  * • **工程上还要补：**  原始聊天用于追溯，权威状态单独维护，下一棒读取结构化交接。
  * • **试用边界：**  个人低敏任务可以先跑；团队和企业数据进入前，还要看安全、保留与退出策略。

* * *

## 写代码这一棒，我更倾向于 Codex

如果第二棒负责起草文章、归纳材料，ChatGPT 很自然，Unibase 也正好支持这条链路。到了进仓库实现这一步，我更倾向于 Codex。它会读代码、修改文件、执行命令、跑测试，再把 Diff 和结果交出来。

这时要交接的已经不止聊天文字：

  * • 当前仓库、分支和基准提交；
  * • 已修改文件、未提交 Diff 和测试结果；
  * • 当前采用哪版设计，哪些事实已经确认；
  * • 哪些权限已经开放，碰到什么情况要停下来。

**这里要把事实边界说清楚：Unibase Memory 当前官方支持的是 ChatGPT、Claude 和 Gemini，没有写明支持 Codex。**

这个错位正好把边界照出来：浏览器记忆能带走文字，带不走仓库、命令、测试和权限现场。

* * *

## 共享记忆现在能做到哪一步

每个模型各自只看见自己聊天框里的那段过去。上下文每换一次产品，就断一次。

项目目标、表达偏好、已经查过的资料、被否掉的方案，都被留在不同产品的会话历史里。人一换工具，就得把这些内容重新捞出来，判断哪些还有效，再贴到下一个窗口。

Unibase Memory 想在这些聊天框上面加一层用户自己的记忆。按官方文档，它目前支持：

  * • 自动保存新对话，也可以按时间范围批量导入；
  * • 搜索、打标签、收藏和筛选已有记忆；
  * • 保存网页或选中文本，再把需要的内容送入当前 AI；
  * • 默认保存在本地，可选择同步到 Membase，也支持 JSON 导出。

对经常切换模型的人，这些能力很实用。至少，查过的东西不必跟着某个聊天窗口一起沉下去。

它目前做的是记忆捕获、检索和搬运。仓库有没有变化、测试是否通过、谁有权提交代码，这些还不在产品边界里。换个工程说法，它提供了记忆通道，还没有提供共享运行时和任务编排。

我更关心的是这个变化：**记忆正在脱离单个模型，慢慢变成用户自己持有的资产。**

只是产品还很早。

截至 2026 年 7 月 17 日，Chrome 商店里的版本是 0.20.70，约 80 位用户、5 条评分，最近一次更新在 7 月 13 日。Rahul 的原帖浏览量已经超过 21 万，传播热度和实际使用规模并不在一个量级。

## 把整段聊天搬过去，问题就解决了吗

回到开头的任务。

Claude 在研究过程中可能提出过 A、B 两套方案，最后因为一份官方文档否掉了 A。可如果下一棒拿到的是完整聊天，它看到的只是几十轮按时间排列的文字。A 出现得更多，B 出现得更晚，哪一个才是当前结论，需要它自己再猜一次。

Codex 开始写代码后，又会产生新的状态：改了哪些文件，哪项测试通过，哪个边界还没处理，当前分支对应设计的哪个版本。等 Gemini 做审查时，如果这些信息仍然散落在聊天和终端里，它很容易拿旧设计核对新代码，或者把探索阶段的猜测当成已确认事实。

聊天历史带过去了，工作现场还留在原地。

  
完整聊天与结构化 HANDOFF 的差别

 _图 1：共享记忆让下一棒找到过去，结构化 HANDOFF 让它从当前状态继续。_

这其实是个上下文工程问题。

Shopify CEO Tobi Lütke 提到的重点，是给模型足够的上下文，让任务变得可解。Andrej Karpathy 补充说，信息太少会缺依据，太多、太杂同样会拖累效果。

Anthropic 给出的目标更直接：找到**最小的一组高信号内容** ，让目标结果出现的概率尽可能高。

他们谈的都不是 Unibase，我借用的是同一条工程原则：归档可以很全，交给下一棒的工作集要窄。能保存全部历史，和下一棒能把事做对，中间还隔着筛选与交接。

我们在《[Agent Harness 上下文管理：聊天记录还是工作集？](<https://mp.weixin.qq.com/s?__biz=MzAwNjQwNzU2NQ==&mid=2650409162&idx=1&sn=62556a10e227bcb8d977a4f3e0006c8b&scene=21#wechat_redirect>)》里把上下文窗口看作运行时工作集。资料仓库可以很大，当前窗口只放眼下推理需要的东西。放到多模型协作里，可以再拆成四层：

  
  
---  
层| 放什么| 怎么更新  
---|---|---  
归档| 原始对话、工具输出| 尽量只追加，便于追溯  
知识| 已确认事实、来源、适用时间| 核验后写入，纠正时标记旧版失效  
状态| 当前版本、负责人、阻塞项、已完成动作| 只保留一个权威版本  
交接| 下一棒需要的目标、证据、动作和停止点| 每次任务按需生成  
  
原始聊天更像事件日志，记录一路发生过什么；`HANDOFF` 是当前状态的快照，告诉下一棒现在该相信什么、从哪里继续。两者都要留，但不能混成同一个对象。

多做这一步整理，看起来慢了一点，通常比让下一个模型重新考古省事。

## 下一棒到底缺什么

如果把模型当作接力中的下一棒，它真正缺的是几句很朴素的话：

  * • 这次到底要完成什么，怎样才算完成；
  * • 现在做到哪里，以哪个版本为准；
  * • 哪些事实和测试结果可以复核；
  * • 下一步做什么，遇到什么情况先停下来。

我们在《[Anthropic 8x 之后：Agent 协作真正卡在验证和交接](<https://mp.weixin.qq.com/s?__biz=MzAwNjQwNzU2NQ==&mid=2650409776&idx=1&sn=b6d6fa96502cfdc093c21841a568daaf&scene=21#wechat_redirect>)》里，把这些问题拆成 `SPEC`、`STATE`、`EVIDENCE`、`IMPACT`、`PERMISSION`、`HANDOFF` 六个接口。多模型场景没有改变这组问题，只是换模型以后，少了哪一项会更快显现。

模型之间当然有能力差异，项目里更先遇到的却常常是交接损耗。模型明明会做，拿到的却是错版本、半截结论或缺来源的摘要，结果自然不稳。

所以我现在看多模型架构，会先检查交接字段，再看角色分工。角色名换得很漂亮，交接字段还是空的，最后仍要靠人把现场重新拼起来。

## 一份 HANDOFF，应该写什么

团队准备试跨模型协作时，我会先放一份很小的 `HANDOFF.md`，暂时不搭复杂平台。
    
    
    # HANDOFF  
      
    交接编号：  
    任务与完成条件：  
    仓库 / 分支 / 基准提交：  
    权威设计及版本：  
    当前状态：  
    已修改文件 / 未提交 Diff：  
    已确认事实与来源：  
    关键决策及原因：  
    测试命令与结果：  
    待确认问题 / 阻塞项：  
    下一步动作：  
    权限边界与停止条件：

还是前面的例子。Claude 交出系统设计时，不必把两小时探索过程原样塞给 Codex。交接包里留下当前方案、被否掉的备选方案、否决依据、官方来源和未决问题就够了。

Codex 接手后，先确认权威版本，再开始实现。完成一段修改，就把产物位置、Diff、测试结果和仍未覆盖的边界写回去。若实现与已确认事实冲突，差异单独列出，交给人裁决，不在执行过程中直接改写权威事实。

Gemini 做核查时，读取的是当前代码、对应的设计版本和待核清单。它逐项给出通过、存疑或错误，并附上证据，不在审查阶段另起一套方案。

我还会加一个很便宜的动作：接收模型先回一份三行确认，不急着执行。
    
    
    我将基于：设计 v2 / commit abc123  
    本次只做：实现核心路径并补测试，不修改生产配置  
    仍未确认：异常重试上限，遇到该问题先停止

三行对不上，先校准版本和任务范围。确认只多了几行字，却能挡住一类很贵的返工：模型能力没有问题，只是从错误的起点一路做了下去。

这份确认对得上，再开始执行。做完以后，把结果和证据写回权威状态。这样至少能复核接手前后的变化。

  
Claude、Codex、Gemini 的交接流程

 _图 2：一次可复核的接力，每一棒都读取交接、确认边界，并把证据写回权威状态。_

人仍然在流程里，但工作变了。人处理争议、权限和最终取舍，不再反复充当背景复述器。

这套交接里，我最看重的一条是：**权威版本只能有一个。**  三个模型可以各自保留思考过程，项目状态不能在三个地方同时演化。

Alex Iskold 说过一句很短的话：`Context engineering != process engineering`。上下文工程决定下一棒看见什么；流程工程规定任务怎样继续、何时停止、凭什么验收。共享记忆改善前一件事，`HANDOFF` 开始补后一件事。

模型以后可以替换。今天用 Claude 调研，明天换 Gemini，只要交接格式、证据要求和停止条件稳定，流程就不必跟着推倒重来。

## 共享记忆为什么会打架

一旦多人、多模型同时读写，搜索反而成了最简单的问题。

假设 Claude 把设计更新到 v2，Codex 按 v2 写出了实现，Gemini 却从记忆库里搜到了排在前面的 v1。三边都没有出故障，最后仍可能得到一次错误审查。

2026 年 3 月的一篇论文《Multi-Agent Memory from a Computer Architecture Perspective》把这类问题放到计算机体系结构里讨论。论文区分了共享记忆和分布式记忆：前者方便复用，容易遇到覆盖、陈旧数据和版本冲突；后者隔离更好，但同步成本高，状态也容易分叉。

现实系统大多会落在中间：每个 agent 保留局部工作记忆，只把筛选后的产物放进共享层。

要让这层共享长期可用，几条规则绕不过去：

  * • 每条事实带来源、时间、版本和状态，旧结论被纠正后要明确失效；
  * • 只有当前负责人能改权威状态，其他模型先提交建议；
  * • 更新携带 `base_version`，版本已变化就拒绝覆盖并保留冲突证据；
  * • 读、写、外发权限分开，删除策略也要写清楚。

这套规则缺位时，共享层主要承担检索和归档。单写者加 `base_version`，其实就是很朴素的乐观并发控制，已经能挡住不少旧结论覆盖新状态的问题。

浏览器扩展可以解决个人记忆的入口问题。企业里的权威状态、并发更新和责任边界，还得由更完整的系统承担。

## 这个扩展，能不能放心装

交接机制说清以后，还有一个绕不开的试用边界：这层长期记忆，能不能放心放进浏览器。

这个点我犹豫了一下。客户端加密确实做了，文档和代码又没有完全对上，简单说“安全”或“不安全”都不准确。

先说目前能确认的部分。

官方文档称，记忆默认保存在本地；打开同步后，内容先在设备上加密，再把密文发往 Membase。0.20.70 扩展里的 `crypto.js` 与这份说明基本一致：密钥由钱包签名派生，内容采用 Fernet 格式，底层使用 AES-128-CBC 和 HMAC-SHA256。

扩展的站点范围主要集中在 ChatGPT、Claude、Gemini、Unibase、Privy 和本地模型下载相关域名，没有直接申请所有网站的长期读取权限。

但它依然处在高信任边界里。

Manifest 里的权限包括 `tabs`、`identity`、`scripting`、`cookies`、`storage` 等，并会在三家 AI 网站注入内容脚本。它必须读写聊天页面，才能完成捕获和发送。功能与风险来自同一个入口，没办法只取前者。

代码里还有一处差异，让我停了一下。

官方架构文档写的是只记录匿名、无内容的使用事件，并明确表示不会记录钱包地址。当前发布包的 `usage-report.js` 会在某个地址首次使用 `send` 或 `sync` 时，向官网 `/api/usage` 发送 `{event, address}`，相关调用点的代码注释写着 `KOL funnel`。

**这里发送的是功能事件和钱包地址，不是聊天正文；但“不记录钱包地址”的文档表述与发布代码确实没有完全对齐。**

  
  
Unibase Memory 的数据与信任边界

 _图 3：聊天内容的本地与同步路径、功能使用上报，是两条不同的数据流。_

Chrome 商店的隐私披露还列出了个人身份信息、认证信息、个人通信、用户活动和网页内容等处理类别。这些分类比较宽，不能直接推导出所有内容都会传到远端。安装前，权限和数据路径仍值得看清楚。

删除这件事也要单独看。官方 Features 文档写明，已经同步到 Membase 的记忆采用软删除：内容会从面板隐藏，但远端加密副本仍然保留，可以恢复。对误删恢复是好事；如果团队要求“删除后远端彻底清除”，这份文档还没有给出相应承诺。

这次能确认的范围只到客户端发布包里的关键代码，还算不上完整安全审计。后端、构建链、依赖供应链、远端存储和密钥恢复流程都不在检查范围内。

试用边界可以分成三档：

  * • 个人低敏任务：独立浏览器配置，先关闭远端同步，实际检查一次导出、恢复和删除；
  * • 团队非敏感资料：再补权限评审、统一退出流程和遥测开关，避免每个人各自保存一套长期副本；
  * • 客户数据、源码机密和受监管信息：先等独立安全审计、数据处理协议、企业身份管理和可验证的彻底删除能力。

对效率插件，这套要求也许显得重。对长期记忆层，我觉得刚好。

## 先跑一周，别急着导入全部记忆

跨模型协作的第一步，不必是“把过去一年的聊天全部搬进来”。

挑一条经常发生的链路就够了，比如“Claude 调研、Codex 实现、Gemini 审查”。找一个低敏任务，跑一周，先看看交接到底卡在哪里。

最好先留一组基线：不用共享记忆时，第二个模型接手要花多久、需要补问几次、拿错版本几次。没有这组数字，试用结束后很容易只记得“少复制了几段文字”，却说不清整体返工有没有下降。

我的做法会比较保守：代码以仓库为准，项目状态以 `HANDOFF.md` 为准；设计文档只认当前版本。记忆库负责查找和复用，不负责裁决最终版本。

试跑第一轮，我只会开放读取和生成草稿。自动外发、自动提交、覆盖权威文件这些动作先关着。流程跑顺后，再一点点增加权限。

观察指标也可以很朴素：

  * • **接手成本：**  Codex 开始工作前，还需要人工补充几次背景；
  * • **版本一致性：**  有没有拿错旧版本；事实修正后，其他模型是否还沿用旧结论；
  * • **证据完整度：**  调研有没有来源，代码有没有 Diff 和测试，审查结果能否复核；
  * • **总体返工：**  少复制了一些文字以后，人工纠错和返工到底是多了还是少了。

任务量允许的话，还可以拿两个复杂度相近的任务做个小对照：一个传完整聊天，一个只传结构化 `HANDOFF`。比较接手时间、追问次数、旧版本误用和最终验收结果。样本当然很小，至少能让团队看到自己的工作流究竟卡在记忆、交接，还是验证。

如果接手时间缩短了，错误版本也少了，这层记忆才真的帮上了忙。若只是复制粘贴变少，纠错成本却上升，系统反而被搭重了。

* * *

## 写在最后

Unibase Memory 让我确认了一件事：跨模型记忆已经从一个使用抱怨，开始变成具体产品。它真正有价值的地方，是把记忆从单一聊天产品里拿出来，让下一棒找得到过去。

但过去不等于当前状态。

回到开头那场接力，Codex 还得知道看哪一版设计、从哪个提交开始、哪些测试已经跑过，以及什么情况下不能继续往前改。

我并不期待人完全退出这条链路。我更希望看到的是，人把时间留给争议、边界和取舍，不再每天替几个模型重复介绍同一个项目。

**跨模型协作能不能跑稳，取决于换了模型或换了人以后，工作还能不能沿着同一份状态和证据继续。**
