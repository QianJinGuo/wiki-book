## 一、为什么要知识库

领域知识决定了 AI 在业务中能发挥多大的价值和作用。任何 AI 系统都由模型、知识、架构三部分组成。模型由供应商提供，只能被动接受；架构常因模型能力升级而失效重做。相比之下，领域知识只能从内部积累——不可替代，且随业务演进而持续变化，是最值得长期投入的部分。

领域知识的沉淀面临诸多挑战：知识散落在代码和注释、配置、钉钉文档、沟通记录等各处，没有统一载体。直接套 RAG 解决不了这件事——RAG 不改变原始材料本身的状态，散落的还是散落的，矛盾的还是矛盾的，过期的还是过期的。

> 问题出在知识本身，不在检索。需要的是在检索之前加一道"编译过程"——把散落、矛盾、易腐化的源材料，预先加工为可被 AI 直接消费的知识。这是 LLM Wiki 的起点。

## 二、实际效果

### 指标和维度召回
系统通过 index.md 定位域 → 全域搜索召回候选表 → 逐张读取详情页判断覆盖度和相关性。精准推荐首选表，同时区分增量明细表和宽表，主动排除名称相似的干扰项。

### SQL 代码生成
用户问"不同主播等级的开播场次数怎么算"，系统自动跨域定位到主播基础信息宽表 + 主播标签预聚合表，输出 JOIN 关联的 SQL 骨架，同时标注废弃字段和口径差异。

### 数据模型迭代（核心场景）

传统做法依赖人工逐表排查 SQL 代码，耗时易出错。基于 LLM Wiki 的模型迭代流程拆解为 6 步：

| 步骤 | 做什么 | 输入→输出 |
|------|--------|----------|
| 需求拆解 | 理解需求，识别需求类型 | 需求描述 → 需求类型 + 指标口径 |
| 知识库召回 | 从 Wiki 查目标表，拉取 DDL 和口径定义 | 表名 → 表结构 + 口径说明 |
| 血缘查询 | 用 graph.json 查全链路下游 | 目标表 → 下游表完整列表 |
| 风险评级 | 看下游 SQL，判断业务影响 | 下游表 → 二维风险矩阵 |
| SQL 生成 | 批量生成 DDL+ETL+验证SQL+回滚SQL | 影响分析 → 全套 SQL 文件 |
| 报告输出 | 整合分析结果 | 全流程结果 → 可交付报告 |

**关键发现**：间接下游表中有 3 张参与准入判定，切换后阈值需调整。

**最终效果**：

| 事项 | 传统人工 | AI 编排 | 提效 |
|------|---------|---------|:----:|
| 血缘查询时间 | 30min（手动递归） | 2min（--impact） | 15× |
| 下游表遗漏率 | 20%（间接下游易遗漏） | 0%（强制完整性校验） | 100% |
| SQL 生成时间 | 0.5天（逐表改写） | 10min（批量生成） | 72× |
| 风险判定一致性 | 低（凭经验） | 高（二维矩阵标准化） | 质变 |

## 三、LLM 编译器：怎么建

### 核心思想

LLM Wiki 不是一份"用 LLM 写出来的文档"，而是一份结构化、有约束、可验证的知识资产。四个层面：

1. **结构可解析** — 每个页面是 frontmatter + 正文的双层结构
2. **层级可下钻** — 域按业务主题嵌套组织，形成可逐层下钻的树
3. **关系可遍历** — 页面之间的血缘、归属、消费、引用关系以边的形式显式记录
4. **正确性可度量** — 分结构、语义、人工三层校验

**让 LLM 来建，不是让人来建**。LLM 作为编译器，把散落的源材料（DDL、任务代码、文档、接口配置）编译成结构化页面，人只在关键节点做确认。

构建过程抽象为六个步骤：提取 → 生成 → 归类 → 聚合 → 链接 → 验证——对应编译器的词法分析、代码生成、作用域解析、IR 构建、链接、静态分析。

### LLM Wiki 与 RAG：互补而非冲突

Wiki 是编译时产物（把原始材料预处理成高质量的知识页面），RAG 是运行时手段（在查询时刻做精准召回）。Wiki 提供高质量语料，RAG 提供精准召回，组合起来才是完整的检索栈。

### 知识来源要全

5 类编译时知识载体：表（DDL）、接口（Config + Version）、文档（钉钉文档）、任务（独立任务代码）、看板（看板元数据）。

每张表不止抓 DDL，还要抓产出任务代码。DDL 提供结构信息，任务代码提供逻辑信息。

### 知识构建要准

四道工程纪律：
1. **噪音过滤** — 入口粗筛 + 生成细筛
2. **代码即真相** — 不同来源描述不一致时，以任务代码为权威
3. **生成与判断分离** — 需要推断的字段强制留空，不允许在生成阶段做主观推断
4. **证据链可追溯** — 每个页面在 frontmatter 中保留 sources 字段

### 知识的骨架是关系

关系从正文中抽出来，显式存储为图：血缘、归属、消费、引用等语义落到 frontmatter 字段，再由构建流水线统一扫描提取，沉淀为全局关系图。

### 为检索而组织

三层配合：聚合解决信息爆炸，渐进式披露解决上下文有限，多路召回解决召回不全。

## 四、架构设计

### 多级文件系统

```
${KB_ROOT}/
├── pre/              # 待处理清单 + 临时下载产物
├── raw/
│   ├── ready/        # 完整可用，直接进入 Wiki 生成
│   ├── pending/      # 存在缺失或问题
│   └── archive/      # 已下线或弃用
├── wiki/
│   ├── tables/{db_type}/
│   ├── domains/{parent}/
│   ├── concepts/
│   ├── metrics/
│   ├── dimensions/
│   ├── dashboards/
│   ├── apis/
│   ├── datasets/
│   ├── graph.json    # 全局关系图
│   ├── index.md      # 全局索引
│   └── overview.md   # 全景概览
├── log/
├── tmp/
└── schema/
```

### Schema 即契约

每个页面是 frontmatter + 正文的双层结构。frontmatter 是 YAML 格式的结构化头部，包含共性字段和页面特有字段（如 table 的 upstream/downstream/domain/layer，metric 的 computed_by 等）。正文是 Markdown 格式的语义内容，每种页面类型定义固定的章节模板。

### Agent 编排：编排层与干活层分离

编排器（wiki-orchestrator）只做四件事：意图路由、用户确认、子 Agent 调度、结果汇报。干活层 6 个 skill 各自覆盖一个阶段：

| Skill | 阶段 | 职责 |
|-------|------|------|
| wiki-material-prep | Phase 0 | 维护清单、抓取 DDL 和任务代码、验证分流 |
| wiki-base-generator | Phase 1 基础 | 生成表、接口、数据集、概念四种基础页面 |
| wiki-advanced-generator | Phase 1 高阶 | 生成域、看板、指标、维度、index、overview |
| wiki-graph-builder | Phase 1 链接 | 扫描 frontmatter 和 Wikilink，沉淀关系图 |
| wiki-health-check | Phase 2 | 执行结构、链接、格式等多项校验 |
| wiki-query | 运行时 | 检索页面、查询血缘、关键词搜索 |

## 五、知识编译流水线

### Phase 0：材料预处理
全脚本化执行（抓取、验证、分流是确定性逻辑），支持断点续传。

### Phase 1 基础：基础 Wiki 生成
批间串行、批内并行（每批 5 个对象，spawn 5 个子 Agent）。生成完后统一校验（frontmatter 字段是否符合 Schema 约束，表结构字段是否真实存在于 DDL）。增量感知——只对发生变化的对象触发生成。

### Phase 1 高阶：高阶 Wiki 生成
一阶段三路并行（域页面 + 看板页面 + 指标与维度），二阶段串行汇聚（index.md + overview.md）。

### Phase 1 链接：图构建
8 种节点类型（table/domain/concept/metric/dimension/dashboard/dataset/api），8 种正向边（has_upstream/syncs_from/writes_from/belongs_to/computed_by/uses/queries/wikilink）。只存正向边，反向按需计算，回填关键反向字段到 frontmatter。

### Phase 2：健康检查
6 项检查：数量一致性、结构完整性、链接有效性、Domain 格式、YAML 语法、Graph 完整性。任意一项未通过视为编译失败。

## 六、知识检索

三步检索栈：
1. **意图识别** — 精确查询 / 关系查询 / 模糊搜索（强制域推断）
2. **多路召回** — 域召回 + 关键词召回（两路并行）→ 图扩展补全血缘相关知识
3. **重排序输出** — 粗排（脚本 score）→ 精排（LLM 读详情，三维度：覆盖度/相关性/通用性）

## 七、增量编译与持续 Lint

增量识别的关键是扫描现状对账（源材料目录 vs 已有 Wiki 页面 vs sources 字段）。三类典型场景：新材料入库、知识腐化更新、结构性修复。

Lint 把健康检查从"构建后一次性兜底"扩展为"持续性质量巡检"。

## 八、总结

> 用 LLM 编译思维替代人工写 Wiki，把散落的源材料编译成结构化、有约束、可验证的知识资产。

核心设计决策：代码即真相做仲裁、生成与判断分离、三层校验、Schema 即契约、关系显式存图、编排与干活分层协作。
