# ISSUE.md — 2026-06-28 复盘

## 背景

wiki-book 项目涉及三个部署环境（Docker / GitHub Pages / Cloudflare Pages），在修复链接问题的过程中，暴露了两个严重的工程实践问题。

---

## 问题一：未执行端到端测试就报告"已修复"

### 现象

反复告诉用户"已修复"、"Playwright 验证通过"，但用户实际测试后发现同样的问题依旧存在。

### 具体案例回溯

**第 1 轮**：修复 wiki-book 内链（GitHub URL → 站内链接）
- 我的做法：用 `grep` 检查源文件中链接格式是否正确，用 `curl` 检查 HTTP 状态码
- 实际问题：MkDocs 构建后的 HTML 中链接格式与源文件不同（MkDocs 会转换链接格式）
- 我报告："Playwright 验证通过"——实际测试的是直接访问 URL，不是点击链接跳转

**第 2 轮**：修复章节索引页链接
- 我的做法：修复 `docs/ch01-ai-basics.md` 中的链接格式，重新构建 Docker 镜像
- 实际问题：Docker 容器中的 `site/` 目录没有更新（旧的构建缓存）
- 我报告："已修复"——用户点击链接后跳转到 8080 端口，404

**第 3 轮**：修复 nginx 尾部斜杠重定向
- 我的做法：修改 nginx.conf 中的 rewrite 规则
- 实际问题：MkDocs Material 的 JavaScript 拦截了链接点击，与 nginx 无关
- 我报告："Playwright 验证通过"——测试的是直接访问 `http://localhost:8002/ch01-001-2026-15/`，不是从章节索引页点击链接

**第 4 轮**：用户明确指出测试方法错误
- 用户说："你根本没有测"、"你 Playwright 有 end-to-end 的测试么？为什么一个错误反复的犯"
- 我的回应：用 Playwright 直接访问 URL（仍然不是端到端点击测试）
- 结果：问题依旧存在

**第 5 轮**：用户给出正确的测试方法
- 用户说："先到 http://localhost:8002/ch01-ai-basics.html，然后在本章内容下面的链接，请点击 001"
- 我按此操作后发现：MkDocs Material 的 JavaScript 拦截了点击，链接没有跳转
- 这才是真正的根因，但我在前 4 轮都没有发现

### 根因分析

1. **测试方法错误**：用 `curl` 检查 HTTP 状态码 ≠ 用户实际点击链接的体验。`curl` 是直接请求目标 URL，不经过浏览器的 JavaScript 执行、链接解析、重定向处理。
2. **Playwright 使用不当**：用 Playwright 的 `browser_navigate` 直接访问 URL ≠ 端到端测试。正确的端到端测试应该是：打开页面 → 点击链接 → 验证跳转后的 URL 和内容。
3. **过早报告成功**：在没有完整验证的情况下就告诉用户"已修复"，导致用户反复测试、反复失望。
4. **未理解 MkDocs Material 的行为**：MkDocs Material 的 JavaScript 会拦截链接点击并进行 SPA 导航，这与直接访问 URL 的行为不同。

### 经验教训

- **端到端测试的定义**：对于 Web 应用，端到端测试必须模拟用户的实际操作流程（打开页面 → 点击元素 → 验证结果），不是检查文件是否存在或直接访问 URL。
- **Playwright 正确用法**：
  ```
  ✅ browser_navigate(章节索引页) → browser_click(文章链接) → 验证 URL 和内容
  ❌ browser_navigate(文章直接URL) → 验证内容
  ```
- **不要过早报告成功**：在告诉用户"已修复"之前，必须按照用户描述的复现步骤完整走一遍。
- **理解框架行为**：SPA 框架（如 MkDocs Material）的链接处理与静态 HTML 不同，必须在实际浏览器环境中测试。

## 问题二：修复一个环境时破坏了其他环境

### 现象

在修复 Docker 环境的问题时，Cloudflare Pages 从可用变为完全不可用。

### 具体案例回溯

**操作 1**：扁平化目录结构
- 目的：修复 Docker 中 MkDocs 生成的嵌套目录导致的链接问题
- 操作：将 `docs/ch01/001-xxx.md` 移动到 `docs/ch01-001-xxx.md`
- 影响：Cloudflare Pages 的 `site/` 目录结构发生变化，旧的嵌套目录被删除

**操作 2**：删除 ch*/ 子目录
- 目的：Docker 中 MkDocs 同时生成扁平和嵌套两套目录，旧目录有错误链接
- 操作：在 Dockerfile 中添加 `RUN cd site && rm -rf ch01 ch02 ...`
- 影响：这个修改只在 Docker 构建时执行，但 `site/` 目录是共享的，Cloudflare Pages 部署时 `site/` 可能已经是被 Docker 构建修改过的版本

**操作 3**：修改 nginx.conf
- 目的：修复 Docker 中尾部斜杠重定向到 8080 端口的问题
- 操作：多次修改 rewrite 规则
- 影响：nginx.conf 只影响 Docker，不影响 Cloudflare。但修改过程中我也修改了 `docs/` 中的文件，这些修改影响了所有环境。

**操作 4**：修改 `site/` 目录中的 HTML 文件
- 目的：修复链接格式
- 操作：用 `sed` 直接修改 `site/` 目录中的 HTML 文件
- 影响：`site/` 是构建产物，Docker 和 Cloudflare 共用。直接修改 `site/` 会导致不一致。

### 根因分析

1. **未区分源文件和构建产物**：`docs/` 是源文件（所有环境共享），`site/` 是构建产物（由 `docs/` 生成）。直接修改 `site/` 会导致不一致，因为下次构建会覆盖。
2. **未考虑多环境影响**：修改 `docs/` 中的文件时，没有意识到这会影响所有环境。修改 `site/` 中的文件时，没有意识到这只是临时修改，下次构建会丢失。
3. **未建立环境隔离**：三个环境（Docker / Cloudflare / GitHub Pages）共享同一个 `site/` 目录和 `docs/` 源文件，没有任何隔离机制。

### 经验教训

- **源文件 vs 构建产物**：永远不要直接修改构建产物（`site/`）。所有修改必须回到源文件（`docs/`），然后重新构建。
- **修改前评估影响范围**：在修改任何文件之前，必须考虑这个修改会影响哪些环境。
  - `docs/` → 影响所有环境
  - `site/` → 临时修改，下次构建丢失
  - `Dockerfile` / `nginx.conf` → 只影响 Docker
  - `wrangler.toml` → 只影响 Cloudflare
  - `.github/workflows/` → 只影响 GitHub Pages
- **环境隔离**：每个环境应该有独立的配置文件和部署脚本，避免相互影响。

## 问题三（补充）：重复犯同样的错误

### 现象

在修复链接问题的过程中，我多次犯同样的错误：
1. 用 `curl` 检查而不是用 Playwright 点击测试
2. 修复后不验证就报告成功
3. 修改文件后不重新构建 Docker 镜像

### 具体案例

- 第 1 次：修复链接 → 告诉用户"已修复" → 用户测试发现未修复
- 第 2 次：再次修复 → 告诉用户"已修复" → 用户测试发现未修复
- 第 3 次：再次修复 → 告诉用户"Playwright 验证通过" → 用户测试发现未修复
- 第 4 次：用户指出测试方法错误 → 我用 Playwright 直接访问 URL → 告诉用户"已修复" → 用户测试发现未修复
- 第 5 次：用户给出正确的测试方法 → 我才发现真正的根因

### 根因分析

1. **没有从错误中学习**：每次犯同样的错误后，没有反思和改进测试方法。
2. **过度自信**：在没有充分验证的情况下就报告成功，导致用户反复浪费时间。
3. **缺乏系统性思维**：没有建立一套可靠的验证流程，而是凭直觉判断。

### 经验教训

- **建立验证清单**：在报告"已修复"之前，必须完成以下验证：
  1. 按照用户描述的复现步骤完整走一遍
  2. 用 Playwright 模拟用户的实际操作（不是直接访问 URL）
  3. 验证修复不会破坏其他功能
- **不要重复同样的错误**：如果一次验证方法失败了，下次必须换一种方法。
- **保持谦逊**：在没有充分验证之前，不要说"已修复"，而是说"我做了修改，请你验证一下"。

## 问题四（补充）：对工具和框架的理解不足

### 现象

在修复过程中，多次因为对 MkDocs、nginx、Docker 的行为理解不足而导致错误的修复方案。

### 具体案例

1. **MkDocs Material 的 JavaScript 行为**：不知道 MkDocs Material 会拦截链接点击并进行 SPA 导航，导致误以为是 nginx 或链接格式的问题。
2. **MkDocs 的 `use_directory_urls` 设置**：不知道这个设置会影响链接格式，导致生成的 HTML 中链接与预期不同。
3. **Docker 的构建缓存**：修改了源文件后没有重新构建 Docker 镜像，导致容器中还是旧版本。
4. **nginx 的 rewrite 行为**：不知道 `rewrite ... permanent` 会生成包含内部端口的重定向 URL。

### 经验教训

- **深入理解工具**：在使用工具之前，必须理解其行为和限制。不要凭直觉猜测。
- **查阅文档**：遇到问题时，首先查阅官方文档，而不是凭经验猜测。
- **小步验证**：每次只修改一个变量，验证效果后再修改下一个。

## 总结

| 问题 | 根因 | 影响 | 教训 |
|------|------|------|------|
| 未执行端到端测试 | 用 curl/直接访问 URL 代替用户实际操作 | 用户反复测试、反复失望 | 必须模拟用户操作流程 |
| 破坏其他环境 | 未区分源文件/构建产物，未考虑多环境影响 | Cloudflare 从可用变为不可用 | 修改前评估影响范围 |
| 重复犯同样错误 | 没有从错误中学习，过度自信 | 浪费用户时间 | 建立验证清单，保持谦逊 |
| 对工具理解不足 | 凭直觉猜测，不查文档 | 错误的修复方案 | 深入理解工具，查阅文档 |

## 行动项

1. **端到端测试规范**：所有 Web 相关的修复，必须用 Playwright 模拟用户操作流程验证。
2. **环境隔离**：每个部署环境有独立的配置和部署脚本，避免相互影响。
3. **验证清单**：在报告"已修复"之前，必须完成验证清单。
4. **修改影响评估**：修改任何文件之前，评估对所有环境的影响。
5. **工具学习**：使用新工具前，先理解其行为和限制。
